# <a name="create-and-publish-new-topology-in-skype-for-business-server"></a>Skype for Business Serverで新しいトポロジを作成して発行する
 
**概要：** Skype for Business Serverをインストールする前に、新しいトポロジを作成、発行、検証する方法について説明します。
  
トポロジ内の各サーバーにSkype for Business Server システムをインストールする前に、トポロジを作成して公開する必要があります。 トポロジを発行すると、トポロジ情報が Central Management Microsoft Store データベースに読み込まれます。 これがEnterprise Edition プールの場合は、新しいトポロジを初めて発行するときに、Central Management Microsoft Store データベースを作成します。 これがStandard Editionの場合は、トポロジを発行する前に、展開ウィザードから最初のStandard Editionサーバーの準備プロセスを実行する必要があります。 これにより、SQL Server Express Edition インスタンスをインストールし、Central Management Microsoft Storeを作成することで、Standard Editionの準備が整います。 手順 1 ~ 5 は、任意の順序で実行できます。 ただし、図に示すように、手順 6、7、および 8 を順番に実行し、手順 1 ~ 5 の後に実行する必要があります。 新しいトポロジを作成して発行する方法については、手順 6/8 で説明します。
  
![概要図。](../../media/c5c09ba2-c98b-4194-9857-7c3087c5560e.png)
  
## <a name="create-and-publish-new-topology"></a>新しいトポロジの作成と公開

Skype for Business Server トポロジ ビルダーを使用して、トポロジの設計、定義、構成、および発行を行うことができます。 このツールは、前の記事で管理ツールをインストールしたときにインストールされました。 トポロジを作成する際には、さまざまな選択肢があります。 この手順では、会議を使用して基本的なトポロジを作成します。
  
> [!IMPORTANT]
> Skype for Business Serverを操作するには、SQL Serverが必要です。 プライマリ データベースは、中央管理Microsoft Storeと呼ばれます。 Enterprise Editionをデプロイする場合、これらのデータベースは、次の手順を使用してトポロジを発行するときに作成されます。 この場合、トポロジ ビルダーは、SQL Server インストールへの接続情報を要求します。 Standard Editionのデプロイを計画している場合は、新しいトポロジを定義して発行する前に、SQL Server Express Edition をインストールする必要があります。 SQL Server Express Edition をインストールするには、フロント エンドとして機能するサーバーで展開ウィザードを開き、最初のサーバーの準備Standard Editionサーバーを実行する必要があります。 [最初のStandard Edition サーバーの準備] をクリックすると、展開ウィザードによって自動的に SQL Server Express Edition がインストールされ、中央管理Microsoft Storeデータベースが作成されます。 
  
### <a name="create-a-new-topology"></a>新しいトポロジを作成する

1. トポロジ ビルダーにアクセスできる標準ユーザーとしてログインします。
    
2. トポロジ ビルダー Skype for Business Server開きます。
    
3. [ **新しいトポロジ] を** 選択し、[OK] をクリック **します**。
    
4. トポロジ構成ファイルの場所とファイル名を選択します。
    
    > [!NOTE]
    > トポロジ構成は、トポロジ ビルダー XML (.tbxml) ファイルとして保存されます。 トポロジを発行すると、構成情報がファイルから SQL Server データベースにプッシュされます。 将来トポロジ ビルダーを開くときに、既存の構成をSQL Serverから直接トポロジ ビルダーにダウンロードし、SQL Serverに発行するか、トポロジ ビルダー構成ファイルとして保存することができます。 
  
5. [ **プライマリ ドメインの定義] 画面で**、 **プライマリ SIP ドメイン** を入力し、[ **次へ**] をクリックします。 この例では、図に示すように使用 `contoso.local`しています。
    
     ![プライマリ sip ドメインを定義します。](../../media/353e6b38-485f-4042-8585-aefa6c74b554.png)
  
6. サポートされている SIP ドメインを追加し、[ **次へ**] をクリックします。
    
7. 最初のサイト (場所) の **名前** と **説明** を入力し、図に示すように [ **次へ**] をクリックします。
    
     ![最初のサイト (場所) を定義します。](../../media/d8b6c54a-2011-4efb-97fb-a4de0f11303c.png)
  
8. サイトの **市区町村**、 **都道府県**、 **国/地域コード** を入力し、[ **次へ**] をクリックします。
    
9. **[完了] を** クリックして、新しいトポロジを定義するプロセスを完了します。 新しいフロント エンド ウィザードが自動的に起動します。
    
### <a name="define-a-front-end-pool-or-standard-edition-server"></a>フロントエンド プールまたはStandard Edition サーバーを定義する

2. プールの完全修飾ドメイン名 (FQDN) を入力し、**フロント エンド プール** または **Standard Edition サーバー** Enterprise Edition選択し、図に示すように [**次へ**] をクリックします。
    
    > [!TIP]
    > Skype for Business Server Enterprise Editionには、フロントエンド ロールを提供するために連携する複数のサーバーを含めることができます。 複数のサーバーを使用してロールを満たす場合は、プールと呼ばれます。 したがって、フロントエンド ロールを提供するために連携している複数のサーバーは、フロント エンド プールとも呼ばれます。 Skype for Business Server Standard Editionは、フロント エンド ロールを提供する 1 台のサーバーのみを含めることができます。 1 台のサーバーのみが役割を提供している場合でも、フロントエンド プールを参照するのが一般的です。 
  
     ![フロントエンド プールを定義します。](../../media/c1447557-261e-4260-a362-ab8d19070eb9.png)
  
3. プール内のすべてのコンピューターの完全修飾ドメイン名 (FQDN) を入力し、図に示すように [ **次へ** ] をクリックします。
    
     ![プール内のコンピューターを定義します。](../../media/1106b4f3-8745-485b-b495-f885a5dfefda.png)
  
4. このトポロジに含まれる機能を選択し、図に示すように [ **次へ** ] をクリックします。

     ![デプロイの機能を選択します。](../../media/77257588-d0e1-4517-a12a-869ffe009353.png)
  
5. [ **併置されたサーバーの役割の選択]** ページで、フロントエンド サーバー上の仲介サーバーを併置するか、スタンドアロン サーバーとして展開するかを選択できます。
    
    Enterprise Edition フロント エンド プールで仲介サーバーを併置する場合は、チェック ボックスがオンになっていることを確認します。 このサーバーの役割はプール サーバー上に展開されます。 仲介サーバーをスタンドアロン サーバーとして展開する場合は、適切なチェック ボックスをオフにします。 フロント エンド サーバーを完全に展開した後、仲介サーバーを別の展開手順で展開します。 コロケーションの計画の詳細については、「[Skype for Business Serverのトポロジの基本」を参照してください](../../plan-your-deployment/topology-basics/topology-basics.md)。

    1 つ目は、新しいインストールで新しいトポロジを定義する場合です。 次の 2 つの方法のいずれかでインストールにアプローチできます。

   - インストールする必要がある役割を選択し、選択した役割に対応するハードウェアをセットアップします。
    
     シナリオ 2 では、既存のデプロイがあり、インフラストラクチャは新しいロールに対応しているか、既存のロールを新しいフロントエンド サーバーに関連付ける必要があります。
    
   - この場合は、新しいフロントエンド サーバーを展開または関連付けるロールを選択します。 どちらの場合も、役割の定義を続行して必要なハードウェアを設定し、インストールを進めます。
    
7. 次に、トポロジで使用されるSQL Server ストアを定義します。 この例では、Default インスタンスを使用します。 高可用性などのSQL Server機能の詳細については、「[Skype for Business Serverでの高可用性とディザスター リカバリーの計画](../../plan-your-deployment/high-availability-and-disaster-recovery/high-availability-and-disaster-recovery.md)」を参照してください。
    
   - トポロジで既に定義されている既存の SQL Server ストアを使用するには、[**SQL ストア**] でインスタンスを選択します。
    
   - プール情報を格納する新しいSQL Server インスタンスを定義するには、[**新規**] をクリックし、[**新しい** SQL Microsoft Storeの定義] ダイアログ ボックスで **SQL Server FQDN を** 指定します。
    
   - SQL Server インスタンスの名前を指定するには、[**名前付きインスタンス**] を選択し、インスタンスの名前を指定します。
    
   - 既定のインスタンスを使用するには、[**既定のインスタンス**] をクリックします。
    
   - SQL ミラーリングを使用するには、[**ミラーリングを有効にする] SQL** 選択し、既存のインスタンスを選択するか、新しいインスタンスを作成します。

     > [!NOTE]
     > SQL ミラーリングは 2015 Skype for Business Serverで使用できますが、2019 Skype for Business Serverではサポートされなくなりました。 Skype for Business Server 2019 では、AlwaysOn 可用性グループ、AlwaysOn フェールオーバー クラスター インスタンス (FCI)、およびSQLフェールオーバー クラスタリング方法が優先されます。
    
     この例では、**SQL Server FQDN を** 入力し、関連する高可用性設定を構成し、図に示すように **[OK] をクリックします**。
    
     ![SQL Server ストアを作成します。](../../media/12822cf9-8608-43c0-94ce-2ca8b3a0ffd5.png)

9. 使用するファイル共有を定義します。
    
   - トポロジで既に定義されているファイル共有を使用するには、[**以前に定義したファイル共有を使用する**] を選択します。
    
   - 新しいファイル共有を定義するには、[**新しいファイル共有の定義**] を選択し、[**ファイル サーバー FQDN**] ボックスで、ファイル共有が存在する既存のファイル サーバーの FQDN を入力します。そして、[**ファイル共有**] ボックスでファイル共有の名前を入力します。
    
     この例では、[ **新しいファイル ストアの定義**] をクリックし、 **ファイル サーバーの FQDN** と **ファイル共有** を入力して、[ **次へ**] をクリックします。

10. [Web サービス URL の指定] ページで、内部 Web サービス プールのベース URL をオーバーライドする必要があるかどうかを決定する必要があります。 このオーバーライドの理由は、負荷分散に関係しています。 基本的な SIP トラフィックは、単純な DNS 負荷分散を使用して負荷分散できます。 ただし、HTTP/S Web Services ネットワーク トラフィックでは、サポートされているハードウェアまたはソフトウェア負荷分散ソリューションを使用する必要があります。 サポートされているロード バランサーについては、[Skype for Businessのインフラストラクチャに関するページを](../../../SfbPartnerCertification/certification/infra-gateways.md)参照してください。 この例では、SIP トラフィック用の DNS 負荷分散とサポートされているソフトウェア負荷分散ソリューションを使用しました。 この方法でトラフィックを分割しているため、内部 Web サービス プールの FQDN をオーバーライドする必要があります。 または、上位の回線ロード バランサーがあり、SIP トラフィックに DNS 負荷分散を使用する代わりにすべてのトラフィックを送信した場合は、Web サービス URL をオーバーライドする必要はありません。 
    
    このトピックの DNS セクションで `webint.contoso.local`、. これは Web サービスの HTTP/S トラフィックに使用している URL であり、設定したサポートされているソフトウェア ロード バランサーを経由する必要があります。 したがって、この例では、URL をオーバーライドして、図に示すように、すべての HTTP/S トラフィックが代わりに`pool.contoso.local`行く`webint.contoso.local`必要があることをSkype for Business Serverに知らせます。 負荷分散の詳細については、「[Skype for Businessの負荷分散要件」を](../../plan-your-deployment/network-requirements/load-balancing.md)参照してください。
    
    > [!IMPORTANT]
    > ベース URL は、https:// の部分を除いた URL の Web サービス ID です。 たとえば、プールの Web サービスの完全な URL が次の場合、 `https://webint.contoso.local`ベース URL は `webint.contoso.local`. 
  
    - この例に示すように、DNS 負荷分散を構成する場合は、[ **内部 Web サービス プールの FQDN をオーバーライド** する] チェック ボックスをオンにし、内部ベース URL (プール FQDN とは異なる必要があります) **を内部ベース URL** に入力します。 
    
    > [!CAUTION]
    > 内部 Web サービスを自己定義 FQDN でオーバーライドする場合、各 FQDN は他のフロントエンド プール、ディレクター、またはディレクター プールから一意である必要があります。 URL または完全修飾ドメイン名を定義するときは、標準文字 (A-Z、a-z、0-9、ハイフンを含む) **のみを使用** します。 Unicode 文字およびアンダースコアは使用しないでください。 URL または FQDN の標準以外の文字は、外部 DNS および公的機関 (CA) ではサポートされないことがよくあります (つまり、URL または FQDN を証明書のサブジェクト名またはサブジェクトの別名に割り当てる必要がある場合)。
  
    - 必要に応じて、外部ベース URL に **外部ベース URL を入力します**。 外部ベース URL を入力して、内部ドメイン名と区別します。 たとえば、内部ドメインは 、外部ドメイン名は `contoso.local``contoso.com`. パブリック DNS から解決できる必要があるため、 `contoso.com` ドメイン名を使用して URL を定義します。 これは、リバース プロキシの場合も重要です。 外部ベース URL ドメイン名は、リバース プロキシの FQDN のドメイン名と同じになります。 モバイル クライアントでのインスタント メッセージングとプレゼンスには、フロントエンド プールへの HTTP アクセスが必要です。
    
      ![Web サービスをオーバーライドします。](../../media/8f95313c-2df4-4885-adc5-9fc9ea775406.png)
  
11. **[機能の選択**] ページで **[会議**] を選択した場合は、Office Web Apps サーバーを選択するように求められます。 [ **新規]** をクリックしてダイアログ ボックスを起動します。
    
12. [**新しいOffice Web Apps サーバーの定義**] ダイアログ ボックスの [Office Web Apps サーバー FQDN] ボックスに **、Office Web Apps サーバーの FQDN を** 入力します。これを行うと、Office Web Apps サーバー検出 URL は、[**Office Web Apps サーバー検出 URL**] ボックスに自動的に入力する必要があります。
    
    Office Web Apps サーバーがオンプレミスで、Skype for Business Serverと同じネットワーク ゾーンにインストールされている場合 **は、サーバーが外部ネットワーク (境界/インターネット) に展開Office Web Appsオプションを** 選択しないでください。
    
    Office Web Apps サーバーが内部ファイアウォールの外部に展開されている場合は、**サーバーが外部ネットワーク (境界/インターネット) に展開Office Web Apps** オプションを選択します。
    
13. [ **完了] を** クリックして構成を完了します。 **[サーバーの役割をこのフロントエンド プールに関連付ける**] ページで他の役割サーバーを定義した場合は、サーバーの役割を構成できる別の役割構成ウィザード ページが開きます。 この例では、会議のみを選択しました。
    
### <a name="configure-simple-urls"></a>単純な URL を構成する

1. トポロジ ビルダーで、**Skype for Business Server** 上のノードを右クリックし、図に示すように **[プロパティの編集]** をクリックします。
    
     ![Skype for Business Server右クリックし、[プロパティの編集] を選択します。](../../media/692c18dd-8e99-4239-ae7b-5e855d866afa.png)
  
2. **[単純な URL**] ウィンドウで、**電話アクセス URL(** (ダイヤルイン) または **会議 URL:** (会議) を選択して編集し、[**URL の編集]** をクリックします。
    
3. URL を目的の値に更新し、[**OK**] をクリックして編集した URL を保存します。 外部 SIP ドメインを使用して単純な URL を構成し、外部ユーザーが会議に参加できるようにする必要があります 。たとえば、 `contoso.com`外部のユーザーは、内部ドメインではなく、外部の会議に `contoso.local`参加できます。 そのため、SIP ドメインは外部 DNS で解決できる必要があります。
    
4. 必要に応じて、同じ手順を使用して Meet URL を編集します。
    
### <a name="to-define-the-optional-admin-simple-url"></a>オプションの管理用の簡易 URL を定義するには

1. トポロジ ビルダーで、**Skype for Business Server** ノードを右クリックし、[**プロパティの編集]** をクリックします。
    
2. [**管理アクセス URL**] ボックスに、Skype for Business Server コントロール パネルへの管理アクセスに必要な単純な URL を入力し、[OK] をクリック **します**。
    
    > [!TIP]
    > 管理 URL にできるだけ簡易 URL を使用することをお勧めします。 最も簡単なオプションは https://admin. _\<domain\>_ 管理 URL には、内部ドメインまたは外部ドメインを指定できます。たとえば、 `contoso.local` `contoso.com`内部 DNS でいずれかのレコードを解決できる限り、URL を指定できます。 
  
    > [!IMPORTANT]
    > 最初の展開後に簡易 URL を変更する場合、簡易 URL のドメイン ネーム システム (DNS) レコードと証明書に影響する変更について注意する必要があります。 変更が単純な URL のベースに影響する場合は、DNS レコードと証明書も変更する必要があります。 たとえば、ベース URL を sfb から `https://sfb.contoso.com/Meet` 変更する `https://meet.contoso.com` 場合などです。`contoso.com` に `meet.contoso.com`変更する必要があるため、参照する DNS レコードと証明書を変更する `meet.contoso.com`必要があります。 単純な URL を変更した`https://sfb.contoso.com/Meet``https://sfb.contoso.com/Meetings`場合、ベース URL `sfb.contoso.com` は同じままであるため、DNS または証明書の変更は必要ありません。 ただし、単純な URL 名を変更するときは常に、各 Director サーバーとフロントエンド サーバーで **Enable-CsComputer** コマンドレットを実行して変更を登録する必要があります。
  
### <a name="publish-and-verify-the-topology"></a>トポロジを発行して確認する

1. すべての簡易 URL が正しく構成されていることを確認します。
    
2. SQL Server ベースのサーバーがオンラインであり、必要なファイアウォール規則を含め、トポロジ ビルダーがインストールされているコンピューターで使用できることを確認します。
    
3. ファイル共有が使用可能であり、適切なアクセス許可が定義されていることを確認します。
    
4. トポロジで、展開要件を満たす正しいサーバーの役割が定義されていることを確認します。
    
5. Active Directory Domain Services (AD DS) にサーバーが存在することを確認します。 これは、サーバーをドメインに参加させると自動的に発生します。
    
    トポロジの検証が完了して検証エラーがないことが確認できたら、トポロジの公開の準備に移ります。 検証エラーがある場合は、トポロジを公開する前に修正する必要があります。
    
6. **Skype for Business Server** ノードを右クリックし、[**トポロジの発行**] をクリックします。
    
7. **[トポロジの公開]** ページで、**[次へ]** をクリックします。
    
8. [ **中央管理サーバーの選択** ] ページで、図に示すようにフロントエンド プールを選択します。
    
    > [!NOTE]
    > [ **詳細設定** ] をクリックして、データベース ファイルの場所を構成できます。
  
     ![サーバーの中央管理Microsoft Store選択します。](../../media/764478b5-8480-42c5-854f-649bb121cd94.png)
  
9. [ **データベースの選択]** ページで、発行するデータベースを選択します。
    
    > [!NOTE]
    > データベースを作成する適切な権限がない場合は、それらのデータベースの横にあるチェック ボックスをオフにし、適切な権限を持つユーザーが後でデータベースを作成できます。 要件の詳細については、「[Skype for Business Serverのサーバー要件」を](../../plan-your-deployment/requirements-for-your-environment/server-requirements.md)参照してください。 
  
10. オプションで [**詳細設定**] をクリックします。 詳細SQL Serverデータ ファイルの配置オプションを使用すると、次のオプションから選択できます。 
    
    - **データベース ファイルの場所を自動的に決定** する - このオプションは、ログ ファイルとデータ ファイルを最適な場所に配布することで、SQL Server ベースのサーバー上のディスク構成に基づいて最適な運用パフォーマンスを決定します。
    
    - **インスタンスの既定値SQL Server使用** する - このオプションでは、インスタンス設定を使用して、ログ ファイルとデータ ファイルをSQL Server ベースのサーバーに配置します。 このオプションでは、ログとデータの最適な場所を決定するのに、SQL Server ベースのサーバーの操作上の機能は使用されません。 SQL Server の管理者が通常、SQL Server ベースのサーバーおよび組織の管理手順に適した場所にログ ファイルとデータ ファイルを移動します。
    
    [**OK**] をクリックして、[**次へ**] をクリックします。 
    
11. 必要に応じて、[ **詳細設定**] をクリックします。 詳細SQL Serverデータ ファイルの配置オプションを使用すると、次のオプションから選択できます。 
    
    - **データベース ファイルの場所を自動的に決定** する - このオプションは、ログ ファイルとデータ ファイルを最適な場所に配布することで、SQL Server ベースのサーバー上のディスク構成に基づいて最適な運用パフォーマンスを決定します。
    
    - **インスタンスの既定値SQL Server使用** する - このオプションでは、インスタンス設定を使用して、ログ ファイルとデータ ファイルをSQL Server ベースのサーバーに配置します。 このオプションでは、ログとデータの最適な場所を決定するのに、SQL Server ベースのサーバーの操作上の機能は使用されません。 SQL Server の管理者が通常、SQL Server ベースのサーバーおよび組織の管理手順に適した場所にログ ファイルとデータ ファイルを移動します。
    
    [**OK**] をクリックします。
    
12. [**次へ**] をクリックして公開プロセスを完了します。
    
    > [!NOTE]
    > この手順で発生する一般的なエラーは、SQL Server データベースを作成できないことです。 プロセスが完了できない場合は、図に示すようにエラーが発生します。 最も可能性の高い原因は、データベースを作成しようとしているユーザーが適切なアクセス許可を持っていないか、ファイアウォールやその他のネットワークの問題のためにSQL Server システムに接続できないことです。 
  
     ![中央管理ストアの作成時にエラーが発生しました。](../../media/558bd2e4-2721-422d-9986-df86f642e6a1.png)
  
13. 発行プロセスが完了すると、次の手順の一覧を開くためのリンクが表示されます。 **ここをクリックして To Do リストを開き**、次の手順を表示し、[完了] をクリック **します**。 
    
    データベースの作成に対する "警告で完了しました" メッセージは、エラーが発生したことを意味しません。 インストール プロセスでは、Skype for Business Serverが正しく動作するようにSQL Serverの設定を変更する必要があります。 SQL Serverで設定が変更されると、SQL Server管理者がインストール プロセスが完了したことを正確に把握できるように、警告としてログに記録されます。 警告が表示された場合は、レコードを選択し、[ **ログの表示** ] をクリックして警告の詳細を表示できます。
    
    トポロジが正常に発行されたら、トポロジでSkype for Business Serverを実行している各サーバーに Central Management ストアのローカル レプリカのインストールを開始できます。 最初のフロントエンド プールから開始することをお勧めします。
