# <a name="manage-skype-for-business-server-2015-using-scom-management-pack"></a>SCOM Management pack を使用した Skype for Business Server 2015 の管理
 
**の概要:** Skype は、System Center Operations Manager を使用するビジネス サーバー 2015 のインフラストラクチャを構成する方法について説明します。
  
理想の世界でビジネス サーバー 2015 の Skype で問題が発生する、ことはありません。 ただし、Skype ビジネス サーバーの影響を受ける可能性の外部要因などのクラッシュやハードウェア障害など、ネットワークの。 Skype ビジネス サーバー 2015 の管理パックを使用すると、識別し、潜在的な問題をプロアクティブに対処できます。 この方法では、ビジネス サーバー 2015 の管理パックの Skype は、System Center Operations Manager の機能を拡張します。
  
この情報に基づいて作成されて監視パックのバージョン 9319.0 Skype のビジネス サーバー 2015 通信ソフトウェアです。
  
## <a name="configuration-overview"></a>構成の概要

 Skype は、System Center Operations Manager を使用するビジネス サーバー 2015 のインフラストラクチャを構成するには、次の 3 つを行う必要があります。
  
識別し、[プライマリ管理サーバーを構成](configure-the-primary.md)します。 これを行うには、System Center オペレーション マネージャー 2012 SP1 または R2 をインストールする必要があります。 
  
 識別および[ビジネスのサーバーのコンピューターを監視するための Skype を構成します](configure-computers-to-monitor.md)。 System Center Operations Manager を使用してビジネスのサーバー コンピューターに、Skype を監視するには、System Center Operations Manager エージェントのファイルをインストールし、プロキシとして動作する各サーバーを構成する必要があります。 
  
 識別し、[をインストール ウォッチャー ノードを構成して](watcher-nodes.md)。 ウォッチャー ノードは、定期的に Skype をビジネス サーバー代理トランザクションを実行するコンピューター、システムにログオンする機能、瞬時に交換する機能など、ビジネスのサーバー コンポーネントのキー Skype を検証する Windows PowerShell コマンドレットメッセージは期待どおりに作業しています。

## <a name="whats-new"></a>新機能

Skype ビジネス サーバー 2015 の管理パックには次の機能します。
  
- **クライアントの自動検出**クライアント アプリケーションにサインインするには Skype ビジネス サーバー 2015 の多くの場合に自動的にサインインするにはサーバーを検出します。 代理トランザクションでは、検証をサポートする自動検出が正しく構成されています。
    
- **カスタマイズされた代理トランザクションの実行間隔**ウォッチャー ノードのプロセスの設定を簡素化するには、代理トランザクションは、ユーザー アカウントを共有できます。 テストは、競合を避けるためにシリアル化するテストの実行頻度が低下します。 既定では、すべてのテストを確認するのには 15 分ごとに実行する代理トランザクションを実行する時間のあります。 多くのユーザーまたはユーザーごとの少数のテストを使用する管理者も実行間隔を短くようになりました可能性があります。
    
- **ビデオの相互運用機能サービスの代理トランザクション**移行する Skype をビジネス サーバー 2015 の他のベンダーのソリューションから多くのお客様は、これらの他のベンダーからのビデオ会議デバイス (VTCs) を使用し続ける希望です。 ビデオの相互運用機能のサーバーは、Cisco CUCM ビデオの SIP トランク経由で接続することによって、会議室で Cisco の VTCs を使用し続けることにより、ビジネス サーバー 2015 サーバーの役割を新規の Skype です。 この機能は、ビデオの相互運用機能のサーバーが起動して、ビデオの SIP トランク経由で着信接続を処理できることを確認するために代理トランザクションを追加します。
    
- **アプリケーション共有会議代理トランザクション** アプリケーション共有会議のエンドツーエンド シナリオ検証がサポートされるようになりました。

### <a name="alert-prioritization"></a>アラートの優先度設定

アラートは、次のカテゴリに分類されます。 
  
 **優先度の高い警告:** これらのアラートは、多数のユーザーのサービスの停止が発生して、即座のアクションを必要とする条件を指定します。 代理トランザクションとビジネス サーバー オーディオ/ビデオ会議用 Skype) などのオフラインのサービスによって検出された障害は、優先度の高い警告と見なされます。 対照的に、1 台のコンピューターのコンポーネントの障害は、優先度の高いアラートではありません。 ビジネス サーバー 2015 の Skype がこのような場合は組み込みの高可用性機能を持つ-たとえば、複数フロント エンド サーバー ロード バランサーの背後にあります。
  
 **[中] の優先度の警告:** これらのアラートは、一部のユーザーに影響を与えるか、通話の音質に問題を示す状態を示します-などのコンポーネントの障害、または呼び出しでオーディオ品質の呼び出しの確立に遅延します。 このカテゴリの通知には、ステートフル (つまり、ネットワーク接続の状態に基づく警告の変更の性質)。などの呼び出しの確立の時間遅延を示す通常のしきい値に戻る場合は、この優先順位が中程度のアラートは、System Center Operations Manager での自動解決だろうし、管理者が操作を実行する必要はありません。 自動解決できないアラートは通常、その日のうちに管理者が対応します。
  
 **その他のアラート:** これらのアラートは、特定のユーザーまたは一部のユーザーに影響を与える可能性があるコンポーネントから生成されます。たとえば、よくあるアラートとしては、アドレス帳サービスがユーザー testuser@contoso.com の Active Directory® ドメイン サービス (AD DS) エントリを解析できない場合などです。管理者は、時間に余裕があるときにこれらのアラートに対応できます。

|簡易 URL ウォッチャー  <br/> |展開で構成されている簡易 URL の HTTPS 調査を実行します。  <br/> |

ビジネス サーバー プールのため、Skype では、ビジネス サーバー ・ システムの複数の個別 Skype (ビジネス サーバーの役割、ビジネス サーバー サービスでは、Skype と Skype ビジネス サーバー コンポーネントの 1 つ以上の Skype) とを含めることができます。 したがって、個々 のサーバーまたはコンポーネントのエラーは、同じプール内の他のサーバーは、クライアントにアプリケーション サービスを提供することができますのでビジネス サーバー プールの場合は、Skype の全体的な稼働状態になります。 状態は、ビジネス サーバー プールの Skype を率のレベルにロールアップされます。 
  
ビジネス サーバー プールの監視の Skype は、ビジネス サーバー プールのため、Skype に対して代理トランザクションを実行します。 1 つ以上の代理トランザクション (連続ポーリング間隔として知られるプロセス) が連続して失敗すると、次の図に示すように、重要という正常性の状態をプール レベル (あらゆる代理トランザクションのうち、最悪のもの) までまとめられます。

## <a name="best-practice-create-a-management-pack-for-customizations"></a>ベスト プラクティス: カスタマイズ用の管理パックを作成する

既定では、Operations Manager は既定の管理パックへの上書きなど、すべてのカスタマイズを保存します。ベスト プラクティスとして、カスタマイズする封印された管理パックごとに別個の管理パックを作成することをお勧めします。 
  
封印された管理パックに対してカスタマイズした設定を格納するための管理パックを作成するとき、新しい管理パックに「Skype for Business Server 2015 カスタマイズ」など、適切な名前を付けることをお勧めします。 
  
封印された各管理パックのカスタマイズを格納するための新しい管理パックを作成すると、テスト環境からカスタマイズを運用環境にエクスポートすることが容易になります。これにより、管理パックを削除することも容易になります。管理パックを削除するには、依存関係を削除する必要があるためです。すべての管理パックのカスタマイズが既定の管理パックに保存され、1 つの管理パックを削除する必要がある場合は、まず、既定の管理パックを削除する必要があります。この操作により、他の管理パックへのカスタマイズも削除されます。
