# <a name="troubleshoot-statistics-manager-for-skype-for-business-server"></a>統計マネージャーは、Skype のビジネス サーバーのトラブルシューティングを行う
 
**の概要:** ビジネス サーバーの Skype の統計マネージャーの展開のトラブルシューティングを行うには、このトピックを参照してください。
  
このトピックでは、アプリケーション イベント ログに表示されるイベントとイベントを修正するための適切なアクションを記述して、統計マネージャーの展開をトラブルシューティングする方法について説明します。 このトピックには、次のセクションがあります。
  
- [エージェント イベント](troubleshoot.md#BKMK_Agent)
    
- [リスナー イベント](troubleshoot.md#BKMK_Listener)
    
- [Web サイトの問題](troubleshoot.md#BKMK_Website)
    
## <a name="agent-events"></a>エージェント イベント
<a name="BKMK_Agent"> </a>

- **1000** - プロセッサー制限をセットアップできない (ジョブ オブジェクト) - 原因不明
    
- **1001** -プロセスでは、プロセスを制限することが許可されていない (おそらくジョブ オブジェクト内)
    
    エージェントは Windows のジョブ オブジェクト内部で実行され、自動でエージェントの使用するメモリ容量を制限します。 エージェントが開始しておらず、このジョブ オブジェクトのイベント エントリがイベント ログに存在する場合、このジョブ オブジェクトをサーバー上でインスタンス化できません。 これを回避するには、設定ファイルで値を変更してメモリの上限を外してください。
    
  ```
  C:\Program Files\Skype for Business Server StatsMan Agent\PerfAgent.exe.config
  ```

    "MaxProcessMemoryMB"を検索しのように値を「0」に変更します。
    
  ```
  <setting name="MaxProcessMemoryMB" serializeAs="String"> <value>300</value> </setting>
  ```

    > [!NOTE]
    > エージェントが消費されますが一般的にこの変更を加えた場合\<100 MB のメモリ、しかしないことが 300 MB に制限が強制的に既定値としてします。 この変更を行った場合、メモリの使用状況を注意深く監視しそのエージェントがホスト マシンで大容量のメモリを消費しないようにすることをお勧めします。 
  
- **2000** - クライアントの初期化に失敗しました
    
- **2001** - このサービスと接続されているソース IP はありません
    
    エージェントがリスナーのコンピューターと接続できない場合、以下を確認してください。
    
1. リスナー サービスがリスナーのコンピューターで実行されていることを確認します。 実行されていない場合は、サーバーで Redis が実行していることを確認してからリスナー サービスを再起動します。
    
    統計マネージャーのリスナー サービス自体に問題がないことを確認するのにはリスナーのコンピューターで統計マネージャーのイベント ログを確認してください。
    
2. telnet などの接続ツールを使用して、エージェントのコンピューターから適切なポートのリスナーへの接続性を確認します。
    
    接続されていない場合は、リスナーのコンピューターでのファイアウォールの受信ルールで、リスナーのコンピューターが接続されているネットワークのタイプ (プライベート/パブリック/ドメイン) が有効になっていることを確認します。 リスナーのコンピューターがドメインに参加していない場合、ネットワークは、パブリックとして表示される場合があり、場合は統計マネージャーがインストールされているファイアウォールの規則は既定で適用されません。
    
- **4000** - リスナーからのサーバー情報のダウンロードに失敗しました (理由不明)
    
  - **4001** - リスナーのトポロジでサーバーが見つかりませんでした
    
    サーバーがリスナーに正常に接続していますが、サーバーは、リスナーのキャッシュのトポロジに追加されませんでしたこのエラーが発生します。 解決策の選択肢には、以下のものがあります。
    
  - 	トポロジのインポートの手順に従っていることを確認します。「[Import the topology (英語)](deploy.md#BKMK_ImportTopology)」を参照してください。   
    
  - このトポロジ内のリストに記述されていないサーバーにエージェントがある場合 (たとえば、SQL AlwaysOn クラスタ内のノード)、「[トポロジのインポート](deploy.md#BKMK_ImportTopology)」にある手順に従ってエージェントを手動で追加する必要があります。
    
  - **4002** - 無効なリスナー パスワード
    
    エージェントが使用しようとした暗号化されたパスワードが、リスナー自体のサービス パスワードと一致しません。 エージェントをアンインストールし、正しいサービス パスワードを使用して再度インストールしてください。
    
  - **4003** - 証明書の拇印の不一致
    
    インストール時に、エージェントに指定された証明書の拇印がないリスナーの証明書の拇印を使用して現在の一致との接続は拒否されます。 エージェントをアンインストールして再インストールして、正しい証明書の拇印を使用します。
    
  - **4004** - 無効な応答または無効な HttpStatusCode
    
    リスナーの応答が、期待した状態ではありません。   
    
  - 接続がプロキシ化されている場合、プロキシ設定を確認します。
    
  - リスナー コンピューターの StatsMan のログの構成に関する問題を確認してください。
    
  - **4005** - XML のシリアル化解除ができませんでした
    
    リスナー サーバー上の情報が破損しているか、エージェント コンピューターとリスナー コンピューターの間でバージョンの不一致があります。 バージョンが一致しているか確認し、リスナーのイベント ログで問題を確認します。
    
## <a name="listener-events"></a>リスナー イベント
<a name="BKMK_Listener"> </a>

- **10000** - 不明な原因で起動に失敗しました (回復不能な障害があるため、サービスは停止またはクラッシュします)
    
  - **10001** - 構成の問題
    
    この問題は通常、[listener_install_location]\PerfAgentListener.exe 構成ファイルを手動で変更したために、アプリケーションから読み取れなくなったことが原因で発生します。
    
  - **10002** - HTTP リスナー初期化エラー
    
    このイベントは通常、インストール時に URL ACL が適切に設定されていないか、または SSL 証明書が無効な場合に発生します。 使用中の設定にある証明書が有効であることを確認してください。 有効な場合は、「[Deploy Statistics Manager (英語)](deploy.md#BKMK_Deploy)」の指示に従ってリスナーを再度インストールします。
    
  - **10003** - Redis エラー
    
  - **10004** - キャッシュ インフラストラクチャーのエラー
    
  - **10007** - 設定 (redis に保存)
    
    リスナーが Redis に連絡できないか、適切に書式化されたデータをキャッシュから取得できないか、クラウドを開始できません。 Redis サービスが開始されていて、サーバーが適切に構成されていることを確認します。
    
  - **10005** - サーバー情報の取得中または解析中です
    
    Redis キャッシュ内のトポロジ情報が無効です。 まず、Redis とリスナーを再起動してみてください。 解決しない場合は、「[Import the topology (英語)](deploy.md#BKMK_ImportTopology)」を参照して、トポロジ データを再度作成します。
    
- **10100** - Redis PING が停止しました
    
  - **10101** - Redis PING の停止を継続します (60 秒毎)
    
  - **30100** - Redis PING の停止から復元しました
    
    これらのイベントは、リスナーが Redis に接続できない場合にログに記録されます。 Redis が開始していること、およびリスナーと Redis 間の接続が使用可能なことを確認します。
    
- **10200** - Redis の書き込みが停止しました
    
  - **10201** - Redis の書き込みの停止を継続します (60 秒毎)
    
  - **30100** - Redis の書き込みが解決しました
    
    これらのイベントは、リスナーが Redis キャッシュへの書き込みに失敗したときにログに記録されます。 Redis が開始していること、およびリスナーと Redis 間の接続が使用可能なことを確認します。
    
- **30000** - 正常に開始しました
    
    リスナーが開始するたびにログに記録されます。
    
- **22000** -初期化の統計マネージャー エージェントが成功しました。
    
- **23000** - EventLogQueryManager の初期化に成功しました (最初、その後は失敗した場合)
    
- **24000** - サーバー情報の初期化に成功しました (最初、その後は失敗した場合)
    
- **25000** - リスナーが投稿に失敗した後、オンラインに復帰しました (または、最初に成功した投稿)
    
- **5000** - データを投稿するためオフラインのリスナーを開始します
    
- **5001** - リスナーは拡張された期間中さらにオフラインになります
    
    これらのイベントは、監視/警告/問題の消去で有用です。
    
## <a name="website-issues"></a>Web サイトの問題
<a name="BKMK_Website"> </a>

- クロムに繰り返しログインを求められます - このバージョン 1.1 では、解決されているバグです。 クロムのブラウザーで何度もログイン プロンプトが表示される場合に、統計マネージャーの最新バージョンにアップグレードすることを確認します。 実行中の Web サイトのバージョンを確認するには、次の手順を実行してください。
    
  - 	エクスプローラーで、デフォルトディレクトリを開きます
    
  - StatsManHubWebSite.dll を右クリックし、このファイルのプロパティを表示します。
    
  - [KHI 横方向] ビューまたは [カウンター詳細] ビューでコンピューターが見つからない場合、コンピューターがサイトまたはプールのメンバーであることを確認します。 メンバーでない場合は、これらのビューに表示されません。 トポロジでのサーバーに対するサイトおよびプールの定義の詳細については、「[Import the topology (英語)](deploy.md#BKMK_ImportTopology)」を参照してください。
    
  - 製品のバージョンは [説明] の詳細に表示されます。
