# <a name="direct-sip-connections-in-skype-for-business-server"></a>Skype for Business Server での直接 SIP 接続

直接 SIP 接続は、Skype for Business Server と、SKYPE for Business Server の PSTN ゲートウェイと IP-PBX の両方エンタープライズ VoIP。

直接 SIP 接続を使用して、Skype for Business Server を次のどちらかに接続できます。

- IP-PBX

- PSTN ゲートウェイ

直接 SIP 接続を実装する場合、基本的に SIP トランクの実装と同じ展開ステップを実行します。 どちらの場合も、仲介サーバーの外部インターフェイスを使用して接続を実装します。 唯一異なるのは、SIP トランクを外部エンティティ (ITSP ゲートウェイなど) に接続し、直接 SIP 接続をローカル ネットワーク内の内部エンティティ (IP-PBX や公衆交換電話網 (PSTN) ゲートウェイなど) に接続するという点です。

## <a name="direct-sip-deployment-options"></a>直接 SIP 展開オプション

### <a name="skype-for-business-server-stand-alone"></a>Skype for Business Server Stand-Alone
<a name="BKMK_CommunicationsServerStand-Alone"> </a>

組織でこのセクションで説明されている展開のいずれかを使用している場合は、組織の一部またはすべての組織の唯一のテレフォニー ソリューションとして Skype for Business Server を使用できます。 このセクションでは、以下の展開について詳しく説明します。

- **増分展開:** このオプションでは、既存の PBX (Private Branch Exchange) インフラストラクチャを使用し、組織内の小規模なグループまたはチームに エンタープライズ VoIP を段階的に導入する予定である必要があります。

- **VoIP 専用** の展開: このオプションは、従来のテレフォニー インフラストラクチャを持エンタープライズ VoIPサイトへの展開を検討している必要があります。

#### <a name="incremental-deployment"></a>増分展開

増分展開では、Skype for Business Server は個々のチームまたは部門の唯一のテレフォニー ソリューションですが、組織内の残りのユーザーは引き続き PBX を使用します。 この増分展開戦略は、制御されたパイロット プログラムを使用して IP テレフォニーを企業に導入する 1 つの方法を提供します。 Microsoft Unified Communications が最適な通信を必要とするワークグループは エンタープライズ VoIP に移動され、他のユーザーは既存の PBX に残ります。 必要に応じて、追加のワークグループエンタープライズ VoIPに移行できます。

通信要件が共通で、一元的な管理に利用できるユーザー グループを明確に定義している場合は、増分オプションをお勧めします。 このオプションは、長距離料金を大幅に節約できる、地理的に広い地域に広がるチームまたは部署がある場合にも有効です。 実際、このオプションは、メンバーが世界中に分散している可能性がある仮想チームを作成する場合に便利です。 ビジネス要件の変化に迅速に対応して、このようなチームを作成、変更、またはバンド解除できます。

#### <a name="enterprise-voice-behind-the-pbx"></a>エンタープライズ VoIP PBX の背後にある

#### <a name="voip-only-deployment"></a>VoIP-Only展開

エンタープライズ VoIP は、新しいビジネス、および既存のビジネス向け新しいオフィス サイトを提供し、PBX 統合を心配したり、IP-PBX インフラストラクチャの大幅な展開と保守コストを発生させずに、フル機能の VoIP ソリューションを実装する機会を提供します。 このソリューションは、オンサイト ワーカーとリモート ワーカーの両方をサポートします。

この展開では、すべての通話が IP ネットワークを使用してルーティングされます。 PSTN への通話は、適切な PSTN ゲートウェイにルーティングされます。 Skype for Business または Lync Phone Edition はソフトフォンとして機能します。 ユーザーが制御する PBX 電話がないので、リモート通話コントロールは使用できなくなったり不要になったりします。 ボイス メール サービスと自動応答サービスは、Exchange ユニファイド メッセージング (UM) のオプションの展開によって利用できます。

## <a name="pstn-gateway-deployment-options"></a>PSTN ゲートウェイ展開オプション

### <a name="pstn-gateways"></a>PSTN ゲートウェイ

公衆交換電話網 (PSTN) ゲートウェイは、直接または SIP トランクへの接続を介して、エンタープライズ VoIP インフラストラクチャと PSTN の間で信号とメディアを変換するサードパーティのハードウェア コンポーネントです。 どちらのトポロジでも、ゲートウェイは PSTN を終了します。 ゲートウェイは独自のサブネットに分離され、仲介サーバーを介してエンタープライズ ネットワークに接続されます。

複数のサイトを持つ企業は、通常、各サイトに 1 つ以上のゲートウェイを展開します。 ブランチ サイトは、ゲートウェイを介して PSTN に接続するか、ゲートウェイとサーバーを 1 つのボックスに結合する存続可能ブランチ アプライアンスを介して接続できます。 ブランチ サイトでゲートウェイを使用する場合は、WAN リンクが回復力がない限り、レジストラーと仲介サーバーの両方がサイトで必要です。 1 つ以上の仲介サーバー (フロントエンド サーバーに同時に展開) は、各サイトで 1 つ以上のゲートウェイの通話をルーティングできます。 サイトで必要なレジストラー、仲介サーバー、およびゲートウェイを存続可能ブランチ アプライアンスとして展開することをお勧めします。

PSTN ゲートウェイの数、サイズ、場所の決定は、多くの場合、PSTN ゲートウェイインフラストラクチャの計画時に行う必要のある最も重要でコストの高い決定エンタープライズ VoIPがあります。

考慮すべき主な質問を次に示します。 これらの質問に対する回答はすべて相互に依存する点に気を付ける

- 必要な PSTN ゲートウェイの数 回答は、ユーザー数、予想される同時通話数 (トラフィック負荷)、サイト数 (各サイトに 1 つが必要) によって異なります。

- ゲートウェイのサイズ その答えは、サイトのユーザー数とトラフィック負荷によって異なります。

- ゲートウェイの場所 その答えは、トポロジの一部と、組織の地理的な分布の一部によって異なります。

  ゲートウェイ トポロジのオプションも考慮する必要があります (詳細については、このトピックの後半の「ゲートウェイ トポロジ」を参照してください)。

#### <a name="mn-trunk-support"></a>M:N トランクのサポート

仲介サーバーは、複数のゲートウェイ、インターネット テレフォニー サービス プロバイダーが提供するセッション ボーダー コントローラー (SPC)、またはこれらの 2 つの組み合わせを介して通話をルーティングできます。 さらに、プール内の複数の仲介サーバーが複数のゲートウェイと対話できます。 仲介サーバーとゲートウェイの間で定義される論理ルートをトランクと呼ぶ。 内部ユーザーが PSTN 通話を発信すると、フロントエンド プールの発信ルーティング ロジックによって、その特定の通話のルーティングに使用できる可能性があるすべての組み合わせからルーティングするトランクが選択されます。 DNS 負荷分散では、プール内の特定の仲介サーバーの問題が原因で通話がゲートウェイに到達できない場合、その通話はプール内の代替仲介サーバーに対して再試行されます。

複数のゲートウェイの計画の詳細については [、Skype for Business Server の M:N トランクを参照してください](m-n-trunk.md)。

その他の発信ルーティングの拡張機能の詳細については、「通話ルート [」を参照してください](https://technet.microsoft.com/library/a2ddf327-2ec4-407b-af0f-276f2b13eefd.aspx)。

#### <a name="gateway-topologies"></a>ゲートウェイ トポロジ

ゲートウェイの展開に関する基本的な質問を検討する場合は、次の手順を実行します。

1. PSTN 接続を提供するサイトを数えるには、次のエンタープライズ VoIP。

2. 各サイトのトラフィックを見積もります (ユーザー数と、ユーザーごとの 1 時間あたりの平均通話数)。

3. 予想されるトラフィックを処理するために、各サイトに 1 つ以上のゲートウェイを展開します。

このトポロジでは、各サイトおよびサイト間のワーカー間の通話はすべてイントラネットを使用してルーティングされます。 PSTN への通話は、エンタープライズ IP ネットワークを通して宛先番号の場所に最も近いゲートウェイにルーティングされます。しかし、多くの金融機関や他の大企業が行うのと同様に、組織が 1 つ以上の大陸に分散した数十、数百、数千のサイトをサポートしている場合は、どうなるでしょうか。 このような場合、各サイトに個別のゲートウェイを展開する方法は実用的ではありません。

この問題に対処するために、多くの大企業は、1 つ以上の大規模なテレフォニー 中央サイトを展開することを好む。

このトポロジでは、予想されるユーザー負荷に対応するのに十分な大きなゲートウェイが各中央サイトに展開されます。 企業内のユーザーへのすべての通話は、会社の電話サービス プロバイダーによって中央サイトに転送されます。 中央サイトのルーティング ロジックによって、通話をイントラネットまたは PSTN にルーティングする必要があります。

#### <a name="gateway-location"></a>ゲートウェイの場所

ゲートウェイの場所によって、選択するゲートウェイの種類と構成方法が決めらる場合があります。 数十の PSTN プロトコルが存在しますが、いずれも世界標準でもありません。 すべてのゲートウェイが単一の国/地域にある場合、これは問題ではありません。ただし、複数の国/地域にゲートウェイを見つける場合は、それぞれのゲートウェイをその国/地域の PSTN 標準に従って構成する必要があります。 さらに、カナダなど、運用が認定されているゲートウェイは、インド、ブラジル、または欧州連合で認定されていない場合があります。

#### <a name="gateway-size-and-number"></a>ゲートウェイのサイズと番号

ほとんどの組織が展開を検討する PSTN ゲートウェイのサイズは 2 ~ 960 ポートです。 (ゲートウェイはさらに大きくなりますが、主に電話サービス プロバイダーによって使用されます)。組織で必要なポート数を見積もる場合は、次のガイドラインに従います。

- テレフォニー使用量が少ない組織 (1 人あたり 1 時間に 1 つの PSTN 通話) は、15 ユーザーごとに 1 つのポートを割り当てる必要があります。 たとえば、ユーザーが 20 人の場合は、2 つのポートを持つゲートウェイが必要です。

- テレフォニー使用率が中程度の組織 (1 時間あたり 1 ユーザーあたり 2 つの PSTN 通話) は、10 ユーザーごとに 1 つのポートを割り当てる必要があります。 たとえば、ユーザーが 100 人の場合は、1 つ以上のゲートウェイに合計 10 個のポートを割り当てる必要があります。

- テレフォニーの使用率が高い組織 (1 時間あたり 1 ユーザーあたり 3 つ以上の PSTN 通話) は、5 人のユーザーごとに 1 つのポートを割り当てる必要があります。 たとえば、ユーザーが 47,000 人の場合は、少なくとも 10 個の大規模なゲートウェイに割り当てられたポートが合計 9,400 個必要です。

- ユーザー数や組織内のトラフィック量の増加に応じて、追加のポートを取得できます。

サポートする必要がある任意の数のユーザーに対して、展開するゲートウェイの数を少なくするか、大きくするか、小さいゲートウェイを展開するかのいずれかを選択できます。 原則として、1 つのゲートウェイで障害が発生した場合でも可用性を維持するには、1 つの組織に対して少なくとも 2 つのゲートウェイをお勧めします。

展開する各 PSTN ゲートウェイには、少なくとも 1 つの対応する仲介サーバーが必要です。
