# <a name="load-balancing-requirements-for-skype-for-business"></a>Skype for Business の負荷分散の要件
 
**の概要:** ロード ・ バランシング ・ Skype をビジネスのサーバーを実装する前に注意事項を確認します。
  
負荷分散プール内のサーバー間でトラフィックを分散します。 フロント エンド プール、仲介サーバー プールまたはエッジ サーバーのプールがあれば、これらのプールの負荷分散を展開する必要があります。
  
Skype ビジネス サーバーの負荷分散のクライアントからサーバーへのトラフィックのためのソリューションの 2 つの種類をサポートしています: ドメイン ネーム システム (DNS) の負荷を分散し、ハードウェア負荷分散 (HLB として省略) ことがよくあります。 DNS は、分散サービス管理の合理化より効率的なトラブルシューティング、およびハードウェア ロード バランサー機器の潜在的な問題からビジネスのサーバーのトラフィックは、Skype の大部分を分離する機能を含むいくつかの利点をロードします。
  
負荷分散ソリューションでは、配置内の各プールの適切な自分の決定が、次の制限事項に留意してください。 
  
- 内部エッジ インターフェイスと外部エッジ インターフェイスでは、同じ種類の負荷分散を使用する必要があります。1 つのインターフェイスで DNS 負荷分散を使用し、もう 1 つのインターフェイスでロード バランサー機器を使用することはできません。
    
- 一部の種類のトラフィックでは、ロード バランサー機器が必要です。たとえば、HTTP トラフィックでは、DNS 負荷分散ではなくロード バランサー機器が必要です。クライアントとサーバー間の Web トラフィックでは、DNS 負荷分散は正常に機能しません。
    
プールで DNS 負荷分散を使用するように選択し、HTTP トラフィックなどのトラフィック用にロード バランサー機器も実装する必要がある場合、ロード バランサー機器の管理は非常に簡素化されます。たとえば、ロード バランサー機器の構成は、他のすべてのプロトコルが DNS 負荷分散によって管理されるのに対して、HTTP および HTTPS のトラフィックを管理するのみであるため簡素になります。詳細については、「[DNS Load Balancing](load-balancing.md#BKMK_DNSLoadBalancing)」を参照してください。 
  
サーバーからサーバーへのトラフィックは、Skype ビジネス サーバーは、トポロジに対応した負荷分散を使用します。 サーバーは、トポロジ内のサーバーの Fqdn を取得するのには中央管理ストアで公開されているトポロジを読み取るし、サーバー間でトラフィックを自動的に配布します。 この種の負荷分散では、管理者が設定または管理を行う必要はありません。 
  
DNS 負荷分散を使用していて、特定のコンピューターへのトラフィックを遮断する必要がある場合は、プールの FQDN から IP アドレス エントリを削除するだけでは十分ではありません。コンピューターの DNS エントリも削除する必要があります。

## <a name="dns-load-balancing"></a>DNS Load Balancing
<a name="BKMK_DNSLoadBalancing"> </a>

ビジネス サーバーにより DNS の負荷分散、管理の負荷分散ネットワーク上のオーバーヘッドを大幅に軽減できるソフトウェア ・ ソリューションの Skype です。 DNS 負荷分散は、SIP トラフィックとメディア トラフィックなどのビジネス サーバーでは、Skype に固有のネットワーク トラフィックを分散します。
  
DNS 負荷分散を展開する場合は、ハードウェア ロード バランサーのオーバーヘッド、組織の管理が最小化します。 さらに、SIP トラフィックの負荷分散装置の構成の誤りに関連する問題の複雑なトラブルシューティングは、削除されます。 サーバーをオフラインにすることができるようにも、サーバーの接続を防止できます。 DNS 負荷分散は、ハードウェア ロード バランサーの問題が基本的な通話のルーティングなどの SIP トラフィックの要素に影響しないことも確実にします。

以下の図の例を両方が含まれる内部と外部の DNS の負荷分散をします。 
  
**パブリック IPv4 アドレスを使用したエッジ ネットワーク ダイアグラム**

![DNS ネットワーク ダイアグラムの例](../../media/2cc9546e-5560-4d95-8fe4-65a792a0e9c3.png)
  
DNS 負荷分散を使用する場合、すべての種類のトラフィックに対してハードウェア ロード バランサーを使用した場合よりも低コストのハードウェアのロード バランサーを購入することがあります。 Skype でのテストは、ビジネスのサーバーの相互運用性認定に合格したロード バランサーを使用する必要があります。 ロード バランサー機器の相互運用性テストについての詳細は、 [Lync Server 2010 のロード バランサー パートナー](https://go.microsoft.com/fwlink/p/?linkId=202452)を参照してください。 ビジネス サーバーの Skype にある内容が適用されます。
  
フロント エンド プール、エッジ サーバー プール、ディレクター プール、およびスタンドアロンの仲介サーバー プールの DNS 負荷分散はサポートされています。
  
DNS 負荷分散は、通常、アプリケーション レベルで実装されます。 返される IP アドレスのいずれかに接続することによって、プール内のサーバーに接続しようとするアプリケーション (たとえば、クライアント ビジネスの Skype を実行している)、DNS の A および AAAA (IPv6 のアドレス指定が使用されます) の場合は、プールの完全修飾ドメイン名 (FQDN) のクエリを記録します。 
  
たとえば、pool01.contoso.com をという名前のプールで 3 つのフロント エンド サーバーがある場合は、次が行われます。
  
- ビジネス用の Skype を実行しているクライアントは、pool01.contoso.com の DNS を照会します。 クエリは、次の 3 つの IP アドレスし、必ずしもこの順序で) 次のようにキャッシュに保持を返します。
    
    pool01.contoso.com 192.168.10.90
    
    pool01.contoso.com 192.168.10.91
    
    pool01.contoso.com 192.168.10.92
    
- クライアントが IP アドレスのいずれかの伝送制御プロトコル (TCP) 接続を確立しようとするとします。 失敗した場合、クライアントはキャッシュ内の次の IP アドレスを試みます。
    
- TCP 接続が成功した場合、クライアントは TLS のネゴシエーションを行い、pool01.contoso.com のプライマリ レジストラーに接続します。
    
- クライアントには、接続が成功せず、すべてのキャッシュされたエントリが試行されると、そのビジネス サーバーの Skype を実行しているサーバーがある時点でユーザーに通知されます。
    
> [!NOTE]
> DNS ベースの負荷分散とは異なる DNS ラウンド ロビン (DNS RR) の IP とは異なる順序を提供するように DNS に依存することによって負荷分散には、通常は、プール内のサーバーに対応します。 通常 DNS RR を有効には負荷分散の場合、だけですがないフェイル オーバーを有効にします。 1 つの IP アドレスへの接続によって返される DNS の A および AAAA (IPv6 のアドレス指定を使用している) 場合のクエリが失敗した場合、接続に失敗した場合などです。 したがって、DNS ラウンド ロビンを単独では、DNS ベースの負荷分散よりも信頼性が低い。 DNS を使用して DNS と共にラウンド ロビン方式で負荷分散します。 
  
DNS 負荷分散は、次のような使用されます。
  
- 負荷分散、エッジ トランスポート サーバーにサーバーからサーバーへの SIP
    
- 負荷分散の会議自動応答、応答グループ、コール パークなどのアプリケーションの統合コミュニケーション アプリケーション サービス (UCAS)
    
- (とも呼ばれる「ドレイン」) の UCAS アプリケーションへの新しい接続を防止します。
    
- クライアントとエッジ サーバー間のすべてのクライアントからサーバーへのトラフィックを分散するロード
    
DNS 負荷分散は、次の使用できません。
  
- ディレクターまたはフロント エンド サーバーにクライアントからサーバーへの web トラフィック
    
DNS は分散とフェデレーション トラフィックをロードします。
  
DNS SRV クエリを使用して、複数の DNS レコードが返される場合、優先度の低い数値と数値の最大の重量とアクセス エッジ サービスは、DNS SRV を常に選択を記録します。 「サービス (DNS SRV) の場所を指定する DNS RR」を文書化、インターネット技術標準化委員会[RFC 2782、DNS SRV RR](https://www.ietf.org/rfc/rfc2782.txt)を指定する場合は複数の DNS SRV レコードが定義されている、優先順位最初に使用される重量、します。 たとえば DNS SRV レコード A 20 のウエイトがあり、40 と B の DNS SRV レコードの優先順位があり、10 の重み 50 の優先順位。 40 優先順位を持つ DNS SRV レコードの A が選択されます。 DNS SRV レコードの選択範囲には、次の規則が適用されます。
  
- 優先順位は、最初と見なされます。 クライアントは、DNS SRV レコードが最も小さい番号の優先順位でそれが到達可能で定義されているターゲット ・ ホストにお問い合わせくださいしようとする必要があります。 同じ優先順位を持つターゲットは、重量] フィールドで定義されている順に試行する必要があります。

#### <a name="deploying-dns-load-balancing-on-edge-server-pools"></a>展開する DNS での負荷分散エッジ サーバー プール

DNS での負荷分散、エッジ サーバー プールの外部インターフェイスを配置するには、次の DNS エントリが必要です。
  
- アクセス エッジ サービスの場合、プール内の各サーバーの 1 つのエントリを作成する必要があります。 各エントリは、プール内のエッジ サーバーの 1 つ上のアクセス エッジ サービスの IP アドレスに、アクセス エッジ サービス (たとえば、sip.contoso.com) の FQDN を解決する必要があります。
    
- Web 会議エッジ サービスの場合、プール内の各サーバーの 1 つのエントリを作成する必要があります。 各エントリは、プール内のエッジ サーバーの 1 つ上の Web 会議エッジ サービスの IP アドレスを Web 会議エッジ サービス (たとえば、webconf.contoso.com) の FQDN を解決する必要があります。
    
- オーディオ/ビデオ エッジ サービスの場合、プール内の各サーバーのエントリが 1 つを作成する必要があります。 各エントリは、A の IP アドレスを (たとえば、av.contoso.com) のオーディオ/ビデオ エッジ サービスの FQDN を解決する必要があります音声ビデオ エッジ サービスのプール内のエッジ サーバーの 1 つにします。
    
DNS での負荷分散、エッジ サーバー プールの内部インターフェイスを配置するには、エッジ サーバー プールの内部 FQDN をプール内の各サーバーの IP アドレスに解決する 1 つの DNS A レコードを追加してください。

### <a name="blocking-traffic-to-a-server-with-dns-load-balancing"></a>DNS の負荷分散とサーバーへのトラフィックをブロックします。

DNS 負荷分散を使用していて、特定のコンピューターへのトラフィックを遮断する必要がある場合は、プールの FQDN から IP アドレス エントリを削除するだけでは十分ではありません。コンピューターの DNS エントリも削除する必要があります。
