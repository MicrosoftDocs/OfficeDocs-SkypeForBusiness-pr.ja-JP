**概要:** Skype for Business Server のエッジサーバーの環境要件について説明します。
  
Skype for Business Server エッジサーバー環境の外側で、多くの計画と準備を行う必要があります。 この記事では、以下のリストに従って、組織環境で実行する必要のある準備内容を確認します。
  
- [トポロジの計画](edge-environmental-requirements.md#TopoPlan)
    
- [DNS の計画](edge-environmental-requirements.md#DNSPlan)
    
- [証明書の計画](edge-environmental-requirements.md#CertPlan)
    
- [ポートとファイアウォールの計画](edge-environmental-requirements.md#PortFirewallPlan)
    
## <a name="topology-planning"></a>トポロジの計画
<a name="TopoPlan"> </a>

Skype for Business Server エッジサーバーのトポロジでは、次のものを使用できます。
  
- ルーティング可能なパブリック IP アドレス。
    
- **対称**ネットワークアドレス変換 (NAT) を使用する場合は、ルーティング可能でないプライベート IP アドレス。
    
> [!TIP]
> エッジサーバーは、各サービスに個別のポートを持つ単一の IP アドレスを使用するように構成することも、サービスごとに個別の IP アドレスを使用することもできますが、既定では同じポート (既定では TCP 443 になります) を使用します。 詳細については、以下の「IP アドレス要件」セクションを参照してください。 
  
NAT でルーティング可能でないプライベート IP アドレスを選択する場合は、次の点に注意してください。
  
- **3 つすべて**の外部インターフェイスで、ルーティング可能なプライベート IP アドレスを使用する必要があります。
    
- 受信トラフィックと送信トラフィック用に**対称**NAT を構成する必要があります。 対称 NAT は、Skype for Business Server エッジサーバーで使用できる唯一のサポートされる NAT です。
    
- 受信元アドレスを変更しないように NAT を構成します。 音声ビデオエッジサービスは、受信元アドレスを受信して最適なメディアパスを見つけることができる必要があります。
    
- エッジサーバーは、公開音声ビデオエッジ IP アドレスから互いに通信できる必要があります。 ファイアウォールでこのトラフィックを許可する必要があります。
    
- DNS 負荷分散を使用する場合は、拡張統合エッジサーバーに対して**のみ**NAT を使用できます。 ハードウェア負荷分散 (HLB) を使用する場合は、NAT を使用**せず**に、一般にルーティング可能な IP アドレスを使用する必要があります。
    
ハードウェア負荷分散を使用していない場合は、単一および拡張統合エッジサーバートポロジの両方で対称 NAT を実行しているときに、アクセス、Web 会議、音声ビデオエッジのインターフェイスの背後に問題はありません。
  
### <a name="summary-of-edge-server-topology-options"></a>エッジサーバートポロジオプションの概要

- プライベート IP アドレスと NAT を使用する単一統合エッジ
    
- パブリック IP アドレスを使用する単一統合エッジ
    
- プライベート IP アドレスと NAT を使用する拡張統合エッジ
    
- パブリック IP アドレスを使用する拡張統合エッジ
    
- 拡張統合エッジ (ロードバランサー機器を使用)

### <a name="ip-address-requirements"></a>IP アドレスの要件

基本的なレベルでは、3つのサービスには IP アドレスが必要です。アクセスエッジサービス、Web 会議エッジサービス、音声ビデオエッジサービス。 3つの IP アドレスを使用するか、それぞれのサービスに対して1つを使用するか、または1つを使用して各サービスを異なるポートに配置することができます (そのうちのいくつかについては、「[ポートとファイアウォールの計画](edge-environmental-requirements.md#PortFirewallPlan)」セクションを参照してください)。 単一の統合エッジ環境の場合、これは非常に多くなります。
  
> [!NOTE]
> 前述したように、3つのサービスすべてに対して1つの IP アドレスを選択し、それらを異なるポートで実行できます。 明確にするために、この方法はお勧めしません。 このシナリオで使用する代替ポートにお客様がアクセスできない場合は、エッジ環境のすべての機能にアクセスできません。 
  
拡張統合トポロジを使用すると、もう少し複雑になることがあります。そのため、IP アドレス要件をレイアウトするいくつかの表を見てみましょう。これは、トポロジ選択の主要な決定ポイントが高可用性と負荷分散であることに留意してください。 高可用性が必要な場合は、負荷分散の選択に影響を与える可能性があります (これについては、表の後に説明します)。

    - Microsoft 365 または Office 365
- Skype for Business Server 2019:
    - Lync Server 2013
    - Skype for Business Server 2015

## <a name="certificate-planning"></a>証明書の計画
<a name="CertPlan"> </a>

Skype for Business Server は、サーバーとクライアントの間の安全で暗号化された通信のために証明書を使用します。 ご想像のとおり、証明書の DNS レコードを、証明書のサブジェクト名 (SN) およびサブジェクト代替名 (SAN) に一致させる必要があります。 これは計画段階ですぐに機能し、証明書のための SN と SAN のエントリについて、DNS に正しい Fqdn が登録されていることを確認します。
  
外部および内部証明書のニーズについて個別に説明し、両方の要件を提供する表を参照してください。
  
### <a name="external-certificates"></a>外部証明書

少なくとも、外部エッジサーバーインターフェイスに割り当てられている証明書は、パブリック証明機関 (CA) によって提供される必要があります。 特定の CA をお勧めすることはできませんが、推奨 CA が表示されているかどうかを確認するには、Ca の[統合コミュニケーション証明書パートナー](/SkypeForBusiness/certification/services-ssl)の一覧を取得します。
  
この公開証明書の CA に要求を送信する必要があるのはいつですか。 これを行うには、いくつかの方法があります。
  
- Skype for Business Server のインストールとエッジサーバーの展開を進めることができます。 Skype for Business Server 展開ウィザードには、証明書要求を生成するための手順が用意されており、選択した CA に送信できます。
    
- また、この要求を生成するために Windows PowerShell コマンドを使用することもできます。
    
- 最後に、CA に独自の送信プロセスがある場合があります。これには、Windows PowerShell または別の方法も含まれます。 その場合は、参考として、ここに記載されている情報に加えて、ドキュメントに頼る必要があります。
    
証明書を入手した後は、次のようにして Skype for Business Server のサービスに割り当てる必要があります。
  
- アクセスエッジサービスインターフェイス
    
- Web 会議エッジサービスインターフェイス
    
- 音声ビデオ認証サービス (音声ビデオおよびビデオストリームを暗号化するために証明書を使用しないため、これは音声ビデオエッジサービスと混同しないでください)
    
> [!IMPORTANT]
> すべてのエッジサーバー (同じエッジサーバーのプールに属している場合) は、メディアリレー認証サービスに対して同じ秘密キーを持つ証明書を正確に持つ必要があります。 
  
### <a name="internal-certificates"></a>内部証明書

内部エッジサーバーインターフェイスでは、パブリック CA からのパブリック証明書、または組織の内部 CA から発行された証明書を使用できます。 内部証明書について覚えておくべきことは、SN エントリを使用し、SAN エントリを使用しないことです。したがって、内部証明書で SAN について心配する必要はありません。
