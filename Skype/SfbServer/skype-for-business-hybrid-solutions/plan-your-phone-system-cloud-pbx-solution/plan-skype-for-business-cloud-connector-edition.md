# <a name="plan-for-skype-for-business-cloud-connector-edition"></a>Skype for Business Cloud Connector エディションの計画

Skype for Business Cloud Connector エディションについて紹介します。Skype for Business Cloud Connector エディションは、Office 365 の電話システム (クラウド PBX) でのオンプレミスの PSTN 接続を実装する、パッケージ化された仮想マシン (VM) のセットです。

クラウド コネクタのエディションは、既にがありません、既存の Lync Server または Skype ビジネス サーバー導入の場合、組織に最適なソリューションにすることがあります。 ソリューションで Office 365 の電話システムはビジネスに適したを調査しているが場合、は、 [Office 365 (クラウド PBX) ソリューションの電話システムの計画](plan-your-phone-system-cloud-pbx-solution.md)を参照してください。

このドキュメントでは、クラウド コネクタのエディションの要件とサポートされているトポロジでは、説明、クラウド コネクタ Edition の展開を計画するのに役立ちます。 されますが、コネクタのクラウド環境を構成する前に、このトピックを参照してください。 展開とクラウド コネクタのエディションの設定を参照してくださいする準備ができたら[構成およびビジネス クラウド コネクタ ・ エディションの Skype を管理する](configure-skype-for-business-cloud-connector-edition.md)。

クラウド コネクタ版 2.1 は、提供されています。 まだ、2.1 に、アップグレードができない場合は、[クラウドのコネクタの新しいバージョンへのアップグレード](upgrade-to-a-new-version-of-cloud-connector.md)を参照してください。 インストール ファイルを見つけることができます[https://aka.ms/CloudConnectorInstaller](https://aka.ms/CloudConnectorInstaller)。

> [!NOTE]
> マイクロソフトでは、新しいバージョンのリリースから 60 日間の以前のバージョンのクラウド コネクタのエディションをサポートします。 Microsoft は、ユーザーによる更新期間を確保するため、2.1 のリリース後 60 日間においてバージョン 2.0.1 をサポートします。 2.0.1 以前のすべてのバージョンは今後サポートされなくなります。

クラウド コネクタのエディションは、Office 365 の電話システムに設置した PSTN 接続を実装するパッケージ化された仮想マシン (Vm) のセットで構成されるハイブリッドを提供することです。 ビジネス サーバー トポロジの場合、仮想化された環境で最小限の Skype を配布すると、クラウドに置かれている組織内のユーザーはマイクロソフトのクラウドから PBX サービスを受信することができますが、PSTN への接続は、既存のオンプレミス音声を通じて提供されます。インフラストラクチャです。

![クラウド PBX を Skype for Business のオンプレミス展開に接続するクラウド PBX ゲートウェイを示す、トポロジ図](../../media/bd898e69-6458-4276-aebe-1854f28ed6fa.png)

Cloud Connector により Office 365 の電話システム サービスを、たとえば PBX、アナログ機器、およびコールセンターで構成される既存のテレフォニー環境と統合できるようになるため、既存のテレフォニー ソリューションから Office 365 の電話システムへの段階的な移行を実施できます。

たとえば、会社に Office 365 の電話システムが用意されていない特定の機能を備えた高度なコール センターがあるとします。 既存のソリューションでは、コール センターのユーザーのままにして、他のユーザーを Office 365 の電話システムに移動することができます。

Cloud Connector はオンプレミスおよびオンラインに存在するユーザーの間のルーティングを提供しますので、独自の PSTN プロバイダーと Office 365 の電話システムを使用する選択をすることができます。

- ユーザーはオンラインに所属します。

## <a name="cloud-connector-edition-components"></a>Cloud Connector エディションのコンポーネント
<a name="BKMK_Components"> </a>

クラウド コネクタ エディションでは、最小限の Skype ビジネス サーバー トポロジが含まれているパッケージ化された仮想マシンのセットを展開するなど、エッジ コンポーネント、コンポーネントの仲介、および中央管理ストア (CMS) の役割で構成されます。 クラウド コネクタの内部の機能に必要なドメイン コント ローラーもインストールします。 ハイブリッド ビジネス オンライン サービスの Skype を含む、Office 365 テナントには、これらのサービスが構成されています。

![Cloud Connector エディションのコンポーネント](../../media/f2d4b8a7-c2f4-4cfc-8137-f187399c1298.png)

クラウド コネクタ コンポーネントは、次の機能を提供します。

- **エッジ コンポーネント**の設置型トポロジと、オンライン サービス間の通信はエッジの構成部品は、次のコンポーネントが含まれています。

  - **アクセス エッジ**の設置型展開と Skype との間のルーティングは、オンライン ビジネスの SIP を提供します。

  - **メディア リレー**では、仲介コンポーネントおよびその他のメディア エンドポイント間でのメディアのルーティングを提供します。

  - **メディア リレー認証/MRAS** ・ メディア リレーへのアクセス トークンが生成されます。

- **発信ルーティング**では、ゲートウェイ間の音声トラフィックの分散を読み込むまたは SBCs、クラウド コネクタ ・ アプライアンスに接続を提供します。 通話はすべてのゲートウェイ間で、または Cloud Connector アプライアンスに接続された SBC 間で均等に分割されます。

    ポリシーに基づくゲートウェイへのルーティングを提供します。 宛先 (発信) PSTN 番号に基づくグローバル ポリシーのみがサポートされています。

- **中央管理ストア (CMS) の役割**- には、CMS ファイルの転送を含め、トポロジのコンポーネント用の構成ストアが含まれています。

- **中央管理ストア (CMS) のレプリカ**の CMS の役割を持つサーバー上のグローバル CMS データベースから構成情報を同期します。

- **ドメイン コント ローラー**のクラウド コネクタ Active Directory ドメイン サービスにすべてのグローバル設定とクラウドのコネクタ コンポーネントを配置するために必要なグループを格納します。 クラウド コネクタ アプライアンスそれぞれに対して 1 つのフォレストが作成されます。 ドメイン コント ローラーは運用環境の Active Directory に接続します。 Active Directory サービスには以下が含まれます。

  - Active Directory ドメイン サービス

  - 内部証明書を発行するための Active Directory 証明書サービス

- **仲介コンポーネント**- Skype ビジネスと PSTN ゲートウェイの間で SIP の実装とメディア ゲートウェイのマップ型プロトコルです。 グローバル CMS データベースからの構成を同期化する CMS のレプリカが含まれています。

## <a name="cloud-connector-edition-topologies"></a>Cloud Connector エディションのトポロジ
<a name="BKMK_Topologies"> </a>

ここでは、説明のために PSTN サイトを引き合いに出します。 PSTN のサイトは、クラウドのコネクタのアプライアンスを使用して、共通の PSTN ゲートウェイに接続されていると同じ位置では、展開の組み合わせです。 PSTN サイトでは以下を実現できます。

- ユーザーに最も近いゲートウェイへの接続を提供する。

- PSTN に使用される 1 つまたは複数のサイト内の複数のコネクタのクラウド アプライアンスを導入することによってスケーラビリティを使用できます。

- 1 つの PSTN サイト内の複数のコネクタのクラウド アプライアンスを導入することによって高可用性を実現できるようにします。

このトピックでは PSTN サイトについて説明します。 PSTN のサイトの計画の詳細については、[クラウド コネクタのエディションの PSTN のサイト計画](plan-for-cloud-connector-edition-pstn-sites.md)を参照してください。

次のクラウド コネクタ トポロジを展開できます。

- PSTN のサイトあたり 1 つのクラウド コネクタ版アプライアンスです。 このトポロジは高可用性を提供しないので、評価用に限定することをお勧めします。

- 高可用性を実現するために PSTN のサイトあたり複数のコネクタのエディションをクラウド アプライアンスです。

- 高可用性とスケーラビリティを提供する複数のコネクタのエディションをクラウド アプライアンスと複数の PSTN のサイトです。 高可用性のあるスケーラビリティが提供され、200 サイトまで展開できます。

トポロジを計画する場合は、次の点を考慮します。

- クラウド コネクタ 2.0 以降のバージョンと 1 つの PSTN サイトは最大 16 個のコネクタのクラウド アプライアンスを持つことができます。 以前のバージョンではサイトあたり 4 個までのアプライアンスがサポートされました。

- クラウドのコネクタを使用してテストのハードウェア構成の 2 種類があります。

  - 大規模なバージョンでは大量の通話を同時に処理でき、あらゆる種類の本稼働環境でサポートされています。

  - 小規模なバージョンは、性能がそれほど高くないハードウェアで実行することが意図されており、通話量の少ないサイトでの評価用として使用できます。小規模バージョンの Cloud Connector を展開する場合でも、本稼働クラスのハードウェア要件 (電源の二重化など) を念頭に置く必要があります。

- クラウド コネクタ 2.0 またはそれ以降のバージョンがあり、最大数の構成 (大規模なハードウェア) で 16 台のアプライアンスを配置する場合は、PSTN のサイトは、最大 8,000 個の同時呼び出しを処理できます。 小規模バージョンでは 800 件までに制限されます。

    一部のアプライアンスを高可用性のための専用とする必要もあります。最低限、1 つのアプライアンスを高可用性のために予約しておくことを推奨します。

  - バージョン 2 では、15 + 1 の構成を展開する場合、PSTN サイトで処理できる同時呼び出しの最大 7,500。

  - 以前のバージョンが存在し、最大である 3 + 1 構成を (より大掛かりなハードウェアを使用して) 展開する場合は、PSTN サイトで最大 1500 件の通話を同時に処理できます。小規模バージョンでは 150 件までに制限されます。

-  PSTN サイトでこれよりも多くの通話を処理する必要がある場合は、同じ場所に追加 PSTN サイトを展開することによってスケールアップできます。

### <a name="multiple-cloud-connector-appliances-within-a-single-pstn-site"></a>単一の PSTN サイト内の複数の Cloud Connector アプライアンス

 スケーラビリティと高可用性を高めるのため、次の図に示すように、1 つの PSTN サイト内で複数のクラウド コネクタのエディションがあることを選択できます。 次の点に注意してください。

- 通話は、1 つのプール内の Cloud Connector 間でランダムな順序で割り当てられます。

- 処理能力の計画のためには、次の計算に基づいて、1 つまたは複数の Cloud Connector がオフラインになった場合の負荷への対応能力を検討する必要があります。

  - **N+1 ボックス。** N + 1 ボックスが 500 をサポートする、クラウドのコネクタのより大きなバージョンの\*99.8% の可用性と N の同時呼び出しします。

    N + 1 ボックスが 50 をサポートする、クラウドのコネクタの小さいバージョンの\*99.8% の可用性と N の同時呼び出しします。

  - **N+2 ボックス。** N + 2 ボックスが 500 をサポートする、クラウドのコネクタのより大きなバージョンの\*N 99.9% の可用性を同時に呼び出す。

    N + 2 ボックスが 50 をサポートする、クラウドのコネクタの小さいバージョンの\*N 99.9% の可用性を同時に呼び出す。

![1 つの PSTN サイト内に 2 つの Cloud Connector](../../media/fc0dc47f-5595-42cb-9432-9c8ff3e134e9.png)

### <a name="multiple-pstn-sites-with-one-or-more-cloud-connectors-per-site"></a>サイトあたり 1 つまたは複数の Cloud Connector が展開された複数の PSTN サイト

複数の PSTN サイトにそれぞれ 1 つまたは複数の Cloud Connector エディションを展開できます。PSTN サイトが同時通話の制限に達した場合は、別の PSTN サイトを追加してその負荷に対応することができます。

複数の PSTN サイトを使用すると、ユーザーに最も近いゲートウェイへの接続を提供することもできます。 たとえば、シアトルとアムステルダムに PSTN ゲートウェイがあるとします。 シアトルとアムステルダムに 1 つずつ、2 つの PSTN サイトを展開し、最も近い PSTN サイトを使用するようにユーザーを割り当てます。 シアトルにいるユーザーはシアトルの PSTN サイトとゲートウェイに、アムステルダムにいるユーザーはアムステルダムの PSTN サイトとゲートウェイに、それぞれルーティングされます。

## <a name="requirements-for-deployment"></a>展開の要件

メディアのポートの範囲を定義する場合は、次のことに注意してください。

- クライアントがメディア トラフィック用のポートの範囲 50000 に 50019 を常に使用: この範囲はビジネス オンラインの Skype で事前に定義されて、変更できません。

- 仲介コンポーネント、既定では、使用ポートの範囲 49 152 を 57 500 メディア トラフィック用です。 ただし、内部のファイアウォールを経由して接続が確立され、セキュリティ上の理由から、トポロジでは、このポートの範囲を制限することができます。 コールあたりの最大 4 つのポートを必要があります。 仲介コンポーネントと PSTN ゲートウェイ間でポートの数を制限する場合は、ゲートウェイに対応するポート範囲を構成する必要がもあります。

- 境界ネットワーク上のクラウドのコネクタを展開する必要があります。 つまり、2 つのファイアウォールを持つことになります。

  - 最初のファイアウォールは、インターネットと境界ネットワークの間にある外部ファイアウォールです。

  - 2 番目のファイアウォールは、境界ネットワークと内部ネットワーク間の内部です。

    クライアントは、インターネットと内部ネットワークのどちらにあってもかまいません。

Get-CsService -MediationServer | Select-Object Identity, AudioPortStart, AudioPortCount

### <a name="dns-name-resolution-for-the-edge-component"></a>エッジ コンポーネントの DNS 名前解決
<a name="BKMB_Ports"> </a>

エッジ コンポーネントは、Office 365 サービスの外部名およびその他のクラウドのコネクタ コンポーネントの内部名を解決する必要があります。

各エッジ コンポーネントは、外部インターフェイスと内部インターフェイスを備えたマルチホーム コンピューターです。 クラウド コネクタでは、境界ネットワーク内のドメイン コント ローラーのコンポーネント上の DNS サーバーを展開します。 すべての名前解決のための境界内にある DNS サーバーをエッジ サーバーをポイントすることができますが、名前を参照する外部のクエリの 1 つまたは複数の DNS A レコードを含む DNS ゾーンを設定することで外部名を解決するのにはクラウド コネクタの DNS サーバーを有効にする必要があります。他のパブリック DNS サーバーを検索します。

.ini ファイルにおいて、SIP ドメインと同じドメイン空間のゲートウェイに FQDN 名を設定すると、この SIP ドメインの認証ゾーンが境界内の DNS サーバーに作成されます。 エッジ サーバーは、名前を解決するには、この DNS サーバーを指すが場合、エッジは、_sipfederationtls を解決することはありません。\<別\>の呼び出しのフローに必要な DNS レコードです。 この例では、マイクロソフトは、インターネット名の参照を解決するのには、エッジの外部インターフェイスの DNS サーバーを指定して、エッジの各コンポーネントは他のクラウドのコネクタ コンポーネントの名前を IP アドレスに解決するのには、ホスト ・ ファイルを使用する必要があることを推奨します。

> [!NOTE]
> セキュリティ上の理由から、名前解決のための運用環境のドメイン内の内部サーバーをクラウド コネクタの DNS サーバーを指していないことをお勧めします。

### <a name="certificate-requirements"></a>証明書の要件
<a name="BKMK_Certs"> </a>

各エッジ コンポーネントは、パブリック証明機関からの証明書を必要とします。 この証明書には、エッジ コンポーネント間でコピーできるようにエクスポート可能な秘密キーが必要です。 証明書の要件を満たすには、次のオプションのいずれかを選択し、証明書のサブジェクト名 (SN) およびサブジェクトの別名 (SAN) を指定する必要があります。

 **単一の SIP ドメインがある場合は次のとおりです。**

- **オプション 1** サブジェクト名には、エッジ コンポーネントに割り当てたプール名を含める必要があります。 ビジネス エッジ コンポーネントのオンラインの Skype のこの名前が予約されているために、サブジェクト名に sip.sipdomain.com をすることはできませんに注意してください。 SAN には、sip.sipdomain.com とアクセス エッジ プールの名前を含める必要があります。

  ```
  SN = accessedgepoolnameforsite1.sipdomain.com, SAN = sip.sipdomain.com,
  acessedgepoolnameforsite1.sipdomain.com
  ```

- **オプション 2 です。** 展開すると、すべてのエッジ プール サーバーの 1 つのワイルドカード証明書を使用したいかどうかは、ワイルドカード SAN エントリを使用することがあります\*です。 証明書のエッジ プール名の代わりに sipdomain.com。 サブジェクト名には、アクセスのいずれかの展開しているエッジ プールのエッジのプール名を指定できます。

  ```
  SN = accessedgepoolnameforsite1.sipdomain.com, SAN = sip.sipdomain.com, SAN = *.sipdomain.com
  ```

> [!NOTE]
> Sip の外部 DNS エントリを作成する必要がありません。\<sipdomain\>.com この名前は、Office 365 の展開に属しているためです。

> [!NOTE]
> 組織で展開されているすべてのエッジ プールに単一の証明書を使用する場合に、オプション 2 で定義したワイルドカード証明書を使用できないときは、証明書の SAN 名にすべての展開済みエッジ プールの FQDN を含める必要があります。

 **複数の SIP ドメインがある場合は次のとおりです。**

すべての SIP ドメインについて sip.sipdomain.com を追加したうえで、ドメインごとにアクセス エッジ プールの名前を追加する必要があります (1 つの物理プールに対して異なる名前を使用できます)。 複数の SIP ドメイン シナリオにおける SN と SAN のエントリの例を次に示します。

- **オプション 1** サブジェクト名には、エッジのコンポーネントについては、割り当てプールの名前を含める必要があります。 ビジネス エッジ コンポーネントのオンラインの Skype のこの名前が予約されているために、サブジェクト名に sip.sipdomain.com をすることはできませんに注意してください。 SAN には、sip.sipdomain.com とアクセス エッジ プールの名前を含める必要があります。

  ```
  SN = accessedgepoolnameforsite1.sipdomain1.com, SAN = sip.sipdomain1.com, sip.sipdomain2.com,
  acessedgepoolnameforsite1.sipdomain1.com
  ```

- **オプション 2 です**。展開すると、すべてのエッジ プール サーバーの 1 つのワイルドカード証明書を使用したいかどうかは、ワイルドカード SAN エントリを使用することがあります\*です。 証明書のエッジ プール名の代わりに sipdomain.com。 サブジェクト名には、アクセスのいずれかの展開しているエッジ プールのエッジのプール名を指定できます。

  ```
  SN = accessedgepoolnameforsite1.sipdomain.com, SAN = sip.sipdomain1.com, sip.sipdomain2.com,
  SAN = *.sipdomain1.com
  ```

> [!NOTE]
> Sip の外部 DNS エントリを作成する必要がありません。\<sipdomain\>.com この名前は、Office 365 の展開に属しているためです。

展開用には、次の表を使用できます。

|**オプション**|**説明**|**メモ**|
|:-----|:-----|:-----|
|展開環境に使用するオプション  <br/> |オプション 1 または 2  <br/> ||
|SN  <br/> |証明書の SN を指定  <br/> ||
|SAN  <br/> |証明書の SAN を指定  <br/> ||

ゲートウェイと仲介サーバーの間に TLS を使用している場合は、ゲートウェイに割り当てる証明書として、ルート証明書または完全な証明書チェーンを取得する必要があります。

## <a name="high-availability-considerations"></a>高可用性に関する考慮事項
<a name="BKMK_HA"> </a>

高可用性を実現するクラウド コネクタのエディションを配置するときは、互いのバックアップとして機能する 2 つ以上のアプライアンスを配置します。 各アプライアンスは、次の 4 つのコンポーネントで構成されます: エッジ、課金仲介、中央管理ストア (CMS)、およびドメイン コント ローラーです。

一般に、アプライアンス内の 1 つのコンポーネントで障害が発生した場合クラウド コネクタのエディションは、呼び出しを処理するために続行できますが、次を考慮する必要があります。

- **仲介、CMS、ドメイン コントローラーの各コンポーネントに関する考慮事項**

    あるアプライアンスの CMS コンポーネントまたはドメイン コントローラー コンポーネントがダウンしたとします。 このアプライアンスは引き続き通話の発着信を処理できますが、ドメイン コントローラー コンポーネントまたは CMS コンポーネントにアクセスできない状況で仲介コンポーネントを再起動しても、仲介コンポーネントは機能しません。 同じことは、ドメイン コントローラーがタウンしている状況で CMS コンポーネントを再起動した場合にも当てはまります。

    **推奨値:** コンポーネントを再起動する前に、アプライアンス内の他のコンポーネントの可用性を確認します。

- **エッジ コンポーネントに関する考慮事項**

    あるアプライアンスに属するエッジ インスタンスが使用できない場合、発着信通話に対する動作は以下のように変わります。

  - **送信を呼び出す**-インターネット上で、ユーザーから PSTN ネットワークへの呼び出しです。

    クラウドの通話配信メカニズムは、エッジ コンポーネントがダウンしていることを認識し、すべての通話を他のインスタンスにルーティングするので、通話は発信できます。

  - **受信を呼び出して**PSTN ネットワークからローカル ネットワークまたはインターネットのいずれかであるユーザーへの呼び出しです。

     通話を受け取ったアプライアンスのエッジ コンポーネントが機能していない場合、仲介コンポーネントは通話を他のアプライアンスのエッジ コンポーネントにリダイレクトできないため、その通話の着信は正常に完了しません。

    **推奨値:** 場所の監視システムがあります。 エッジ コンポーネントの誤動作を識別した後シャット ダウン、アプライアンス内のすべてのコンポーネント エッジ コンポーネントは使用できません。

## <a name="monitoring-and-troubleshooting"></a>監視とトラブルシューティング
<a name="BKMK_Monitor"> </a>

監視とトラブルシューティングのメカニズムは Cloud Connector アプライアンスそれぞれに自動的にインストールされます。このメカニズムにより次のイベントが検出されます。

- Cloud Connector アプライアンスの 1 つまたは複数の仮想マシンが内部またはインターネットの仮想スイッチに接続されていない。

- Cloud Connector アプライアンスの 1 つまたは複数の仮想マシンが保存済みまたは停止された状態にある。

- 実行されていないサービス。

  次のイベントのいずれかが検出された場合は、全体のクラウド コネクタ アプライアンスが放電され、誤動作しているアプライアンスへの呼び出しを確立する試みを防止するのにはオフラインとしてマークされています。 Cloud Connector の自動復元機能はその後にサービスを復元してアプライアンスをオンラインとしてマーク付けします。 自動復旧は、何らかの理由により失敗した場合、[コネクタのクラウド展開のトラブルシューティング](troubleshoot-your-cloud-connector-deployment.md)を参照してください。

  - 中央管理ストアの仮想マシン: 

     - Skype for Business マスター レプリケーター エージェント

     - Skype for Business レプリカ レプリケーター エージェント

  - 仲介サーバーの仮想マシン:

     - Skype for Business レプリカ レプリケーター エージェント

     - Skype for Business Server 仲介

  - Edge サーバーの仮想マシン:

     - Skype for Business レプリカ レプリケーター エージェント

     -  Skype for Business Server Access エッジ

     - Skype for Business Server 音声ビデオ エッジ

     - Skype for Business Server 音声ビデオ認証

     - Skype for Business Server Web 電話会議エッジ

- エッジ上の「CS RTCSRV」、仲介サーバー上の「CS RTCMEDSRV」に対する Windows ファイアウォールの受信の規則は無効になります。

Cloud Connector 2.1 以降では、Operations Management Suite (OMS) を使用して Cloud Connector を監視することがサポートされます。 詳細については、[運用管理スイート (OMS) を使用してモニターのクラウド コネクタ](monitor-cloud-connector-using-operations-management-suite-oms.md)を参照してください。
