<span data-ttu-id="f8cfd-111">Lync Server 2013 で監視を有効にする場合は、CDR 監視と QoE 監視の両方を有効にすることも、一方の種類の監視を有効にして他方の種類を無効のままにすることもできます。</span><span class="sxs-lookup"><span data-stu-id="f8cfd-111">If you choose to enable monitoring in Lync Server 2013 you can enable both CDR monitoring and QoE monitoring, or you can choose to enable one type of monitoring while leaving the other type disabled.</span></span> <span data-ttu-id="f8cfd-112">たとえば、ユーザーがインスタント メッセージングとファイル転送だけを使用し、音声またはビデオ通話を行わないとします。</span><span class="sxs-lookup"><span data-stu-id="f8cfd-112">For example, suppose your users only use instant messaging and file transfers, and do not make audio or video calls.</span></span> <span data-ttu-id="f8cfd-113">この場合、QoE 監視を有効にする意味はほとんどありません。</span><span class="sxs-lookup"><span data-stu-id="f8cfd-113">In that case, there might be little reason to enable QoE monitoring.</span></span> <span data-ttu-id="f8cfd-114">同様に、Lync Server を使用すると、監視が展開された後の監視を簡単に有効または無効にすることができます。</span><span class="sxs-lookup"><span data-stu-id="f8cfd-114">Likewise, Lync Server makes it easy to enable and disable monitoring after monitoring has been deployed.</span></span> <span data-ttu-id="f8cfd-115">たとえば、監視を展開するが、最初は QoE 監視を無効にする場合があります。</span><span class="sxs-lookup"><span data-stu-id="f8cfd-115">For example, you might choose to deploy monitoring but initially leave QoE monitoring disabled.</span></span> <span data-ttu-id="f8cfd-116">ユーザーが音声またはビデオ通話の問題に遭遇し始めた場合は、QoE 監視を有効にし、そのデータを使用してトラブルシューティングとそれらの問題の解決を行います。</span><span class="sxs-lookup"><span data-stu-id="f8cfd-116">If your users begin to experience problems with audio or video calls you could then enable QoE monitoring and use that data to help you troubleshoot and resolve those problems.</span></span>

<span data-ttu-id="f8cfd-117">Lync server をインストールした後に監視をインストールすることは、特に有利ではありません。また、Lync Server をインストールした後に監視をインストールすることもできません。</span><span class="sxs-lookup"><span data-stu-id="f8cfd-117">There is no particular advantage (or disadvantage) to installing monitoring at the same time you install Lync Server vs. installing monitoring after Lync Server has been installed.</span></span> <span data-ttu-id="f8cfd-118">注意する必用がある点は、監視をインストールする前に、バックエンドの監視ストアをホストするコンピューターを選択し、そのコンピューターを監視に使用する前に、サポートされているバージョンの SQL Server をそのコンピューターにインストールして構成する必用があるということです。</span><span class="sxs-lookup"><span data-stu-id="f8cfd-118">The one point to keep in mind is that, before you install monitoring, you must select a computer to host the backend monitoring store, and a supported version of SQL Server must be installing and configured on that computer before that computer can be used for monitoring.</span></span> <span data-ttu-id="f8cfd-119">SQL Server をコンピューターに既にインストールしてあり、そのコンピューターを使用できる状態になっている場合は、Lync Server のインストールと同時に監視をインストールすることができます。</span><span class="sxs-lookup"><span data-stu-id="f8cfd-119">If you have already installed SQL Server on a computer and that computer is ready for use then you can install monitoring at the same time you install Lync Server.</span></span> <span data-ttu-id="f8cfd-120">バックエンドコンピューターの準備ができていない場合は、Lync Server を単独でインストールして、バックエンドコンピューターを使用する準備ができたらいつでも監視をインストールすることができます。</span><span class="sxs-lookup"><span data-stu-id="f8cfd-120">If you do not have a backend computer ready then you can proceed to install Lync Server by itself, then install monitoring whenever the backend computer is ready for use.</span></span>

<span data-ttu-id="f8cfd-121">**監視をインストールするタイミング**</span><span class="sxs-lookup"><span data-stu-id="f8cfd-121">**When do you want to install monitoring?**</span></span> <span data-ttu-id="f8cfd-122">Lync Server 2013 をインストールおよび構成するときに、監視をインストールして構成することができます。Lync Server 展開ウィザードは、セットアップ時にフロントエンドプールを監視データベースに関連付ける機会を提供します。</span><span class="sxs-lookup"><span data-stu-id="f8cfd-122">Monitoring can be installed and configured at the same time you install and configured Lync Server 2013; the Lync Server Deployment Wizard will provide you with the opportunity to associate your Front End pools with a monitoring database during setup.</span></span> <span data-ttu-id="f8cfd-123">または、Lync Server 自体がインストールされた後に監視をインストールすることもできます。これを行うには、トポロジビルダーを使用してフロントエンドプールとサーバーを監視データベースに関連付け、変更したトポロジを発行します。</span><span class="sxs-lookup"><span data-stu-id="f8cfd-123">Alternatively, you can install monitoring after Lync Server itself has been installed; this can be done by using Topology Builder to associate your Front End pools and servers with a monitoring database, and then publishing the revised topology.</span></span>

<span data-ttu-id="f8cfd-124">**必要なバックエンド監視データベースの数**</span><span class="sxs-lookup"><span data-stu-id="f8cfd-124">**How many backend monitoring databases do you need?**</span></span> <span data-ttu-id="f8cfd-125">1つの監視データベースで数十台のユーザーをサポートできます (Microsoft Lync Server 2010 では、監視とアーカイブの両方に併置されたデータベースでは24万ユーザーがサポートされると予測していました)。</span><span class="sxs-lookup"><span data-stu-id="f8cfd-125">A single monitoring database can support tens of thousands of users (for Microsoft Lync Server 2010, it was estimated that a collocated database for both monitoring and archiving could support 240,000 users).</span></span> <span data-ttu-id="f8cfd-126">さらに、単一の監視データベースを複数のフロントエンドプールで使用できます。組織内に3つのフロントエンドプールがある場合は、これら3つのプールすべてを同じバックエンドストアに関連付けることができます。</span><span class="sxs-lookup"><span data-stu-id="f8cfd-126">In addition, a single monitoring database can be used by multiple Front End pools; if you have three Front End pools in your organization then you could associate all three of those pools with the same backend store.</span></span>

<span data-ttu-id="f8cfd-p107">これは単に、多くの組織では、必要なバックエンド監視データベースの数を決定するときに、データベースの容量が決定的要素にならないということです。その代わり、ネットワークの速度がより重要な考慮事項になります。フロントエンド プールが 3 つあるが、そのうちの 1 つのプールが低速のネットワーク接続に配置されているとします。この場合、2 つの監視データベースを使用することをお勧めします。つまり、1 つのデータベースを良好なネットワーク接続を持つ 2 つのプールに使用し、もう一方のデータベースを低速のネットワーク接続を持つプールに使用します。</span><span class="sxs-lookup"><span data-stu-id="f8cfd-p107">This simply means that, for many organizations, database capacity will not be the deciding factor when determining the number of backend monitoring databases that will be required. Instead, a more important consideration could be network speed. Suppose you have three Front End pools, but one of those pools is located across a slow network connection. In that case, you might want to use two monitoring databases: one database to service the two pools with the good network connection, and a separate database to service the pool with the slower network connection.</span></span>

<span data-ttu-id="f8cfd-131">監視インフラストラクチャを計画するときは、Lync Server 2013 がミラーデータベースの使用をサポートしていることも考慮する必要があります。</span><span class="sxs-lookup"><span data-stu-id="f8cfd-131">When planning your monitoring infrastructure you should also take into account that Lync Server 2013 supports the use of mirror databases.</span></span> <span data-ttu-id="f8cfd-132">「データベース ミラーリング」を使用すると、データベースの 2 つのコピーを、それぞれ別のサーバー上で同時に保持することができます。</span><span class="sxs-lookup"><span data-stu-id="f8cfd-132">"Database mirroring" provides a way for you to simultaneously maintain two copies of a database, with each database residing on a different server.</span></span> <span data-ttu-id="f8cfd-133">データがプライマリ データベースに書き込まれるたびに、同じデータがミラー データベースに書き込まれます。</span><span class="sxs-lookup"><span data-stu-id="f8cfd-133">Any time data is written to a primary database that same data is also written to the mirror database.</span></span> <span data-ttu-id="f8cfd-134">プライマリデータベースが失敗するか、または使用できなくなると、単純な Lync Server PowerShell コマンドを使用してミラーデータベースに "フェールオーバー" することができます。</span><span class="sxs-lookup"><span data-stu-id="f8cfd-134">If the primary database should fail or otherwise become unavailable, you can "fail over" to the mirror database by using a simple Lync Server PowerShell command.</span></span> <span data-ttu-id="f8cfd-135">次に例を示します。</span><span class="sxs-lookup"><span data-stu-id="f8cfd-135">For example:</span></span>

    Invoke-CsDatabaseFailover -PoolFqdn atl-cs-001.litwareinc.com -DatabaseType "Monitoring" -NewPrincipal "Mirror"

<span data-ttu-id="f8cfd-136">これが重要なのは、ミラーリングでは、必要なデータベースの数が 2 倍になるからです。</span><span class="sxs-lookup"><span data-stu-id="f8cfd-136">This is important for planning purposes simply because mirroring will require you to double your required number of databases: in addition to each primary database you will need a second database to act as the mirror.</span></span>

<span data-ttu-id="f8cfd-137">**Lync Server サイトには独自のカスタム監視構成が必要ですか。**</span><span class="sxs-lookup"><span data-stu-id="f8cfd-137">**Do your Lync Server sites need their own custom monitoring configurations?**</span></span> <span data-ttu-id="f8cfd-138">Lync Server 2013 をインストールすると、CDR と QoE の構成設定のグローバルコレクションもインストールされます。これらのグローバルコレクションでは、組織全体に同じ CDR と QoE の設定を適用することができます。</span><span class="sxs-lookup"><span data-stu-id="f8cfd-138">When you install Lync Server 2013 you also install global collections of CDR and QoE configuration settings; these global collections give you the ability to apply the same CDR and QoE settings to your entire organization.</span></span> <span data-ttu-id="f8cfd-139">多くの場合、これは、すべてのユーザーに対して CDR の監視を有効にする必要があると考えられる場合があります。</span><span class="sxs-lookup"><span data-stu-id="f8cfd-139">In many cases, this will be sufficient: often-times you will want, say, to have CDR monitoring enabled for all of your users.</span></span>

<span data-ttu-id="f8cfd-140">ただし、サイトごとに異なる設定を適用する必要がある場合もあります。</span><span class="sxs-lookup"><span data-stu-id="f8cfd-140">However, there might also be times when you want to apply different settings to different sites.</span></span> <span data-ttu-id="f8cfd-141">たとえば、Redmond サイトで CDR と QoE の両方の監視を使用したいが、Dublin サイトでは CDR 監視のみを使用するとします。</span><span class="sxs-lookup"><span data-stu-id="f8cfd-141">For example, perhaps you want to use both CDR and QoE monitoring in your Redmond site, but only use CDR monitoring in your Dublin site.</span></span> <span data-ttu-id="f8cfd-142">同様に、Redmond サイトでは60日間の監視データを保持する必要がありますが、ダブリンサイトではこの種のデータを30日間保持する必要があります。</span><span class="sxs-lookup"><span data-stu-id="f8cfd-142">Likewise, you might want to retain monitoring data for 60 days in the Redmond site but only need to maintain this type of data for 30 days in the Dublin site.</span></span> <span data-ttu-id="f8cfd-143">Lync Server 2013 では、サイトスコープで CDR と QoE 構成設定の個別のコレクションを作成できます。を使用すると、各サイトを異なる方法で管理できます。</span><span class="sxs-lookup"><span data-stu-id="f8cfd-143">Lync Server 2013 allows you to create separate collections of CDR and QoE configuration settings at the site scope; that enables you to manage each site differently.</span></span> <span data-ttu-id="f8cfd-144">(これには、監視を有効または無効にすることと、データの保持期間などの管理設定を構成することが含まれます)。</span><span class="sxs-lookup"><span data-stu-id="f8cfd-144">(This includes both enabling and disabling monitoring as well as configuring management settings such as how long data is to be retained.)</span></span>

<span data-ttu-id="f8cfd-p111">監視を展開する前、または監視を展開した後、これを決定できることに注意してください。たとえば、監視を展開してから、グローバル設定を使用して組織全体を管理できます。後で変更する場合には、たとえば、Redmond サイト用の設定の別個のコレクションを作成し、これらの設定を使用して Redmond 用の監視を管理します (サイト スコープに適用された設定は、グローバル スコープに適用された設定よりも常に優先されます)。再び変更する場合は、Redmond サイトに適用された構成設定を単に削除します。サイト設定のコレクションを削除すると、設定のグローバル コレクションがそのサイトに自動的に適用されます。</span><span class="sxs-lookup"><span data-stu-id="f8cfd-p111">Note that you can make this decision before you deploy monitoring or after you deploy monitoring. (Settings applied at the site scope always take precedence over settings applied at the global scope.) If you change your mind again, you can simply delete the configuration settings applied to the Redmond site. When a collection of site settings is removed then the global collection of settings will automatically be applied to that site.</span></span>
