# <a name="manage-emergency-calling"></a>緊急通話を管理する

この記事では、緊急通話を管理するために知る必要がある概念について説明します。これには、緊急対応の住所、動的緊急対応の住所、緊急通話ルーティングに関する情報が &mdash; 含まれます。 この記事では、次の用語を使用します。

- **緊急対応** の住所 - 組織の場所の物理的または番地の住所 &mdash; を公の住所で指定します。

  たとえば、住所 *12345 North Main Street、Redmond、WA 98052* は、緊急通話を適切なディスパッチ機関にルーティングし、緊急発信者の特定を支援するために使用されます。

- **[場所** ] - 通常、フロア番号、ビル番号、航空番号、またはオフィス番号。 場所は、建物内のより正確な場所を提供するために緊急対応の住所に関連付けられている。 緊急対応の住所に関連付けられる場所の数に制限はありません。 たとえば、組織に複数のビルがある場合は、各ビルと各ビル内のすべてのフロアの場所情報を含める必要があります。  

- **緊急対応の** 場所 - 場所は、オプションの場所 &mdash; を持つ市民の住所です。 ビジネスに複数の物理的な場所がある場合は、複数の緊急対応の場所が必要な可能性があります。 

  緊急対応の住所を作成すると、この住所の一意の場所 ID が自動的に作成されます。 たとえば、緊急対応の住所に場所を追加する場合、建物の住所にフロアを追加すると、緊急対応の住所と場所の組み合わせの場所 &mdash; &mdash; ID が作成されます。  この例では、2 つの場所の IP があります。1 つは、市民アドレス用です。参加した市民アドレスと関連付けられている場所用の 1 つ。

  緊急対応の場所をユーザーまたはサイトに割り当てると、そのユーザーまたはサイトに関連付けられている一意の場所 ID です。

- **登録済み住所** - 各ユーザーに割り当てられる緊急対応の住所。 登録された住所は、静的緊急対応の住所またはレコードのアドレスと呼ばれる場合があります。 (現時点では、登録済みのアドレスは直接ルーティングではサポートされていません。 更新プログラムについては、後でもう一確認してください)。

>[!Note]
>PSTN 接続に Microsoft 通話プラン、オペレーター Connect、または直接ルーティングを使用するかどうかによって、緊急通話の管理方法にいくつかの[違いがあります](pstn-connectivity.md)。 これらの考慮事項については、この記事全体で説明します。

## <a name="emergency-address-validation"></a>緊急対応の住所の検証

緊急対応の住所をユーザーまたはネットワーク識別子に割り当てるには、緊急対応の住所が "検証済み" としてマークされている必要があります。 アドレスの検証により、アドレスが正当であり、割り当て後に変更できないことが保証されます。 

Teams 管理センターのアドレス マップ検索機能を使用して緊急対応の住所を定義すると、その住所は自動的に検証済みとしてマークされます。 住所の形式または表現が変更された場合は、検証済み緊急対応の住所を変更できないので、更新された形式で新しい住所を &mdash; 作成する必要があります。


## <a name="emergency-address-geo-codes"></a>緊急対応の住所の geo コード

各緊急対応の住所には、地理コード (緯度と経度) を関連付けできます。 これらの geo コードは、動的な場所での緊急通話のルーティングを支援するために、一部の国で使用されます。 

Teams 管理センターのアドレス マップ検索機能を使用して緊急対応の住所を定義すると、geo コードは緊急対応の住所に自動的に関連付けされます。 PowerShell を使用してアドレスを定義する場合は、geo コードをアドレスに関連付けすることもできます。 

microsoft では、Teams 管理センターでマップ検索機能を使用して緊急対応の住所を作成し、住所の書式設定、検証、適切な geo コードの設定を行う方法をお勧めします。 

>[!Important]
>緊急対応の場所を動的緊急通話のネットワーク識別子に割り当てるには、緊急対応の住所に適切な geo コードを含める必要があります。

### <a name="emergency-call-enablement-for-calling-plans"></a>通話プランの緊急通話の有効化

各通話プランのユーザーは、緊急通話に対して自動的に有効になります。割り当てられた電話番号に関連付けられている登録済みの緊急対応の住所が必要です。 

電話番号に関連付けられている場所は、国/地域によって異なります。

- たとえば、米国とカナダでは、ユーザーに番号を割り当てるときに緊急対応の場所が必要になります。

- ヨーロッパ、中東、アフリカ (EMEA) などの他の国では、Microsoft 365 から電話番号を取得する場合、または別のサービス プロバイダーまたは運送業者から電話番号を転送するときに緊急対応の場所が必要です &mdash; &mdash; 。


### <a name="dynamic-emergency-calling-for-calling-plans"></a>通話プランの動的緊急通話

通話プランの動的緊急呼び出しは、クライアントの現在の場所に基づいて緊急通話を構成およびルーティングTeamsします。 適切な公衆安全応答ポイント (PSAP) への自動ルーティングを行う機能、またはセキュリティ デスクの担当者に通知する機能は、Teams ユーザーの使用国によって異なります。  

米国では、次のように、緊急通話をルーティングする動的な場所がサポートされています。 

- 米国通話プランのユーザーの Teams クライアントが米国内で緊急対応の住所を動的に取得した場合、その住所は登録された住所ではなく緊急ルーティングに使用され、そのアドレスのサービス領域の PSAP に自動的にルーティングされます。

- 米国通話プランTeamsの顧客が米国内で緊急対応の住所を動的に取得しない場合は、登録された緊急対応の住所を使用して通話の画面とルーティングを行います。 ただし、呼び出し元を適切な PSAP に接続する前に、更新されたアドレスが必要かどうかを判断するために、呼び出しが画面に表示されます。

緊急通話をルーティングする動的な場所は、カナダでは米国と同じようにサポートされます。ただし、すべての緊急通話は、PSAP に転送される前に国内で表示されます。

詳細については、「動的緊急通話の [計画と構成」を参照してください](configure-dynamic-emergency-calling.md)。

### <a name="emergency-call-routing-for-calling-plans"></a>通話プランの緊急通話ルーティング

通話プランTeamsユーザーが緊急電話番号にダイヤルすると、呼び出しが PSAP にルーティングされる方法は、次の条件によって異なります。

- 緊急対応の住所がクライアントによって動的にTeamsかどうか。

- 緊急対応の住所が、ユーザーの電話番号に関連付けられている登録済み住所かどうか。

- その国の緊急通話ネットワーク。

次に例を示します。

**米国の場合:**

- テナントがTeamsが定義した動的緊急対応の場所にクライアントが存在する場合、そのクライアントからの緊急通話は、その地理的な場所を提供する PSAP に自動的にルーティングされます。

- Teams クライアントがテナント定義の動的緊急対応の場所に存在しない場合、そのクライアントからの緊急通話は、その地理的な場所にサービスを提供する PSAP に通話を転送する前に、呼び出し元の場所を特定するために、国内のコール センターによってスクリーンされます。

- 緊急発信者が緊急対応の場所をスクリーニング センターに更新できない場合、通話は発信者の登録された住所を提供する PSAP に転送されます。

**カナダ、アイルランド、** 英国では、緊急通話が最初に画面に表示され、通話を適切なディスパッチ センターに接続する前に、ユーザーの現在の場所が決定されます。

**フランス、ドイツ**、スペインでは、緊急通話は、発信者の場所に関係なく、番号に関連付けられている緊急対応の住所を提供する PSAP に直接ルーティングされます。

**オランダでは、** 発信者の場所に関係なく、緊急通話は番号のローカル エリア コードの PSAP に直接ルーティングされます。

**オーストラリアでは**、緊急対応の住所が構成され、運送業者パートナーによってルーティングされます。

**日本では**、緊急通話はサポートされていません。

## <a name="considerations-for-direct-routing"></a>ダイレクト ルーティングに関する考慮事項

次のセクションでは、ダイレクト ルーティング ユーザーの緊急通話を管理する方法について説明します。直接ルーティングがビジネスに適切なソリューションであるかどうかを確認するには、「PSTN 接続オプション」 [を参照してください](pstn-connectivity.md)。

### <a name="emergency-call-enablement-for-direct-routing"></a>ダイレクト ルーティングの緊急通話の有効化

直接ルーティングの場合は、Teams 緊急通話ルーティング ポリシー (TeamsEmergencyCallRoutingPolicy) を使用して緊急電話番号と関連するルーティング先を定義することで、ユーザーの緊急通話ポリシーを定義する必要があります。 (現時点では、登録済みの緊急対応の場所は、直接ルーティング ユーザーではサポートされていません)。

緊急通話ルーティング ポリシーを直接ルーティング ユーザー アカウント、ネットワーク サイト、または両方に割り当てできます。 クライアントがTeams接続を開始または変更すると、Teams は、クライアントがあるネットワーク サイトの参照を次のように実行します。

- 緊急通話ルーティング ポリシーがサイトに関連付けられている場合は、サイト ポリシーを使用して緊急通話を構成します。

- サイトに関連付けられている緊急通話ルーティング ポリシーがない場合、クライアントが未定義のサイトに接続されている場合、またはダイヤルされた番号が、サイトに関連付けられている緊急通話ルーティング ポリシーで定義されている緊急電話番号と一致しない場合は、ユーザー アカウントに関連付けられている緊急通話ルーティング ポリシーを使用して緊急通話を構成します。 

- クライアントTeams緊急通話ルーティング ポリシーを取得できない場合、ユーザーは緊急通話を有効にできません。


### <a name="dynamic-emergency-calling-for-direct-routing"></a>ダイレクト ルーティングのための動的緊急呼び出し

ダイレクト ルーティングの動的緊急呼び出しでは、クライアントの現在の場所に基づいて緊急通話を構成およびルーティングTeamsします。 適切な公衆安全応答ポイント (PSAP) への自動ルーティングを行う機能、またはセキュリティ デスクの担当者に通知する機能は、Teams ユーザーの使用国によって異なります。

ダイレクト ルーティング ユーザーの場合、緊急通話をルーティングする動的な場所は、次のように米国でのみサポートされます。

-   米国ダイレクト ルーティング ユーザーの Teams クライアントが米国内で緊急対応の住所を動的に取得した場合、そのアドレスは緊急ルーティングに使用され、そのアドレスのサービス領域の PSAP に自動的にルーティングされます。

-   米国直接ルーティング ユーザーの Teams クライアントが米国内で緊急対応の住所を動的に取得しない場合、呼び出し元を適切な PSAP に接続する前に、更新されたアドレスが必要かどうかを判断するために通話が画面に表示されます。

緊急通話をルーティングする動的な場所は、カナダでは米国と同じようにサポートされます。ただし、すべての緊急通話は、PSAP に転送される前に国内で表示されます。

詳細については、「動的緊急通話の [構成」を参照してください](configure-dynamic-emergency-calling.md)。

### <a name="emergency-call-routing-for-direct-routing"></a>ダイレクト ルーティングの緊急通話ルーティング

直接ルーティングの緊急通話ルーティング ポリシーは、オンライン PSTN の使用を参照します。これは、緊急通話を適切な PSTN ゲートウェイに適切にルーティングするために適切な直接ルーティング構成が必要です。 特に、緊急ダイヤル文字列に OnlineVoiceRoute が設定されている必要があります。 詳細については、「[ダイレクト ルーティングを構成する](direct-routing-configure.md)」を参照してください。 

> [!NOTE]
> Teams、緊急電話番号の前に "+" 記号が付加されなくなりました。つまり、+911 です。 そのため、Teams 911 番号の前に "+" が送信されなくなりました。 音声ルート パターンにこの変更が反映されている必要があります。

ダイレクト ルーティング ユーザーの緊急通話を動的にルーティングする機能は、特定の国内の緊急通話ネットワークによって異なります。 次の 2 つのソリューションを使用できます。

- [緊急ルーティング サービス プロバイダー (米国のみ)](#emergency-routing-service-providers)
- [緊急対応の場所識別番号アプリケーション](#emergency-location-identification-number-applications)

#### <a name="emergency-routing-service-providers"></a>緊急ルーティング サービス プロバイダー

米国には、発信者の場所に基づいて緊急通話を自動的にルーティングできる多数の認定緊急ルーティング サービス プロバイダー (ERSP) があります。

- 緊急ルーティング サービス プロバイダーが直接ルーティング展開に統合されている場合、動的に取得された場所を持つ緊急通話は、その場所を提供するパブリック セーフティ応答ポイント (PSAP) に自動的にルーティングされます。

-  動的に取得された場所がない緊急通話は、最初に、更新された場所に基づいて適切なディスパッチ センターに通話を接続する前に、ユーザーの現在の場所を決定するために画面が表示されます。

詳細については、直接ルーティングの [認定を受けたセッション ボーダー コントローラーに関するページを参照してください](direct-routing-border-controllers.md)。


#### <a name="emergency-location-identification-number-applications"></a>緊急対応の場所識別番号アプリケーション

セッション ボーダー コントローラー (SBC) には、緊急位置情報識別番号 (ELIN) アプリケーションを含めできます。 SBC ELIN アプリケーションが直接ルーティング展開に統合されている場合は、ELIN アプリケーションで緊急対応の住所と関連する電話番号を構成し、それぞれの PSTN の緊急通話データベースに ELIN レコードをアップロードする必要があります。 Teams ELIN 識別子を持つ緊急対応の場所は、ELIN アプリケーション内の緊急対応の場所と一致する必要があります。

動的に取得された場所を含む緊急通話が適切な SBC にルーティングされる場合、ELIN アプリケーションは次の処理を行います。

- 呼び出し元の緊急対応の場所を解析します。
- ELIN レコードの場所と一致します。
- 緊急発信者番号を ELIN 電話番号に置き換える。
- その場所を提供する PSAP に呼び出しをルーティングし、ディスパッチャーはアップロードされた ELIN レコードから場所を取得します。

緊急電話番号にコールバックすると、ELIN アプリケーションは、元の緊急呼び出し元の番号に対して逆の番号置換を行います。 

詳細については、直接ルーティングの [認定を受けたセッション ボーダー コントローラーに関するページを参照してください](direct-routing-border-controllers.md)。


## <a name="security-desk-notification"></a>セキュリティ デスクの通知

セキュリティ デスク通知は、Microsoft 通話プラン、オペレーター サービス、および直接ルーティングConnect使用できます。

Teams 緊急通話ポリシー (TeamsEmergencyCallingPolicy) を使用して、緊急通話中に通知を受け取るユーザーと、その通知方法を構成します。チャットのみ、会議中とミュート状態、会議中とミュート状態ですがミュート解除機能があります。 また、緊急通話を呼び出して参加するユーザーまたはグループの外部 PSTN 番号を指定できます。 PSTN パーティーのミュート解除は許可されません。

緊急通話ポリシーは、ネットワーク サイトTeams割り当てられたユーザー アカウント、または両方に付与できます。  クライアントがTeamsまたはネットワーク接続を変更すると、クライアントがTeamsがあるネットワーク サイトの参照が実行されます。

- 緊急通話ポリシーがネットワーク サイトに関連付けられている場合、サイト ポリシーはセキュリティ デスク通知の構成に使用されます。

- サイトに関連付けられている緊急通話ポリシーがない場合、またはクライアントが未定義のサイトに接続されている場合は、ユーザー アカウントに関連付けられている緊急通話ポリシーを使用して、セキュリティ デスク通知が構成されます。  

- クライアントTeams緊急通話ポリシーを取得できない場合、ユーザーはセキュリティ デスク通知を有効にしません。

緊急通話中は、セキュリティ デスクが通話に電話会議を行い、セキュリティ デスク ユーザーのエクスペリエンスは、緊急通話ポリシーに基Teams制御されます。 各セキュリティ デスク メンバーとグループ チャットが開始され、緊急呼び出し元の場所は重要なメッセージ通知を介して共有されます。  会議オプションがポリシーの一部として構成されている場合、各セキュリティ デスク ユーザーは会議の一部として追加で呼び出されます。
