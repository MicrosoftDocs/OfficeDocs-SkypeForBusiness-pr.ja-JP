### <a name="what-is-the-cqd"></a>救難とは何ですか。

呼び出し品質ダッシュ ボード (救難) を使用するには、ビジネス ・ サービスのチームと Skype を使用して行われた通話の品質を把握します。 救難、Skype をビジネスおよびチームの管理者のために設計されていて、ネットワーク ・ エンジニアは、ネットワークを最適化します。 救難は、組織全体の情報を集計、全体的なパターンが明らかになります、スタッフに通話品質の情報に基づく評価を許可します。 救難では、呼び出しの全体的な品質、呼び出しの信頼性、およびユーザー エクスペリエンスの手掛かりを得るコール メトリックスのレポートを提供します。

> [!NOTE]
> 救難には、個人を特定できる情報 (PII) が含まれていません。 個人情報は、使用できる独自に、またはその他の情報を識別、取引先担当者、または 1 人のユーザーを検索するか、コンテキスト内の個人を識別する情報です。

#### <a name="audio-pcr-"></a>オーディオ PCR % 

オーディオ不適切な呼び出しの比率 (PCR) は、組織の全体的な率の音質が悪い呼び出しを表します。 このメトリックは、組織がこの値を小さくして PCR を見ると、管理対象のネットワークが主な目的は、ユーザーの操作性の向上に向けた最も強力な影響を与えるには、残存作業時間を集中できる領域を強調表示するものです。 外部ユーザーは、重要ですが、調査は、組織とユーザーごとに異なります。
外部のユーザーに対して、ベスト ・ プラクティスを提供することを検討してくださいし、組織全体とは別の外部の呼び出しを確認します。

#### <a name="call-setup-failures-"></a>呼び出しセットアップ障害 % 

これは、任意のメディア ・ セッションを確立することができませんでしたを表します。 影響の重大度を指定すると、ここで測定されるユーザー エクスペリエンスで、目標は、この値を小さくと可能な限りゼロに近いします。 このメトリックの値が高いより完成度の高い配置では、不完全なファイアウォール規則の新しい展開で一般的なが、定期的に監視することも重要です。 運用上の厳格さが完成に近づくにつれて、ビデオ、およびデスクトップ共有の作業負荷を含めるには、このメトリックを展開できます。

#### <a name="call-drop-failures-"></a>ドロップの呼び出しの失敗の % 

これは、セッションが予期せず終了オーディオのワークロードに適用されます。 運用上の厳格さが完成に近づくにつれて、ビデオ、およびデスクトップ共有の作業負荷を含めるには、このメトリックを展開できます。

### <a name="user-experience"></a>ユーザー エクスペリエンス

科学よりも多くのアートは、ユーザーの操作性の分析、測定値がここで得られたためネットワークまたはサービスの問題があるではなく、それぞれのユーザーが問題を認識していることを意味しない常にします。 マイクロソフトは、組み込みの調査方法を提供しています-としてレート自分を呼び出す (RMC) を呼ばれる-全体的なユーザー エクスペリエンスの測定に役立つようにします。 RMC では、ユーザーの観点から次の質問に回答できます。

-   ソリューションを使用する方法を知っていますか。

-   使いやすく、直感的なソリューションし、は、日常的なコミュニケーションのニーズをサポートしますか。

-   ソリューションに役立つ自分の業務を遂行しますか。

-   ソリューションの私の全体的な認識とは何ですか。

-   使用できるソリューションの任意の時点で私は場所に関係なく、時間でしょうか。

-   設定し、呼び出しを管理できますか。

#### <a name="rmc"></a>RMC

RMC は、チーム、およびビジネスのための Skype に組み込まれているし、10 個の呼び出し、またはすべての呼び出しの 10% ごとに 1 つの後に表示されるように構成されて自動的に。 この簡単なアンケートでは、呼び出しを評価し、なぜ、通話の音質がありますが不十分の少ないコンテキストを提供するユーザーを確認します。 1 つまたは 2 つの評価が低いと見なされます、3 ~ 4 が良いとは優れた 5。 ですが、多少の遅れが生じているインジケーターが、これは、サービス メトリックスを見逃すことがある問題を明らかに便利なメトリックです。

### <a name="stream-vs-call"></a>ストリームの呼び出しとの比較

どのディメンションまたはメジャーを見るに救難を正しく選択するには、呼び出し、ストリームの違いを理解する必要があります。

**ストリーム:** ストリームは、2 つのエンドポイント間で存在します。 、方向ごとに 1 つのストリームがあるし、2 つのストリームは、通信に必要です。 ストリームは、建物、またはネットワークを分析するために便利です。 場合によっては、名前 (たとえば、呼び出しセットアップのストリームまたは呼び出しドロップ ストリーム) の呼び出しとストリームの両方が使用されます。
これらは、まだ 1 つのストリームとして分類されます。

**の呼び出し:** 呼び出しは、すべての参加者からのすべてのストリームのグループです。 呼び出しで構成されます-少なくとも、2 つのストリーム。 1 回の呼び出しは、2 つの参加者ごとに最低 1 つのストリームにがあります。 呼び出しは、長期の傾向を分析するために役立ちます。

呼び出しやストリームにディメンションまたはメジャーを参照するかどうかについての追加の[ディメンションとメジャーの救難に使用可能な](https://aka.ms/cqd-dm)を参照してください。

#### <a name="cqd-filter"></a>救難フィルター

救難フィルター、または、URL フィルターを使用するには、一時的にすべてのレポートのクエリをフィルター処理します。 最も一般的な救難フィルターを使用しますがフェデレーション参加者の「遠隔測定」を除外するのにはレポートをフィルター処理します。 既定のビューになりますのでこのフィルターにブックマークを設定することをお勧めします。 救難レポートから連合のデータを除外すると、管理対象の建物または連合のデータがレポートに影響がありますネットワークを修復しているときに便利です。

/filter/[AllStreams]。[2 番目のテナント Id]\|[、テナント ID]

https://cqd.lync.com/cqd/\#/1234567/2018-02/filter/[AllStreams]。[2 番目のテナント Id]\|[TENANTID] & テナント = TENANTID

### <a name="investigate-tcp-audio-sessions"></a>オーディオの TCP セッションを調査します。

TCP は、フェイル バックのトランスポートおよびリアルタイム メディアに使用するプライマリ トランスポートではないと見なされます。 フェイル バックのトランスポートが TCP のステートフルな性質です。 潜在的なネットワーク上の呼び出しが行われ、メディア パケットが遅延する場合など、数秒前からのパケットを-はもはや使いになる-、悪い状況が悪化すると、受信機に到達する帯域幅の競合が発生します。 これは、オーディオの治癒者ステッチとジッターの形式では多くの場合音の成果物で、ストレッチのオーディオです。

このセクションのレポートには、良いと不適切な呼び出しの間で区別を作成しません。 UDP が多いこと、レポートは、オーディオの TCP の使用を探します。 これは、不完全なファイアウォールの規則によって主に発生します。 ビジネスをオンラインでのチームと Skype のファイアウォール規則の詳細については、 [Office 365 の Url と IP アドレスの範囲](https://aka.ms/o365ips)を参照してください。

### <a name="devices-investigations"></a>デバイスの調査

使用して、次のデバイスのレポートの平均の意見の概念を理解しておくことをおにスコア (MOS)。 MOS は、認識されるオーディオの品質を評価するための最高水準の測定です。 0 から 5 の整数評価として表されます。

音声品質のすべてのメジャーの単位は、ユーザーが音声の品質を判断する方法です。 人間の知覚によって影響を受けるためには、本質的に主観的です。 主観的なテストのいくつかのさまざまな方法論があります。
ほとんどの音声品質の測定は、絶対の分類の評価 (ACR) スケールに基づいています。

ACR の主観的なテストでユーザー数が統計的に有意な評価の経験、品質 1 のスケールで (不良) 5 (優)。 スコアの平均値では、MOS です。 結果の MOS は、グループと評価されている操作性を提供されていた経験の範囲に依存します。

ライブ通信システムの音声品質の主観的なテストを実施するのには実用的でないため、チーム、およびビジネスのための Skype 生成 MOS 値客観的主観的なテストの結果を予測するために高度なアルゴリズムを使用しています。

MOS と関連する基準の利用可能なセットは、ユーザーに配信されている経験の質にビューを提供します。

チームやビジネス用の Skype の認定デバイスとユーザーを指定して、デバイス自体 (である可能性が高く、ラップトップ内蔵のスピーカーとマイクロ フォンを持つなど) のための負の経験が発生する可能性を減らします。 詳細については、[電話や Skype をビジネスのためのデバイス](https://technet.microsoft.com/office/dn947482)を参照してください。
