<a name="define-and-document-your-audio-conferencing-in-teams-test-plan-for-enterprises"></a>定義し、企業のチームのテスト計画で、オーディオ会議を文書化 
===============================================================================

機能を通じて、組織の目標と要件が満たされている次の手順を検証するを定義し、チームのビジネスの成功と技術的な実装計画でのオーディオ会議を Envision フェーズの一部として文書化して後、機能と使いやすさ。 運用環境でパイロットや最後の展開を展開する前に、この検証の手順を行う必要があります。

テスト段階で評価されるように活動、期待される、機能のテストの場合、および全体的な範囲を決定する基準として使用する Envision フェーズで定義したビジネスの成功プランを利用できます。

<a name="identify-testing-support-stakeholders"></a>テストのサポート関係者を識別します。
-------------------------------------

電話会議機能を評価するために準備するときは、テストのフェーズをサポートするために必要な役割を識別するテスト サポート利害関係者行列を作成します。

> [!TIP]
> 利害関係者のサポート ・ マトリックスをテストするチームを設定すると、表示されるいくつかの役割は、テストに向けた斜めロールの説明を持つが、Envision フェーズでは、指定されていると同じです。 テストのシナリオでは、固有の要件に応じて、追加の役割を識別する必要があります。

<a name="define-audio-conferencing-feature-requirements"></a>電話会議機能要件を定義します。 
-----------------------------------------------

スコープ内でユーザーの電話会議の機能の要件の定義の一部としては、[ペルソナの分析](https://docs.microsoft.com/MicrosoftTeams/audio-conferencing#assess-environment-and-evaluate-adoption-readiness)で、オーディオ会議のペルソナとシナリオを定義した Envision フェーズ中に作成した最初の手順のいずれかでした。 このベースラインを識別すると、次の手順を決定するのには最新のチーム パブリック ロードマップを評価するためには。

-   電話会議機能スコープ内でユーザーに対して展開するでしょう。

-   ビジネス、Exchange、および SharePoint の展開の横に、現在 Skype を与えられたユーザーの電話会議の機能要件が必要です。

-   最新のパブリック ・ ロードマップに記載されている電話会議機能が、ユーザー、機能、および配置のタイムラインでスコープの要件を満たすことを確認するかどうか

> [!TIP]
> 展開があるは、スコープ内の電話会議機能を識別するための最新のチーム ロードマップ<https://aka.ms/skype2teamsroadmap>。

オーディオ会議のペルソナと機能を定義すると、評価のための次の条件がチームとの相互運用性の経験になります。 使用可能な構成オプションとの相互運用性の経験についての詳細については、[マイクロソフトのチームとビジネスの相互運用のための Skype](https://docs.microsoft.com/microsoftteams/teams-and-skypeforbusiness-coexistence-and-interoperability)を参照してください。

<a name="define-and-document-your-audio-conferencing-test-plan"></a>定義し、会議のオーディオのテスト計画を文書化
-----------------------------------------------------

電話会議機能を定義したスコープ内で後、は、テスト計画を作成します。 レベルが高い場合、テスト計画には、全体のテスト方法とテスト プロセスの機能検証をサポートするために使用する方法が含まれます。

レベルが高い場合、テスト計画を含める必要があります。

-   **スコープをテスト:** テスト フェーズの一部として (目標、シナリオ、および目標) を評価するのには、重点分野をまとめたもの。

-   **テストの場合:** スコープ内の電話会議機能を検証するテスト_ケースのセット

-   **リソースをテスト:** 環境、技術、および担当者の視点からのテスト作業をサポートするために必要なリソースのマトリックス

-   **不具合の報告と改善策:** どのテストで問題を報告するか、履歴、および優先度を付けたためのガイドライン

-   **の成果物のテスト:** 結果の開発および承認を承認し、テスト プロセスの終了をサポートするために提供のうちの概要

### <a name="define-and-document-test-defect-report-criteria"></a>定義し、テストの問題のレポートの抽出条件を文書化

特定のフェーズまたはストリームに含まれるテスト_ケースが実行されるため、実行するテスト_ケースの結果が予想されていない、問題が起こる可能性があります。

これらの種類の結果が発生するは、問題レポートと改善計画レビュー、レポート、および問題解決のテスト専用の潜在顧客へのエスカレートにキャプチャする必要があります。

通常、問題のレポートと改善計画は、次の。

-   **解除 ID:** 問題に割り当てられた番号

-   **影響を受けるテスト_ケース ID**: 欠陥の識別時に評価されているテスト_ケースに割り当てられている番号

-   **不具合の説明:** 問題の概要

-   **環境と再現手順:** テスト環境のセットアップの問題を再現する正確な手順の概要

-   **障害の重大度**: テスト_ケースを防止、受け入れられているリスクを最小限にするには承認されるまで、問題の影響です。 いくつかの例があります。

-   **低:** 問題は、ほとんど影響し、後で問題を解決できる場合は、受け入れを実現するためのテスト_ケースは拒否されません。

-   **メディア:** 問題は多大な影響を与えますが、テスト段階が完了する前に、問題が解決できる場合は、受け入れを実現するためのテスト_ケースは拒否されません。

-   **高:** の問題が、テスト_ケースの重要な影響を持ちます。 テスト_ケースを認めることができる前に、問題が解決する必要があります。

-   **の状態:** 問題が解決されると、開いているか、現在調査中

-   **によって提出された:** 問題を報告したテスト担当者

-   **割り当て所有者:** 問題の解決を担当するテスト チームから割り当てられたリソース

-   **詳細情報の提供:** これを含めることができます-に制限はありません -クライアント側のログ、スクリーン ショット、またはビデオの問題です。

これは、影響を及ぼすことやテストの評価プロセスを停止する場合でも、潜在的な影響を強調表示されます。

### <a name="define-and-document-the-defect-escalation-process"></a>定義し、問題のエスカレーション ・ プロセスを文書化

計画実行のテストのコースでは、問題を検出またはを推進し、テストを続行できるようにするのには必要な解像度を決定するための最適なリソースへのエスカレーションが必要な欠陥を識別する可能性があります。

エスカレーションの行列を作成して、エスカレーションがトリガーされ、不具合に割り当てられるをさらにレビューをマッピングするためのプロセスの確認を優先順位づけの欠陥は、適切な重大度と優先度が割り当てられていると識別されると解像度です。

通常、問題のレポートと改善計画は、次の。

-   **解除 ID:** 番号に割り当てられた問題

-   **不具合の説明:** 問題の概要

-   **不具合の優先度の評価:** ビジネスへの影響や欠陥に基づいて、問題解決のために割り当てられた優先度のレベルの重大度。 いくつかの例を含める可能性があります。

-   **低:** 問題は、後で問題を解決できる場合の承認を達成するためのテスト フェーズを防止するのにはほとんど影響を持ちません。

-   **メディア:** 問題は、テスト段階が問題を解決できない場合の承認を達成するを防ぐに多大な影響を持ちます。

-   **高:** 問題は、テスト段階が問題を解決できない場合の承認を達成するを防ぐに重要な影響を持ちます。

-   **割り当てられた欠陥の所有者:** 問題の解決を担当するテスト チームから割り当てられたリソース

-   **問題のエスカレーション ・ ポイントが割り当てられている:** 解像度を推進するため (必要な場合)、組織内の問題を解決するためのポイントには、どのリソースが割り当てられています。障害の重大度に基づく

-   **状態を解除する:** 問題が解決されると、開いているか、現在調査中

-   **日で解像度を必要な:** 問題を解決する必要がある日付

-   **状況報告日:** 欠陥トリアージのレビューの結果として時間の最新のステータスを反映した日付が更新されました
