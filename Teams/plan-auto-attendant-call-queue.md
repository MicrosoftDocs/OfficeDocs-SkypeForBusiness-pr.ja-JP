# <a name="plan-for-teams-auto-attendants-and-call-queues"></a>Teams の自動応答と通話キューを計画する

自動応答を使うと、発信者の入力に基づいて着信をルーティングするためのメニューオプションを設定できます。 メニューオプション (「売上」、「1」など)  サービスについては、自動応答の場合、「2」と入力すると、着信通話の処理に人間のオペレーターを使用することなく、発信者にすばやく誘導する一連の選択肢が提供されます。

通話キューは、発信者の待機領域です。 特定の担当者ではなく、特定の専門分野 (販売またはサービスなど) を使用して、発信者が連絡を取る必要がある場合は、通話キューを使って、発信者をサポートできるエージェントグループに接続することができます。 キューに割り当てられているエージェントが通話を受けることができるようになるまで、発信者は保留になります。

自動応答と通話キューを併用すると、発信者を組織内の適切なユーザーまたは部門に簡単にルーティングすることができます。

## <a name="auto-attendants"></a>自動応答

自動応答の主な目的は、指定したメニューオプションへの呼び出し元の入力に基づいて、適切なユーザーまたは部門に発信者に指示することです。 発信者は、組織内の特定のユーザーに転送したり、次の使用可能なエージェントへの通話を待機するキューを呼び出したり、ボイスメールを送信したりすることができます。 営業時間、休暇、休日のさまざまな通話ルーティングオプションを指定できます。

メニュープロンプトは、音声合成 (システムによって生成されるプロンプト) を使用するか、録音された音声ファイルをアップロードすることで作成できます。 音声認識では、ハンズフリーのナビゲーション用に音声コマンドが受け入れられますが、通話発信者は、電話のキーパッドを使ってメニュー間を移動することもできます。

各自動応答には、特定の言語とタイムゾーンがあります。 複数の言語または世界の複数の部分で事業を行っている場合、発信者に対応するために必要なだけさまざまな自動応答を作成できます。

自動応答ごとに、オペレーターを構成できます。 さまざまな宛先に移動するようにオペレーター呼び出しを構成することはできますが、オペレーター機能は、発信者が組織内の特定のユーザーと会話できるように設計されています。

自動応答を構成して、発信者が名前または内線番号で組織のディレクトリを検索できるようにすることができます。 自動応答では、ユーザーのグループに含めるか除外するかを選択することによって、ディレクトリ検索で利用可能なユーザーを指定できます。 (これは「 *ダイヤルスコープ* 」と呼ばれます)

発信者は、直通電話番号 (構成されている場合)、または別の自動応答または通話キューからリダイレクトされた場合に、自動応答に到達できます。

## <a name="call-queues"></a>通話キュー

通話キューは、物理的な建物内の待機中の会議室と似ています。 呼び出しは保留になり、キュー内のエージェントが利用可能になったらキューにルーティングされます。 通話キューは、一般に営業およびサービス機能に使用されます。 ただし、通話の数が内部の容量を超過している場合は、通話キューを使用することができます。これには、取り込み中の施設の受付などが含まれます。

通話キューによって、キュー内の発信者の合計数または待機時間が指定した限度を超えている場合に、通話の特定のルーティングが許可されます。 通話は、特定のユーザー、ボイスメール、その他の通話キュー、または自動応答にルーティングすることができます。

自動応答と同じように、各通話キューには [言語設定] があります。 複数の言語で事業を行っている場合は、別の通話キューを使用できます。 複数のキューが多言語の場合、エージェントは複数のキューのメンバーになることができます。

通話キューごとに、キューのエージェントが通話を受けられないようにするかどうかを指定できます。また、Teams でのプレゼンス表示に基づいて、通話をルーティングする必要があります。

電話番号を通話キューに割り当てることはできますが、通話キューによって、休暇と休日の個別の通話ルーティングは提供されません。 通話キューに24/7 が割り当てられていない場合は、勤務時間中に通話キューにリダイレクトする自動応答に電話番号を割り当てることをお勧めします。

## <a name="prerequisites"></a>前提条件

自動応答と通話キューを構成するには、次のリソースが必要です。

- 各自動応答と各通話キューのリソースアカウント
- 各リソースアカウントの無料電話システム仮想ユーザーライセンス
- 直接 dialable する各自動応答または通話キューのサービス番号
- 通話キュー通話を受信する各ユーザの通話プラン

> [!NOTE]
> 自動応答と通話キューの直接ルーティングサービス番号は、Microsoft Teams ユーザーに対してサポートされており、通話エージェントにのみ対応しています。

## <a name="business-decisions"></a>ビジネス上の意思決定

自動応答と通話キューを設定する前に、ビジネスでこれらの機能を使用する方法についていくつかの決定を行う必要があります。 これらの決定により、自動応答と通話キューを構成するときに選択した設定が決定されます。

これらの質問に対する回答を文書化し、構成を行う管理者に情報を提供します。

- 必要な言語 これらの言語はどこで必要ですか?
- 発信者からの音声入力を許可しますか。またはダイヤルのみの入力を許可しますか?
- 休暇や休日のために別々の通話ルーティングが必要ですか? 時間と休日は何ですか?
- 通話キューのエージェントが通話を受けることを許可しないようにしますか?
- 発信者がダイヤルアウトした場合に、通話キューに含まれているエージェント、または電話会社の携帯電話会社が特定の発信者番号を持つことができるようにしますか?
- 組織での [コールパーキングと取得](call-park-and-retrieve.md) を有効にして、ユーザーまたは部署間の通話引き渡しをサポートしますか?
- 音声のプロンプトについては、自分で録音またはシステムによって生成された音声を使用しますか? (システムで生成された音声は簡単に更新できます。)

## <a name="technical-decisions"></a>技術的な決定事項

自動応答と通話キューを使って発信者を組織内のユーザーに接続する場合、構成を開始する前にいくつかの技術的な決定を行う必要があります。

次のような方法で、通話キューにエージェントを追加できます。

- 個々のユーザー
- 配布リスト
- メールが有効なセキュリティグループを含むセキュリティグループ
- Microsoft 365 グループまたはチーム

必要に応じて、これらのオプションを組み合わせて各キューに使用できます。 メールアドレスを持つグループをボイスメールに使うことができます。 Teams を使用することには、エージェント間の共有ファイルストレージとチャット、ボイスメールを受け取ることができる共通のメールボックス、基幹業務アプリケーションや電源アプリとの統合を含めることができる拡張可能なプラットフォームなど、さまざまな利点があります。

構成を開始する前に、通話エージェントをキューに追加する戦略を選ぶことをお勧めします。

既存の自動応答と通話キューインフラストラクチャを使用していて、Teams に移行している場合は、既存の電話番号を新しい自動応答と通話キューに移行する計画が必要になります。 別のプロバイダーから番号を移行するには、 [ポートの順序](phone-number-calling-plans/port-order-overview.md) を作成する必要がある場合があります。 現在のサービスの番号に切り替える前に、1つまたは複数の新しい電話番号を一時的に取得し、自動応答と通話キューフローをテストすることをお勧めします。

*会議モード* は、チームの VOIP 通話と PSTN 通話をエージェントに接続するのにかかる時間を大幅に短縮する、通話キューのオプションです。 会議モードで機能するには、通話キュー内のエージェントで次のいずれかのクライアントを使用する必要があります。

- 最新バージョンの Microsoft Teams デスクトップクライアント、Android アプリ、または iOS アプリ
  - Microsoft Teams phone バージョン 1449/1.0.94.2020051601 以降
  
エージェントの Teams アカウントをチーム専用モードに設定します。 要件を満たしていないエージェントは、通話ルーティングリストに含まれません。

すべてのエージェントが互換性のあるクライアントを使用している場合は、通話キューの会議モードを有効にすることをお勧めします。

> [!NOTE]
> Busy がビジー状態であるため、会議モードでサポートされていません。 プレゼンスベースのルーティングが有効になっていない場合は、着信キュー以外の通話に含まれているエージェントでも、通話キューの呼び出しと共に表示されることがあります。

## <a name="plan-your-call-routing-flow"></a>通話ルーティングフローを計画する

計画プロセスの一環として、組織の通話ルーティングを図面で処理することをお勧めします。 この図は、組織に発信するユーザーに対して最も効率的なルーティングを決定するのに役立ちます。 また、図面を使用して、作成する必要がある自動アテンダントと通話キューと、サービス番号、ライセンス、リソースアカウントなどの関連要件を特定することもできます。

自動応答と通話キューによる通話のルーティング方法を見てみましょう。

自動応答では、次のいずれかの方法ですべての通話がルーティングされます。

- 「 **直ちにリダイレクト** 」-通話は、最初の応答メッセージの後に、通話ルーティング先 (以下に表示) のいずれかにリダイレクトできます。
- **[ダイヤルオプション] に基づいてリダイレクト** する–発信者は、電話のキーパッド (0-9) の番号に割り当てられているオプションを選ぶように指示できます。 各ダイヤルキーには、通話ルーティング先を割り当てることができます。
- **名前または内線番号を使ってユーザーをダイヤル** する-発信者は、組織のディレクトリで、またはユーザーの名前のスペルをチェックして、内線番号をダイヤルすることができます。
- **切断** -自動応答は通話を切断することができます。

> [!NOTE]
> 1つの自動応答でサポートされるのは、1つの "dial by" メソッドだけです。  発信者に名前と番号でダイヤルすることを許可するには、名前でダイヤルするオプションと内線番号をダイヤルするためのオプションを備えた自動応答を作成する必要があります。  これらの各オプションは、これらの「ダイヤル者」シナリオに対して構成されている個別の自動応答にルーティングします。

自動応答または通話キューによって着信がリダイレクトされた場合は、次の通話ルーティング先から選ぶことができます。

- **組織内のユーザー** -音声通話を受信できる組織内のユーザー。 これは、オンラインユーザーまたは Skype for Business Server を使用してオンプレミスでホストされているユーザーのいずれかになります。
- **音声アプリ** -別の自動応答または通話キュー。 リンク先に関連付けられているリソースアカウントを選択します。
- **外部電話番号** -任意の電話番号。 (「 [外部転送技術の詳細](create-a-phone-system-auto-attendant.md#external-phone-number-transfers---technical-details)」を参照してください)。
- **ボイス** メール-指定した Microsoft 365 グループに関連付けられているボイスメールボックス。
- [ **Operator** (自動応答のみ)]-自動応答に対して定義されているオペレーター。 演算子の定義は省略可能です。 演算子は、この一覧にある他の任意の移動先にすることができます。

自動応答では、営業時間外に受信した通話に対して、個別の通話ルーティングオプションが提供されます。 通話ルーティングを有効にすると、上記のすべてのオプションが有効になります。休日の通話ルーティングでは、通話のリダイレクトのみが可能ですが、ダイヤルキーオプションは使用できません。

通話キューは、キューに割り当てられているエージェントが通話を受けることができるようになるまで、発信者を保留にします。 発信者がキューから送信される可能性があるのは、次の2つの場合です。

- **呼び出しのオーバーフロー** -キュー内の呼び出しの数が設定した制限を超えている場合、新しい呼び出し元はキューからリダイレクトされます。
- [ **呼び出しのタイムアウト** ]-呼び出し元が、構成されているタイムアウト設定よりも長くキューにある場合は、キューからリダイレクトされます。

キューからリダイレクトされた呼び出しは、演算子を除き、上記のいずれかの通話ルーティング先に送信できます。 (通話キューには演算子はありませんが、自動応答用に構成したオペレーターと同じ宛先に発信者をリダイレクトすることができます。)

次の例は、自動応答と通話キューを使った通話ルーティングの例を示しています。

![自動応答と通話キューを使った通話ルーティングの図](media/attendant-and-queue-call-routing.png)

上の例では、次の操作を行います。

- ゼロ (0) キーは、呼び出し元をオペレーターにリダイレクトします。 この自動応答のオペレーターは、 **組織内のユーザー** として構成されています。
- 一方の (1) キーを呼び出すと、発信者は販売キューにリダイレクトされます。 この通話キューは、キューに割り当てられている営業チームが含まれているチームに接続されます。
- 2つのキーは、発信者をサポートコールキューにリダイレクトします。 この通話キューは、チームに割り当てられているサポートチームが含まれているチームに接続されます。
- サポートの通話キューには、介在することで自動応答があります。 自動応答を使うと、サポートラインは、個別の勤務時間と休日の通話ルーティングを許可します。
- 3つ目のキーは、会社のディレクトリの別の自動応答にユーザーをリダイレクトします。 会社のディレクトリ自動応答を使うと、発信者は名前または内線番号をダイヤルして、組織内の個々のユーザーに電話をかけることができます。

通話ルーティングをマップするために、1つ以上の図を作成することをお勧めします。 図または付属のドキュメントに次のものが含まれていることを確認します。

- 電話番号を使って直接アクセスする自動応答にはどのようなものがありますか?
- 各自動応答のオフ時間および休日のルーティング要件を教えてください。
- 各通話キューのメンバーシップ。 (ユーザーを個別に追加したり、キューを別の種類のグループに割り当てることができます。 キューをチームにマッピングすると、最も汎用性の高いエクスペリエンスが提供されます。)

通話ルーティングのベストプラクティスをいくつか紹介します。

- 既存の通話システムを確認して、着信の種類と頻度を分析します。 この情報を使って、自動応答と通話キューの構造を通知することができます。
- できるだけ早く通話をルーティングするために、最も一般的なオプションをメニューの一番上に配置します。
- キューが24/7 で利用可能でなければ、通話キューにサービス番号を直接接続しない。 通話キューによって、休暇や休日の個別の通話処理を行うことはできません。 電話番号のあるキューを作成する場合は、営業時間中に自動的にキューにリダイレクトされる自動応答に番号を割り当てます。
- 勤務時間、場所、web サイトのアドレスなど、会社に関する基本的な情報を要求する多数の通話を受信する場合は、自動応答を作成して、記録されたメッセージでこれらの質問に回答することを検討してください。
- メニュー項目の一覧は5個以下にします。 発信者が5つ以上のオプションを覚えるのに問題が発生する可能性があります。 通話を適切にルーティングするためにさらに多くのオプションが必要な場合は、ネストされた自動応答を使います。
- まずサービスについて説明します。その後に、次のような別の方法ではなく、press (売上を1回押すなど) というオプションが続きます。 [売上] の場合は1を押します)。
- 内部で使用するものではなく、呼び出し元が理解できるユーザーの用語。
- 通話ルーティングに頻繁に更新されないようにします。 今後、自動応答のメニューオプションを変更した場合は、最初の30日間は音声の入力を求められます。

## <a name="resource-accounts-and-phone-numbers"></a>リソースアカウントと電話番号

チームは *[リソースアカウント](manage-resource-accounts.md)* を使用して自動応答を接続し、キューに通話を発信し、必要に応じて電話番号を割り当てます。

- 各通話キューと自動応答には、少なくとも1つのリソースアカウントが必要です。
- 各リソースアカウントには、無料の電話システム仮想ユーザーライセンスが必要です。
- 必要に応じて、リソースアカウントに1つ以上のサービス番号を割り当てることができます。 (これは、自動応答と通話キューに電話番号が割り当てられる方法です。)

[Microsoft サービス番号](getting-service-phone-numbers.md)、直通ルーティング番号、またはハイブリッド番号をリソースアカウントに割り当てることができます。

有料または無料のサービス番号を使用できます。 自動応答または通話キューにユーザー電話番号を使用することはできません。

## <a name="getting-started"></a>はじめに

この記事の計画タスクが完了したら、次の手順に従って自動応答と通話キューをセットアップします。

1. 組織外から直接ダイヤルすることで、自動応答と通話キューに必要なサービス番号を取得します。 これには、 [別のプロバイダーからの番号の転送](phone-number-calling-plans/transfer-phone-numbers-to-teams.md) や、 [新しいサービス番号のリクエスト](getting-service-phone-numbers.md)などが含まれます。

2. 作成予定の各リソースアカウントについて、 [電話システムの仮想ユーザーライセンス](teams-add-on-licensing/virtual-user.md) を取得します。 これらのライセンスは無料なので、後でリソースアカウントを変更する場合は、さらに追加することをお勧めします。

3. 作成する自動応答と通話キューごとに[リソースアカウントを作成](manage-resource-accounts.md)します。 各アカウントに電話システム仮想ユーザーライセンスを割り当て、必要に応じてサービス番号を割り当てます。

4. 自動応答で個別の通話ルーティングを使用する[休日を作成](set-up-holidays-in-teams.md)します。

5. 必要に応じ [て](call-park-and-retrieve.md) 、この機能を使用して通話転送を行うことができます。

6. 通話キューの通話エージェントを含めるために使用するグループを作成します。

7. 内線番号によるダイヤルを許可する予定の場合は、ユーザーの内線番号を Azure Active Directory プロファイルに追加したことを確認します。

上記の手順を完了したら、自動応答と通話キューを作成することができます。 自動応答と通話キューを使用すると、通話を相互にリダイレクトできるため、作成したワークフロー図を参照して、最初に作成する自動応答または通話キューを決定する必要があります。 上の図の例では、主要な自動応答が発信者を販売およびサポートの通話キューに発信する必要があるため、販売とサポートの通話キューを作成してから、メインの自動応答を作成します。

自動応答と通話キューを作成する方法については、次の記事を参照してください。

- [自動応答を設定する](create-a-phone-system-auto-attendant.md)
- [呼び出しキューを作成する](create-a-phone-system-call-queue.md)
