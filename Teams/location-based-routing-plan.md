# <a name="plan-location-based-routing-for-direct-routing"></a>ダイレクト ルーティングの場所に基づくルーティングを計画する

## <a name="overview-of-location-based-routing"></a>場所に基づくルーティングの概要

一部の国と地域では、公衆交換電話網 (PSTN) プロバイダーを迂回して長距離通話料金を削減することはできません。 この記事では、場所ベースのルーティングを使用して、地理的な場所に基づいて Microsoft Teams ユーザーの有料のバイパスを制限する方法について説明します。 この記事は、電話システムのダイレクトルーティングにのみ適用されます。

ここでは、計画を立てるのに役立つ位置情報に基づくルーティングとガイダンスの概要について説明します。 位置情報に基づくルーティングを適用して有効にする準備ができたら、次を参照してください。

- [場所に基づくルーティングのネットワーク設定を展開する](location-based-routing-configure-network-settings.md)
- [ダイレクト ルーティングの場所に基づくルーティングを有効にする](location-based-routing-enable.md)

> [!NOTE]
> 位置情報に基づくルーティングは、Microsoft 365 Government Community Cloud (GCC) 高または DoD の展開では使用できません。

位置情報に基づくルーティングは、着信または発信の PSTN 通話の時点でのポリシーとユーザーの地理的な位置に基づいて、有料のバイパスを制限できる機能です。 位置情報に基づくルーティングは、無料の通話を防止するためのメカニズムを提供することを目的としています。 これは、ユーザーの場所に基づいて PSTN 通話を動的にルーティングするメカニズムとして使用したり、予期しない結果が生じる可能性があります。

チームユーザーが位置情報に基づくルーティングを有効にしている場合、次のようになります。

- 発信 PSTN 通話を発信するには、次のいずれかの条件を満たしている必要があります。
    - ユーザーのエンドポイントは、位置に基づくルーティングと、位置情報に基づくルーティングが有効になっているゲートウェイ経由での出口の出口に対応しているネットワークサイトにあります。 
    - ユーザーのエンドポイントは、位置情報に基づくルーティング用に有効になっていないゲートウェイ経由での位置情報に基づくルーティングと呼び出しの着信に対応していないネットワークサイトに配置されています。

    発信通話は、他のシナリオでは許可されていません。

- 着信 PSTN 通話を受信するには、ユーザーの応答エンドポイントが、場所に基づくルーティング用に有効になっているゲートウェイを経由して ingresses の同じネットワークサイトに配置されている必要があります。 ユーザーがローミングしている場合など、その他のシナリオでは、通話は許可されず、ユーザーの着信の転送設定 (通常はボイスメール) にルーティングされます。
- PSTN 通話を別の Teams ユーザーに転送するには、移行先ユーザーのエンドポイントが、転送を開始したユーザーと同じネットワークサイトに存在している必要があります。 他のシナリオでは、転送は許可されません。 
- 別の Teams ユーザーを PSTN に転送するには、最初の発信者と同じネットワークサイトにある位置情報に基づくルーティングが有効なゲートウェイを経由して、通話を転送する必要があります。 他のシナリオでは、転送は許可されません。

位置情報に基づくルーティングでは、Skype for Business Server で使用されるものと同じネットワークの領域、サイト、サブネットの定義が使用されます。 有料サービスがある場所に制限されている場合、管理者は各 IP サブネットと各 PSTN ゲートウェイをその場所のネットワークサイトに関連付けます。 ユーザーの位置情報は、ユーザーのチームエンドポイントが PSTN 通話の時点で接続されている IP サブネットによって決定されます。 ユーザーは、複数のサイトにある複数の Teams クライアントを持つことができます。この場合、その場合、場所に基づくルーティングでは、各クライアントのルーティングがエンドポイントの位置に応じて個別に適用されます。 

この記事で使用されているネットワーク用語の一部について理解するには、「 [Teams のクラウド音声機能のネットワーク設定](cloud-voice-network-settings.md)」を参照してください。

## <a name="apply-location-based-routing"></a>場所に基づくルーティングを適用する

ユーザー、ネットワークサイト、PSTN ゲートウェイには、場所に基づくルーティングを適用する必要があります。  

### <a name="apply-location-based-routing-at-the-user-location"></a>ユーザーの場所で位置情報に基づくルーティングを適用する

前に説明したように、位置情報に基づくルーティングは、直接ルーティング用に設定されているユーザーにのみ適用されます。 位置情報に基づくルーティングは、通話プランを設定するユーザーには適用されません。 場所に基づくルーティングを有効にするには、ユーザーが PSTN 通話の発信および受信を許可する条件を制御する [有料のバイパス制限] の下にある場合は、そのユーザーを有効にする必要があります。 位置情報に基づくルーティングが有効になっているユーザーが位置情報に基づくルーティングが有効になっているサイトに配置されている場合、ユーザーは、サイトに接続された位置情報に基づくルーティングが有効なゲートウェイを使用して通話を発信する必要があります。 

位置情報に基づくルーティングでは、ユーザーのチームエンドポイントの IP アドレスに基づいてユーザーの現在の位置を確認し、それに応じてルールを適用します。 位置情報に基づくルーティングが有効になっているユーザーの場所は、次の方法で分類できます。 
- **ユーザーは、そのユーザーが割り当てられている PSTN ゲートウェイに関連付けられている、同じ場所に基づいたルーティングを有効にしたサイトにあります。**<br>このシナリオでは、ユーザーは、位置情報に基づくルーティングが有効になっている既知のネットワークサイトに配置され、同じネットワークサイト内の PSTN ゲートウェイでは、ユーザーの直接内向きダイヤル (DID) 番号は終了します。 たとえば、ユーザーが office を使用している場合です。 
- **ユーザーは、そのユーザーが割り当てられている PSTN ゲートウェイに関連付けられていない別の場所に基づくルーティングが有効になっているサイトに存在します。**<br>このシナリオでは、ユーザーが位置情報に基づくルーティングが有効になっている既知のネットワークサイトにあり、そのサイトが、ユーザーの電話番号が割り当てられている PSTN ゲートウェイに関連付けられていません。 たとえば、ユーザーが別の office に移動します。  
- **ユーザーは、場所に基づくルーティングが有効になっていない内部サイトに配置されています。** <br>このシナリオでは、ユーザーは、位置情報に基づくルーティングが有効になっていない既知の内部ネットワークサイトにあります。 
- **ユーザーが不明なサイトに置かれている。** 
    - ユーザーは、ネットワークサイトとして定義されていない内部ネットワーク内に存在します。 
    - ユーザーは内部ネットワークの外部にあります。 たとえば、ユーザーは自宅または喫茶店でインターネットに接続されています。 

### <a name="apply-location-based-routing-at-the-network-site"></a>ネットワークサイトで位置情報に基づくルーティングを適用する 
位置情報に基づくルーティングのためにネットワークサイトを有効にする必要があります。ローミング時に、位置ベースのルーティングが有効なユーザーをルーティングするゲートウェイを決定するのに役立ちます。 位置情報に基づくルーティングが有効になっているユーザーが、位置情報に基づくルーティングが有効になっているサイトにローミングしている場合、そのサイトの位置情報に基づくルーティングで有効になっている PSTN ゲートウェイのみが、発信通話に使用できます。 位置情報に基づくルーティングが有効になっているユーザーが、位置情報に基づくルーティングに対応していないサイトにローミングしている場合、場所に基づくルーティングが有効になっていないゲートウェイは、発信時に使うことができます。  

### <a name="apply-location-based-routing-at-the-pstn-gateway"></a>PSTN ゲートウェイで位置情報に基づくルーティングを適用する 

ゲートウェイは、ユーザーが PSTN 通話を発信または受信したときに、位置情報に基づくルーティングが有効になっているユーザーがどこにあるかを判断するために、サイトに関連付けられています。 位置情報に基づくルーティングのゲートウェイは有効にしておく必要があります。これは、有料のバイパス制限の対象になっていて、位置情報に基づくルーティングが有効になっていないユーザーが使うことはできません。 同じゲートウェイを複数のサイトに関連付けることができます。また、サイトに応じて、位置情報に基づくルーティング用に有効にするように構成することもできます。

### <a name="teams-user-transfers-or-forwards-call-to-pstn-endpoint"></a>Teams ユーザーが PSTN エンドポイントに通話を転送または転送する

#### <a name="user-not-enabled-for-location-based-routing"></a>位置情報に基づくルーティングでユーザーが有効にされていない

- PSTN 通話を転送して別の PSTN 番号に転送することができます。 
- 着信 VOIP 通話を PSTN に転送して転送するには、発信者の有料バイパスの制限に従う必要があります。 
    - 発信者が位置情報に基づくルーティングを有効にしていない場合は、位置情報に基づくルーティングが有効になっていない PSTN ゲートウェイに転送できます。
    - 発信者が位置情報に基づくルーティングを有効にしている場合、同一のネットワークサイトにある位置情報に基づくルーティングが有効なゲートウェイにのみ転送されます。

- 着信 VOIP 通話を PSTN に転送して転送するには、発信者と発信者の両方の有料バイパス制限を承諾する必要があります。 
    - 発信者が位置情報に基づくルーティングを有効にしていない場合は、位置情報に基づくルーティングが有効になっていない PSTN ゲートウェイに転送できます。

### <a name="simultaneous-ringing"></a>同時呼び出し

#### <a name="inbound-calls-through-voice-app-auto-attendant-or-call-queue"></a>音声アプリ経由での着信通話 (自動応答または通話キュー)

位置に基づくルーティングが有効なゲートウェイからの着信 PSTN 通話は、自動応答または通話キューに接続できます。 位置情報に基づくルーティングが有効になっているユーザーは、受信した PSTN 通話の元のサイトに存在している場合にのみ、着信通話を受信できます。 
 
着信の転送と同時呼び出しをユーザーに対して行うことができます。また、ボイスアプリの転送にも PSTN を使用できます。 ターゲットへの通話の完了には、前に示したのと同じ場所に基づくルーティングルールが適用されます。  
 
ボイスメールへの転送も許可されています。  

### <a name="delegation"></a>デリゲーション

チームユーザーは、代理として通話を発信および受信できる代理人を選ぶことができます。 Teams の委任機能は、次のように位置情報に基づくルーティングによって影響を受けます。 
- 委任者に基づくルーティングが有効になっているデリゲートからの発信通話については、同じ規則が適用されます。 通話ルーティングは、代理人の呼び出し承認ポリシー、ボイスルーティングポリシー、および場所に基づいています。 詳細については、「 [Teams ユーザーが PSTN への発信通話を配置](#teams-user-places-an-outbound-call-to-the-pstn)する」を参照してください。 
- 委任者への着信 PSTN 通話の場合、通話転送に適用される位置情報に基づくルーティングルールと、同時に他のユーザーに転送されるルーティングルールは代理人にも適用されます。 詳細については、「 [teams ユーザーが別の teams ユーザーに電話を転送または転送する](#teams-user-transfers-or-forwards-call-to-another-teams-user)」を参照するか、「 [TEAMS ユーザーが PSTN エンドポイントへの通話を転送または転送する](#teams-user-transfers-or-forwards-call-to-pstn-endpoint) [」を参照](#simultaneous-ringing)してください。 代理人が PSTN エンドポイントを同時呼び出しターゲットとして設定すると、そのデリゲートのボイスルーティングポリシーを使って、PSTN への通話がルーティングされます。 
- 委任の場合、委任者と関連付けられた代理人を同じネットワークサイトに配置することをお勧めします。 

## <a name="other-planning-considerations"></a>計画に関するその他の考慮事項

### <a name="changes-from-an-on-premises-location-based-routing-deployment"></a>オンプレミスの位置情報に基づくルーティングの展開からの変更

ネットワークサイトのボイスルーティングポリシーは使用されなくなりました。 代わりに、ユーザのボイスルーティングポリシーを使用します。 これは、ユーザーに他のサイトへのローミングを許可するために、ボイスルーティングポリシーにはローミングされたサイトのゲートウェイを含める必要があることを意味します。 

### <a name="technical-considerations-for-location-based-routing"></a>場所に基づくルーティングに関する技術的考慮事項

IPv4 と IPv6 のサブネットはサポートされていますが、一致するかどうかを確認するときには IPv6 が優先されます。

### <a name="client-support-for-location-based-routing"></a>場所に基づくルーティングのクライアントのサポート

以下のチームクライアントがサポートされています。
- Teams デスクトップクライアント (Windows と Mac)
- Teams モバイルクライアント (iOS と Android)
- Teams の IP 電話機

Teams web クライアントと Skype for Business クライアントはサポートされていません。

### <a name="capabilities-not-supported-by-location-based-routing"></a>場所に基づくルーティングでサポートされていない機能

位置情報に基づくルーティングは、次の種類の操作には適用されません。 次のシナリオでは、チームのエンドポイントが PSTN エンドポイントを操作するときに位置情報に基づくルーティングが適用されません。 
- 通話保留による PSTN 通話の保留または再開 
- オンプレミスの Skype for Business ユーザーまたは Skype for Business Online ユーザーが Teams ユーザーを呼び出す  

### <a name="location-based-routing-for-conferencing"></a>会議用の位置情報に基づくルーティング

PSTN 通話の位置情報に基づくルーティングが有効になっているユーザーは、他のユーザーや PSTN 番号との会議を開始することはできません。 自動応答または通話キューへの接続は許可されています。 ユーザーが会議ライセンスを持っている場合は、ユーザーは関連ユーザーとの会議を開始して、電話会議を開始するために会議ブリッジ経由で PSTN に通話を発信する必要があります。  

### <a name="media-bypass-requirement-for-location-based-routing"></a>場所に基づくルーティングのメディアバイパスの要件

インドで位置情報に基づくルーティングを展開する場合は、メディアのバイパスも構成する必要があります。 詳細については、「ダイレクト[ルーティングによるメディアのバイパスの計画](direct-routing-plan-media-bypass.md)」および「[直接ルーティング用のローカルメディアの最適化](direct-routing-media-optimization.md)」を参照してください。
