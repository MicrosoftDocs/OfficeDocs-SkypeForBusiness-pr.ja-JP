# <a name="define-my-success"></a>成功を定義する

この記事では、電話会議、通話プランを使用した 電話システム、または組織の 電話システム ダイレクト ルーティングの展開の成功を定義するための要件の概要を説明します。 成功の外観を適切に定義することで、デプロイを進める際に結果を測定し、達成した結果が望みである結果を確認できます。

<!--ENDOFSECTION-->

**電話** 会議は、従来の固定電話、プライベート ブランチ交換 (PBX)、または携帯電話を使用してダイヤルインすることで、会議参加者が公衆交換電話網 (PSTN) 経由で参加できるようにすることで、任意の会議 (臨時またはスケジュール済み) への追加のエントリ ポイントを組織に提供します。 これは、開催者または参加者がコンピューターの前にいなくなったり、データ接続が利用できない場合や、音声通信をサポートできない場合 (モバイル データの範囲が不十分なリモートエリアの場合や、帯域幅が制限された無料のパブリック Wi-Fi サービスに接続されている場合など)、または会議参加者が、すぐにアクセスできるテレフォニー エンドポイントを使用して会議にダイヤルインすることを好む場合に便利です。

**電話システム通話プラン ("通話プラン")** を使用すると、ユーザーがコンピューターやモバイル デバイスからビジネス関連の電話を発信することで、組織の職場を最新化できます。 職場の最新化は、アクティビティベースの作業実装、主要なオフィスの移動、オフィスの調整の更新、従来の PBX ソリューションの廃止、PSTN サービス プロバイダー契約の締結など、任意の数のシナリオの一部になります。 通話プランを使用すると、Microsoft は PSTN への接続を容易にします。

**電話システム ルーティング ("** ダイレクト ルーティング") は、PSTN 接続が Microsoft ではなくサード パーティ プロバイダーによって促進される点を除き、上記の通話プランと同じ利点を組織に提供します。 これにより、通話プランを利用できない国や、既存の PSTN サービス プロバイダー契約を維持する必要があるデプロイ、または特定のオンプレミス システムとの相互運用性が必要なデプロイでの展開が可能になります。 ダイレクト ルーティングを考慮するもう 1 つのシナリオは、テレフォニー システムの相互運用性です。 ユーザーが [通話] に切り替Teams、一部のユーザーは従来の PBX に残っている可能性があります。 ダイレクト ルーティングでは、両方の使用例を共存させることができます。 レガシ システム上のユーザーとユーザー間の通話トラフィックTeams組織内に残っています。

<!--ENDOFSECTION-->

## <a name="define-business-use-cases-for-audio-conferencing-calling-plans-or-direct-routing"></a>電話会議、通話プラン、または直接ルーティングのビジネス の使用例を定義する

ビジネス の使用事例は、期待されるビジネス成果と測定可能なビジネス成果を定義して文書化することを意図し、次の内容を含む。

-   既存のビジネス プロセスに関する課題

-   課題の克服に向けたテクノロジーの有効性。

-   課題を克服したときに予期される測定可能なビジネス成果。

## <a name="define-okrs-ksis-and-risks"></a>OKRs、KSIs、およびリスクを定義する

プロジェクト関係者が集まったら、ビジネス の使用事例、組織の範囲、プロジェクトのタイムラインを目標と主要な結果 (OKRs) に変換し、プロジェクトの成功の尺度を主要な成功指標 (KSIs) の一覧として定義できます。

プロジェクト関係者による OK と KSIs の定義への完全な参加は、所有者の意識を持ち、これらの成功の尺度を組織のビジネス要件に合わせするために不可欠です。

OKRs には、プロジェクトの開始時に設定した目標が含まれているので、四半期ごとに測定可能な主要な結果を定義します。 主要な結果を毎月確認してプロジェクト全体の状態を追跡し、進捗状況に基づいて、必要に応じて四半期ごとに計画を調整します。

KSIs は、主要な結果の品質と成功を測定し、良い結果や悪い結果を詳細に示して、OKRs のバイナリの性質 (達成または達成されない) を補完します。

KSIs を定義する場合は、"特定の、測定可能、割り当て可能、現実的、時間関連" (SMART) 条件を使用することをお勧めします。

-   具体的: 特定の領域を改善対象とする

-   測定可能: 進捗状況を定量化するか、少なくともインジケーターを提案する

-   割り当て可能: 実行するユーザーを指定します。

-   現実的: 使用可能なリソースを指定して、どのような結果を現実的に達成できるのかを示す

-   時間関連: 結果を達成できる時期を指定します。

## <a name="establish-a-steering-committee"></a>運営委員会を設置する

運営委員会は、主要な利害関係者とプロジェクト リーダーの管理グループであり、定義されたビジネス成果に向けてプロジェクトまたはプログラムをガイドするために一緒に開催されています。 運営委員会は、プロジェクトの配信方法に直接責任を負うのではなく、プロジェクトがビジネスに提供する内容に直接責任を負います。

すべてのプロジェクトには、合意されたビジョンと計画が必要です。 プロジェクトから望む結果を得るには、ビジョンを明確に定義し、監視および維持する必要があります。 これは運営委員会の責任になります。意思決定の推進、アドバイス、戦略的監督の提供、プロジェクトのイニシアティブに対する組織の支持者としての役割、および必要に応じてブロックの削除を行います。

組織は、運営委員会の編成に大きな検討を加える必要があります。 委員会は、プロジェクトが組織全体の変化を推進するために定義したビジネス目標を確実に達成し、定期的に会議してプロジェクトの現在のパルスについて話し合い、途中で発生した障害のブロックを解除する必要があります。

委員会は、いくつかの重要な目標を含めるという取り組みについて定義する必要があります。

-   プロジェクト チームとエグゼクティブ スポンサーまたはエグゼクティブ リーダーシップの間で、強力な調整を行います。

-   プロジェクトの状態に関する洞察を、エグゼクティブ スポンサーまたはエグゼクティブ のリーダーシップに提供します。

-   プロジェクト計画、目標主要業績 (OKRs)、その他のプロジェクト 活動を調整することで、エグゼクティブ スポンサーまたはエグゼクティブ リーダーシップ チームがプロジェクトに対する方向性と入力を提供し、ビジネス目標の包括的な目標に合わせることができます。

運営委員会は、組織のリーダーシップとプロジェクト チームの間の調整を確保するために、プロジェクトの有効期間を通して定期的に会議を行います。 この重要な会議では、プロジェクトの方向性がリーダーシップの完全なサポートを受け、成功を後押しするために、リーダーから提供されたフィードバックをプロジェクトに組み込む必要があります。 委員会は、これらの会議を使用して、プロジェクトの状態に関する洞察を得て、次の処理を行います。

-   ビジネス ケースに合ったビジネス成果に同意し、プロジェクトがこれらの結果の配信に向かっている必要があります。

-   ビジネス ケースの正確性とコンプライアンスを確認し、プロジェクトを承認します。

-   定義された結果に影響を与える可能性があるビジネス ケースに加えた変更を確認し、確認します。

-   プロジェクト成果物の事前設定に関して戦略的な決定を行い、中間成果物を承認します。

-   委員会から追加の影響が必要なギャップ、リスク、問題を特定、管理、軽減します。

-   関係者のビジネス ユニット間の競合のエスカレーション、優先順位付け、解決が必要な問題について、エグゼクティブ スポンサーまたはエグゼクティブ リーダーシップ チームからサポートを集める。 

-   必要に応じ、エグゼクティブ リーダーシップ、変更勧告委員会、または他のビジネスおよび IT 関係者に正式なフィードバックと推奨事項を提供します。
