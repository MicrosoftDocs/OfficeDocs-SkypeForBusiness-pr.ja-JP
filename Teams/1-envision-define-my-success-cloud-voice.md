# <a name="define-my-success"></a>自分の成功を定義します。

この資料では、組織の音声会議、電話システムを呼び出すことを計画、または電話システムの直接のルーティングの展開の成功を定義するための要件の概要を示します。 正しく成功次のように定義すると、配置を使用中に、結果を測定し、結果を達成するがしたいと思ったものであることを確認できます。

<!--ENDOFSECTION-->

**オーディオ会議**では、従来の固定、秘密を使用してダイヤルして公衆交換電話網 (PSTN) 経由で参加する参加者を許可することで (アドホックまたはスケジュールされた) すべての会議への追加のエントリ ポイントを持つ組織が用意されています分岐交換機 (PBX)、または携帯電話です。 これは、主催者や参加者は、コンピューターの前に、またはデータ接続は、利用できないか、音声通信をサポートするために非常に信頼性が低い場合に便利です: テクスチャのモバイル データのカバレッジを使用してリモート領域など、無料でパブリック Wi-fi に接続されているか限られた帯域幅、または会議の参加者の方が簡単にアクセス可能なテレフォニー ・ エンドポイントを使用して、会議にダイヤルインするときにサービスします。

**(「呼び出し計画」) 計画を呼び出すと、電話システム**は、組織に電話をかけたりビジネスに関連する、コンピューターやモバイル デバイスからのユーザーを有効にすると、職場を改革する方法を示します。 ワークプ レースの近代化は、さまざまなシナリオの一部など、作業に基づく作業用の実装、主要なオフィスを移動する従来の PBX ソリューション、PSTN サービス プロバイダーの契約の最後の撤去、オフィス フィット アウト更新。 計画を呼び出すには、マイクロソフトでは、PSTN への接続が容易になります。

**電話システム直接ルーティング (「直接ルーティング」)** では同様の利点が計画呼び出すには、上記組織が、マイクロソフトではなく、サード パーティ プロバイダーが PSTN への接続が容易にします。 展開プランを呼び出すことは、国、または既存の PSTN サービス プロバイダーの契約を維持する必要があるか、特定の設置型システムとの相互運用性が必要な展開で可能になります。 直接ルーティングで考慮すべき追加シナリオの 1 つは、テレフォニー システムの相互運用性です。 チームを呼び出すには、ユーザーを移行しています、中に一部のユーザーがレガシー Pbx に残っている場合します。 直接ルーティングを有効に両方のユース ケースを共存させること。 レガシー ・ システム上のユーザーとチームのユーザーの間での呼び出しのトラフィックは、組織内に収めます。

## <a name="define-okrs-ksis-and-risks"></a>OKRs、KSIs、およびリスクを定義します。

プロジェクトでは、アセンブリが利害関係者、ビジネス使用の場合、組織スコープ、およびプロジェクトのタイムラインを変換するには目標と主要な結果 (OKRs)、および重要な成功指標 (KSIs) のリストとしてプロジェクトの成功のメジャーを定義することができます。

OKRs および KSIs を定義する際にプロジェクト関係者から完全に参加は、所有感を感じるし、組織のビジネス要件に成功した場合のこれらのメジャーを配置するようにする必要があります。

OKRs には、プロジェクトの開始時に設定した目標が含まれているし、主要な測定可能な結果を四半期ごとに定義します。 、プロジェクト全体のステータスを追跡するために毎月主要な結果を確認して、進捗状況に基づいて-必要に応じて四半期ごとの計画を調整します。

> 以下は、オーディオ会議の実装に関連する OKRs の例を参照できます。

KSIs は、主要な結果の品質と成功を測定し、良いまたは悪い結果を詳しく説明した、OKRs を実現 (達成できない) のバイナリの性質を補完します。

KSIs を定義する場合は、特定、測定可能な割り当てることができる、現実的な時間に関連する (スマート) の条件を使用することお勧めします。

-   改善のための特定の領域を対象と特定。

-   測定可能な: を定量化し、少なくともを提案すること、進行状況のインジケーター

-   それを実行する担当者を割り当てることができる: 指定します。

-   現実的な: 状態がどのような結果が現実的に実現する、利用可能なリソースが指定されています。

-   結果を得るために指定時間に関連します。

## <a name="establish-a-steering-committee"></a>運営委員会を確立します。

運営委員会の主要な関係者がプロジェクト リーダーの管理グループでは一緒にプロジェクトまたはプログラムは、定義されたビジネス結果になります。 運営委員会が*どのように*プロジェクトが配信されるではなく*どのような*プロジェクトがビジネスに提供を直接担当ではありません。

すべてのプロジェクトには、合意済みのビジョンとチャーターが必要です。 プロジェクトから、必要な結果を実現するには、ビジョンを明確に定義する必要があります、および監視および管理する必要があります。 これは、運営委員会の責任になります: 意思、アドバイスのプロジェクトの計画、組織の代弁者として機能する、戦略的な監視を提供して -必要な場合に、ブロックを削除します。

組織は、運営委員会の形成に重要な思考を置く必要があります。 委員会では、プロジェクトが、プロジェクトの現在のパルスを説明するには、定期的に対応、組織全体にわたって変更を推進するために定義されているし、に沿って発生した障害物をすべてのブロックを解除するためのビジネス目標を達成することを確認する必要があります、方法です。

委員会は、いくつかの重要な目的を含むように、チャーターを定義する必要があります。

-   プロジェクト チームとスポンサー、または経営幹部のリーダーシップの間の厳密な配置を保持します。

-   エグゼクティブ スポンサーまたは経営幹部のリーダーシップ、プロジェクトの状態についての情報を提供します。

-   方向を提供し、プロジェクトへの入力し、ビジネス目標、プロジェクト計画、主要目的の結果 (OKRs)、およびその他のプロジェクトの活動を調整することで包括的なと揃うことを確認するには、エグゼクティブ スポンサーまたは経営幹部のリーダーシップ ・ チームを許可します。

運営委員会は、組織のリーダーシップとプロジェクト チームの連携を確保するためのプロジェクトの有効期間全体での定期的な間隔で満たしています。 この重要な会議では、プロジェクトの方向はリーダーシップの完全なサポートを持つし、ドライブの成功のためのプロジェクトのリーダーシップによって提供されるすべてのフィードバックを組み込むことを保証します。 委員会は、プロジェクトの進捗状況、および見通しを得るためにこれらの会議を使用します。

-   ビジネスの成果にビジネス ・ ケース、およびプロジェクトにこれらの結果の配信に影響を与えていることを確認することに合意します。

-   確認し、正確性とビジネス ・ ケースの遵守のためのプロジェクトを承認します。

-   確認し、任意に定義された結果に影響を与えるビジネス ・ ケースに加えられた変更を確認します。

-   プロジェクトの成果物の優先順位付けに関する戦略的意思決定を行い、暫定的な成果物を承認します。

-   識別、管理、および、ギャップ、リスク、およびその他の影響が、委員会から必要がある問題を軽減します。

-   エグゼクティブ スポンサーまたは優先順位付け、利害関係者のビジネス単位間で競合を解決して、エスカレーションが必要な問題の経営幹部のリーダーシップ ・ チームからのサポートを収集します。 

-   正式なフィードバックと経営幹部のリーダーシップ、変更諮問委員会では、またはその他のビジネスおよび IT 関係者の適切な推奨事項を提供します。
