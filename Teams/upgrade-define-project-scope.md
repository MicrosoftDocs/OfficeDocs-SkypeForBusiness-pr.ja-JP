# <a name="define-your-project-scope"></a>プロジェクトの対象範囲を定義する

![アップグレード体験図。Project定義ステージが強調表示されています。](media/upgrade-banner-project-definition.png "「プロジェクトの定義」段階が強調表示された、アップグレード行程の各段階")

この記事は、アップグレード体験のProject定義ステージの一部であり、特定した利害関係者からスポンサーシップの連立とプロジェクト チームを作成した後に完了するアクティビティが、プロジェクトの成功の鍵となります。 先に進む前に、次のアクティビティを完了していることを確認してください。

- [参加しているプロジェクト関係者](upgrade-enlist-stakeholders.md)

プロジェクトのビジョン、スコープ、目標、ガバナンスを定義する時間を取ることで、すべてのプロジェクト関係者が同じ最終結果に向かって調整され、取り組むことが確実になります。 これは、技術的準備チームとユーザー準備チームが個別に作業し、それぞれの部分をまとめるために取り組んでいることを考えると、特に重要です。 このセクションを完了したら、プロジェクト全体を参照して、目的の終了状態を達成できるように確実に進んでください。 次に示す目標を使用して、結果に対して測定し、必要に応じて軽減します。

## <a name="project-vision"></a>Projectビジョン

ビジョンは、「なぜこのプロジェクトを行うのか」という質問に答える"全体像" または最終的な最終状態です。 理想的なビジョンは、次の例に示すように、組織のビジネス ドライバーとユーザーの付加価値の観点に対処します。

- **組織のビジネス ドライバー**: Microsoft Teamsの標準化は、デジタル ワークプレースの変革に合わせて調整され、運用効率を高め、冗長なソリューションを排除し、500 万ドルを節約できます。
- **ユーザーの値の追加**: Microsoft Teams (1) は、プロジェクト ノート、Office ドキュメント、チーム メンバー、会話、会議の 1 つの場所を提供することで時間を節約します。(2) 会話にすばやくアクセスするための一元的な連絡先リストと常設チャット追跡を使用して通信を簡素化し、(3) ファイルを 1 か所に保存してアクセスすることで、失われたメール添付ファイルを見つけようとするフラストレーションを軽減します。

ビジョンの改善に役立つ次のディスカッション ポイントを検討してください。

- 現在のビジネス プロセスの説明

- 既存のビジネス プロセスに関する課題

- テクノロジがこれらの課題を克服するのにどのように役立つか

- 課題を克服したときに予期される測定可能なビジネス成果。

> [!TIP]
> プロジェクトビジョンをさらに絞り込むために、ユース ケースとペルソナを特定します。

## <a name="project-scope"></a>Projectスコープ

ビジョンは、さまざまなフェーズを通じて、時間の経過と共にのみ実現される可能性があります。 プロジェクト スコープは、 _現在_ のプロジェクトのフォーカスを定義し、プロジェクト チームが現在のタスクに集中し続け、長期的なビジョンを実現できるようにします。 たとえば、スコープでは、パイロットを実行したり、音声や会議などの特定のワークロードをデプロイしたり、時間の経過と共にアップグレードを計画するときにSkype for Businessと共にTeamsを有効にしたりすることができます。 プロジェクト スコープの一部として、次を評価する必要があります。

- 組織に最適な[さまざまな共存モード](./teams-and-skypeforbusiness-coexistence-and-interoperability.md)。
- Teamsに移行する前に、Skype for BusinessとTeamsが共存するための最良の方法です。
- 組織内の技術的およびユーザーの準備状況を検証するために [パイロット](./pilot-essentials.md) を実施する必要があるかどうか。

## <a name="project-goals"></a>Projectの目標

目標は目的の結果を定義し、プロジェクトの成功を測定できるようにします。 目標は _目標と主要な結果_ (OKR) として定義でき、プロジェクトの成功のメジャーは _主要な成功インジケーター_ (KSI) として定義できます。 OKR と KSI の定義にプロジェクト関係者から完全な参加を得て、所有権を感じ、これらの成功の尺度を定義されたプロジェクト タスクに合わせることが不可欠です。 目標には、技術的成功とユーザー重視の成功の組み合わせが含まれている必要があります。

- **OKR には** 、プロジェクトの開始時に設定した目標と、定義されたケイデンス (毎月や四半期など) に対して測定する主要な結果が含まれます。 主要な結果を確認することで、プロジェクト成果物がスケジュールに従っていることを確認したり、問題を特定して軽減してプロジェクトを軌道に戻したりできます。通常、OKR は "達成済み" または "未達成" に分類されます。
- **KSIs は** 、主要な結果の品質と成功を測定し、良い結果や悪い結果を詳細に説明することで、OKR のバイナリ特性を補完します。 KSI を定義する場合は、"特定、測定可能、割り当て可能、現実的、時間関連" (SMART) の条件を使用することをお勧めします。
  - 具体的: 改善のために特定の領域をターゲットにする
  - 測定可能: 進行状況の指標を定量化するか、少なくとも提案する
  - 割り当て可能: 誰が実行するかを指定する
  - 現実的: 使用可能なリソースを指定して、現実的に達成できる結果を示します
  - 時間関連: 結果を達成できるタイミングを指定する

## <a name="risks-and-mitigation"></a>リスクと軽減策

 どのプロジェクトでも、予期しないイベントやその他の要因が発生し、プロジェクトが軌道に入る可能性があります。プロジェクトが目標に向かって継続できるように、潜在的なリスクを事前に評価し、発生する可能性のある問題を克服するための軽減策を定義することが重要です。 _リスク レジスタ_ は、プロジェクト のリスクを追跡し、その可能性と潜在的な影響を追跡し、軽減策計画を取り込むための優れたツールです。 次の表に、リスク レジスタのサンプルを示します。

| リスク | 可能性 | 影響 | 全体 | 軽減計画 |
|---|---|---|---|-------|
| **ネットワークの品質** | 中 | 高 | 高 | ネットワーク計画演習を実行します。 |
| **低いユーザー導入** | 高 | 高 | 高 | パイロットフェーズとデプロイフェーズでユーザーとプロアクティブに作業する。目標を設定した認識とトレーニング キャンペーンを実装して、望みを生み出します。 |
| | | | | |

## <a name="timeline"></a>タイムライン

アップグレード体験のスコープを設定するときは、完了日に加えて、重要なマイルストーンのタイムラインを設定してください (たとえば、すべてのユーザーのSkype for Businessと共にTeamsを有効にするなど)。 定義されたタイムラインは、プロジェクト チームが一貫した終了状態に向かって進み、適切なワークバック スケジュールを通知するのに役立ち、プロジェクトを確実に追跡するのに役立ちます。あまり加速されていないタイムライン (タスクが見落とされる可能性がある) または遠すぎる (勢いが失われる可能性がある) タイムラインを検討してください。 理想的なタイムラインのアカウントは次のとおりです。

- **コンプライアンスとユーザー シナリオの要件に対する製品の準備**: [製品ロードマップ](https://aka.ms/O365Roadmap)を参照して、Teamsが組織の準備が整うタイミングを測定します。
- **アップグレード グループ: アップグレード グループ** でユーザーのTeamsを有効にするかアップグレードするかを決定します。これは、アップグレードの全体的な体験のタイムラインに影響を与える可能性があります。
- **変更の凍結、会計年度の終了、デプロイ ライフサイクルなどの組織の要因**: アップグレード タイムラインに影響を与える可能性がある内部プロセスについて説明し、説明します。
- **その他の変更が同時または同時に発生する** 場合: 変更をバンドルするか、間隔を空けて、ユーザー エクスペリエンスを向上させ、生産性への影響を最小限に抑えるようにすることを検討してください。
- **リソースの再調達**: プロジェクト関係者とリソースの割り当てを確認し、一緒に集めてきたプロジェクト チームが必要なすべてのタスクを完了するのに十分な帯域幅を確保します。

参照ポイントとして、アップグレードPro体験のアップグレード前、アップグレード後、アップグレード後のフェーズのサンプル タイムラインが提供されます[。](./upgrade-start-here.md)これは、組織の特定のニーズに合わせて必要に応じて調整することをお勧めします。

上記のアクティビティを完了したら、プロジェクトの強固な基盤が必要です。 [技術的な準備と](./upgrade-prepare-environment.md)[組織の準備](./upgrade-prepare-organization.md)計画アクティビティを続行します。

Skype for Business Online は 2021 年 7 月 31 日に廃止されました。 特典の実現を最大化し、組織がアップグレードを完了するための適切な時間を確保するために、今すぐMicrosoft Teamsへの道を開始することをお勧めします。
