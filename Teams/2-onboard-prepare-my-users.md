# <a name="prepare-my-users"></a>ユーザーを準備する

この記事では、クラウド ボイス サービスを使用できるようにユーザーを準備するための要件の概要を取り上げます。 ユーザーを適切に準備することで、組織に最大限の価値をもたらすことができます。

<!--ENDOFSECTION-->

## <a name="preparing-for-user-adoption"></a>ユーザー導入の準備

プロジェクト チームのセキュリティ確保と、プロジェクトの範囲や目標の定義が済んだら、次の手順として、組織とユーザーの準備を整えます。この手順は、技術的な準備と並行して完了するアクティビティです。 デプロイの価値を引き出すには、ユーザーが実際に使用して導入する必要があります。 ソリューションを有効にしただけでは、この目標達成を保証できません。 さらに、ユーザーによって用途も学習スタイルもさまざまですし、新しいテクノロジに順応する速さも人それぞれです。 幸い、変更の管理はそれほど複雑ではありませんが、集中して取り組む必要があります。 以下のガイド付きのディスカッションは、ユーザー ベースを把握して、ユーザー導入を容易にして促進するための適切なレベルの教育を準備するのに役立ちます。

アクティビティの所有者: 導入変更主任マネージャー、プロジェクト主任、エグゼクティブ スポンサー、マーケティング/トレーニング/サポート マネージャー

> [!NOTE] 
> 以下で提供される評価アクティビティは、組織のあらゆる変更イニシアチブに対応しています。 プロジェクトの範囲に基づいて質問に答えるだけです。 これらのアクティビティの説明で使用される "新しいソリューション" とは、オーディオ通話、電話会議、または Skype for Business から Teams へのアップグレードを指しています。

<!--ENDOFSECTION-->

## <a name="create-your-vision-to-value-statement"></a>ビジョンからバリューへのステートメントを作成する

プロジェクトのヴィジョンを設定することにより、主だった関係者は全体像を把握できます。 通常これは、&mdash;コスト削減、運用効率、生産性の向上など&mdash;のビジネス ドライバーに相当し、プロジェクトで既に設定した目標と調和したものになります。 ユーザーの立場からは、同じプロジェクトが、使い慣れたツールを失うこと、生産性の低下、経営陣の理解不足として映る可能性があります。 ユーザーの立場を理解するために時間を取り、プロジェクトのヴィジョンをユーザーが魅力を感じるバリュー ステートメントに作り変えます。 情報伝達やトレーニング活動にこのバリュー ステートメントを使用して、Teams を利用する利点をユーザーが理解できるように助けます。

## <a name="determine-organizational-change-readiness"></a>組織的な変更の準備状況を判断する

組織の準備状況は、作業方法の変更に対してユーザーがどの程度受容性があるか (またはないか) を判断するのに役立ちます。 変化に対するユーザーの反応の仕方を理解することは、組織が問題に積極的に取り組むように促し、ロールアウト計画を受容されやすいものへと調整し、変更推進をサポートできる支持者や仲介者になってくれるユーザーを特定するのに役立ちます。

> [!TIP]
> ディスカッションの指針として、過去のロールアウトを評価します。これは、ユーザーが変化にどう反応するかを予測するのに役立ち、予想される反応に積極的に対応する助けになります。

### <a name="what-percent-of-users-fall-into-each-bucket"></a>各区分に何パーセントのユーザーが入りますか?

| **ユーザーの種類**                                                             | **%** |
|------------------------------------------------------------------------------|-------|
| *早期導入者*: 実際に提供される前からソリューションを要求するユーザー。                  |       |
| *情報に通じたユーザー*: ソリューションの価値が実証されるとすぐに受け入れるユーザー。    |       |
| *ラガード*: 変更された後になってもソリューションを拒否するユーザー。                 |       |


> [!TIP]
> 早期導入者は、パイロット展開と支持者のイニシアチブの最有力候補です。 ラガードの割合が大きい場合は、これらのユーザーに価値メッセージを発信し、彼らが変更に取り組むのを助けることに注力します。

### <a name="what-percent-of-users-fall-into-each-competency"></a>各コンピテンシーに何パーセントのユーザーが入りますか?

| **学習者の種類**                                                        | **%** |
|----------------------------------------------------------------------------|-------|
| *自発的*: このタイプの学習者はリソースを探し、実践して学習します。                         |       |
| *チーム プレーヤー*: このタイプのユーザーはグループ トレーニングや対話型のトレーニングを好み、同僚と協力して行動します。 |       |
| *要支援*: このタイプのユーザーはマンツーマンのサポートを期待します。                   |       |

 
> [!TIP]
> 学習の仕方は人によってさまざまです。 ユーザーにとって最適な学習方法を理解することで、スピーディな学習に役立つしっかりしたトレーニング プログラムを作成できます。 自発型のユーザーが多い場合は、ビデオやユーザー ガイドのリンクを提供するだけで十分なこともあります。 特別な注意が必要なユーザーについては、1 対 1 の時間をいくらかスケジュールするか&mdash;、学習曲線の全体でこのようなユーザーを支援する支持者を登録します。

## <a name="assess-your-user-base"></a>ユーザー ベースを評価する

ユーザー レベルで分析すると、ユーザーのペルソナ、コーホート、ユース ケースを理解できます。 ユーザーが誰でどんな仕事をしているかを理解するための時間を取ることにより、変更を実装するときに、ユーザーが成功するのに必要なものを確実に提供できます。 これには、対象を限定したメッセージ、トレーニング、さらにはヘッドセットや Web カメラなどのデバイスも含まれます。 たとえば、役員は 1 対 1 のトレーニングを希望するかもしれませんが、出張の多いフィールド担当者には自分の都合の良い時間に視聴できるトレーニング ビデオの方が便利かもしれません。 また、ユーザー レベルの分析は、新しい機能を今すぐ使用するべき人と、待つのが最善の人を判別するのにも役立ちます。

<table>
<tr><td><img src="media/audio_conferencing_image7.png" alt="An icon depicting decision points"/> <br/>判断ポイント</td><td><ul><li>特定のユーザーやコーホートが、仕事を効果的に遂行する上でこのソリューションにかなり依存していますか?</li><li>この新しいソリューションを使用することで簡単になる特定のユース ケースが存在しますか?</li></ul></td></tr>
<tr><td><img src="media/audio_conferencing_image9.png" alt="An icon depicting the next steps"/><br/>次の手順</td><td><ul><li>ユーザーのペルソナをマッピングします。 まず、ユーザー ベースをペルソナやコーホートのグループ (役員、出張の多い人、コラボレーション ワーカー、リテールなど) に分類します。 管理しやすくするため、グループは 8 つ以下にします。 (会話の切り出し方については後述)。結果を基に、ユーザー準備キャンペーンをさらに計画し、後のアップグレードで恩恵を受ける可能性のあるユーザーの中からパイロット ユーザーを特定します。
</li><li>ユース ケースをマッピングします。 この新しいテクノロジから恩恵を受ける主要なユーザー グループを特定したら、ユーザー全般について検討します。 チーム間のユース ケースやシナリオを理解することにより、組織で新しいテクノロジが使用されている様子をイメージできます。 また、コミュニケーションとトレーニング プランにユース ケースの例を追加することで、ユーザーにとって身近なものにすることができます。</li></ul></td></tr>
</table>

ペルソナ グループごとに、プロジェクト チームと次の点を検討します。

-   このグループの主要な特徴 (標準的な 1 日の業務内容) は何ですか?

-   新しいソリューションは、このグループにとってどのようなメリットがありますか? 効果性を上げるために彼らが必要としているどんな特徴や機能がありますか?

-   彼らが必要とするデバイス (ヘッドセットやスピーカーなど) は何で、それらを既に持っていますか?

-   このグループに新しいソリューションを使いたいと思わせるのに必要な付加価値メッセージがあるとすれば、それはどのようなメッセージですか?

-   このグループが次のものを受けるのに最適な方法は何ですか?

    -   意識向上 (メール、ポスター、部署のマネージャーなど)

    -   トレーニング (オンライン トレーニング、ユーザー ガイド、マンツーマンなど)

    -   サポート (セルフサービス、ヘルプデスク、支持者)  

-   このグループは、パイロット導入プログラムや早期導入プログラムの候補として適していますか? 適している場合は、これらのユーザーからパイロット参加に好意的な反応を引き出すためのプランを作成します。 パイロット プログラムを成功させる方法の詳細については、「[ユーザー パイロットを実施する](pilot-essentials.md)」を参照してください。

次の図に、ペルソナ テーブルの例を示します。

![ペルソナの詳細を示したバリュー テーブル](media/prepare-my-users-image2.png "共通の役割、コミュニケーション、コラボレーション ソリューションとの依存関係が記載されたペルソナの一覧表。[役員] ペルソナが強調表示され、プロファイル (1 日の業務内容) の詳細、役員が必要とする機能、デバイス、トレーニングが表示されている。")

ユース ケースごとに、プロジェクト チームと次の点を検討します。

-   このユース ケース/シナリオの主な特徴は何ですか?

-   現在のソリューションの利点は何ですか?

-   現在のソリューションに課題があるとすれば、それは何ですか?

-   新しいソリューションでは、ユーザー エクスペリエンスはどのように最適化されますか?

ユース ケースの例については、「[生産性向上ライブラリ](https://fasttrack.microsoft.com/microsoft365/productivitylibrary)」を参照してください。

## <a name="optimize-your-readiness-plan"></a>準備計画を最適化する
ユーザーが誰で、彼らが新しいソリューションをどう使うことになるかを理解したら、準備計画を設計できます。 ユーザーの準備計画には、新しいソリューションのパイロット、オンボード、運用にあたって、どのようにユーザーとコミュニケーションを図り、訓練やサポートを提供するかが含まれます。 この記事の前の手順で確認したように、必要な準備は人によってさまざまです。 このプロセスの最初で決定した価値メッセージを組み込んだ、大勢を対象とした意識向上とトレーニング プランから始めます。 次に、メッセージ、トレーニング、サポートを、定義したペルソナやコーホートに合わせて必要に応じて調整します。 パーソナライズされたプランを使用することで、ユーザーは新しいソリューションにすぐに慣れるだけでなく、個人的な必要を理解してもらえていると感じるので、移行をスムーズに行うことができます。
