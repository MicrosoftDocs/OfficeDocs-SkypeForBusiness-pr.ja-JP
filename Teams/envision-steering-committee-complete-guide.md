# <a name="steering-committee"></a>運営委員会

## <a name="steering-committee-overview"></a>運営委員会の概要

運営委員会は、定義されたビジネス成果に向けてプロジェクトまたはプログラムを導くために一緒に取り組んだ主要な関係者とプロジェクト リーダーの管理グループです。 運営委員会は、プロジェクトの配信方法に直接責任を負うのではなく、プロジェクトがビジネスに提供する内容に対して責任を負います。

すべてのプロジェクトには、合意されたビジョンと計画が必要です。 プロジェクトの望ましい成果を実現するには、ビジョンを明確に定義する必要があります。監視と維持が必要です。 これは運営委員会の責任となります。意思決定の推進、アドバイス、戦略的監視の提供、プロジェクトのイニシアティブに対する組織の支持者としての役割、および必要に応じてブロックの削除を行います。

運営委員会の設置には、組織が大きな考え方を持つ必要があります。 委員会は、組織全体の変化を推進するために定義したビジネス目標をプロジェクトが達成し、定期的に会議を開催してプロジェクトの現在の状況について議論し、その間に発生した問題を取り除く必要があります。

委員会は、いくつかの主要な目的を含める取り組み条件を定義する必要があります。

-   プロジェクト チームとエグゼクティブ スポンサーまたはエグゼクティブ リーダーの間で、強い調整を行います。

-   プロジェクトの状態に関する洞察を、エグゼクティブ スポンサーまたはエグゼクティブ のリーダーに提供します。

-   プロジェクト計画、目標主要な結果 (OKRs)、その他のプロジェクトアクティビティを調整することで、エグゼクティブ スポンサーまたはエグゼクティブ リーダー チームがプロジェクトの方向性とインプットを提供し、ビジネス目標の過大化に合わせて調整することができます。

運営委員会は、プロジェクトの有効期間を通じて定期的に会議を行い、組織のリーダーとプロジェクト チームの間で調整を行います。 この重要な会議は、プロジェクトの方向性がリーダーの完全なサポートを受け、成功を後押しするために、リーダーから提供されたフィードバックをプロジェクトに取り込むのを保証します。 委員会は、これらの会議を使用してプロジェクトの状態に関する洞察を得る他、次の場合にも使用します。

-   ビジネス ケースに合ったビジネスの成果に同意し、プロジェクトがこれらの結果の配信に向かっているのを確認します。

-   プロジェクトの正確性とビジネス ケースへの準拠を確認して承認します。

-   定義された結果に影響する可能性があるビジネス ケースに加えた変更を確認して確認します。

-   プロジェクト成果物の評価に関して戦略的な決定を行い、中間成果物を承認します。

-   委員会から追加の影響力が必要なギャップ、リスク、問題を特定、管理、緩和します。

-   関係者のビジネス ユニット間の競合のエスカレーション、優先順位付け、解決が必要な問題について、エグゼクティブ スポンサーまたはエグゼクティブ リーダー チームからサポートを集める。 

-   必要に応じ、エグゼクティブ リーダー、変更勧告掲示板、その他のビジネスおよび IT 関係者に正式なフィードバックと推奨事項を提供します。

## <a name="identify-members-of-the-steering-committee"></a>運営委員会のメンバーを特定する

委員会は、参加者が影響力を持つのと同じだけ効果的です。 運営委員会は、ソリューションに貢献する主要なプロジェクト チームのリーダーおよびパートナー チームと共に、ビジネスと技術の両方の意思決定者からの表現を維持する必要があります。

運営委員会には、強力な指導力と明確に定義された役割と責任が必要です。 委員会の業績とそのメンバーの参加に対する現実的な期待を設定するには、割り当てられた役割と責任を文書化し、合意する必要があります。 組織全体に効果的な影響力を与え、多様な視点を確保するには、この委員会に応答性の高いエグゼクティブ スポンサーとビジネス リーダーが必要です。

> [!NOTE]
> 組織とプロジェクト チームの設計に基づいて役割と責任を調整します。 小規模な組織では、これらの役割が重複する可能性があります。

| エグゼクティブ スポンサー | プロジェクトを支持し、組織のビジョンに合った方向を確保します。 委員会が同意できない決定を下す最終的な権威を務める。 | | |
| ビジネス意思決定者 | プロジェクトを支持し、組織のビジョンに合った方向を確保する | | |

## <a name="schedule-steering-committee-meetings"></a>運営委員会の会議をスケジュールする 

すべての組織とすべてのプロジェクト チームが少し異なる方法で構成されているが、運営委員会の会議が単なる別のプロジェクトの立ち上がり会議になるはずはない。 運営委員会会議の目的を検討します。エグゼクティブスポンサーまたはエグゼクティブ の指導部からサポートと方向性を集め、プロジェクトを追跡する機会です。 全員がこの会議に出席し、プロジェクト チーム全体とエグゼクティブ スポンサーまたはエグゼクティブ の指導部との間に直接メッセージが送信され、メッセージが中継された場合にコミュニケーションミスが起きないので確認します。

委員会の会議をスケジュールする場合は、プロジェクトと組織の更新を考慮します。 エグゼクティブ スポンサーの好みと組織のペースに基づいて、週単位または週単位でスケジュールを設定します。

> [!TIP]
> 移動が速い環境の場合は、会議間で時間を過ごさすぎずに、頻繁に会議を行う場合に回避できる問題が発生する可能性があります。 しかし、ペースが遅い環境では、毎週会議を開催すると会議の疲れが生じ、運営委員会の効果が低下する可能性があります。 経験と判断を使用して適切な更新時間を決定し、スケジュールを変更できると常に注意してください。 会議のためだけの会議は行う必要はない。

### <a name="prepare-for-the-meeting"></a>会議の準備をする

全員の時間を利用するには、会議の準備が必要です。 コア チームは、運営委員会会議の資料の準備を開始できます。 会議の準備に役立つ PowerPoint プレゼンテーションを作成できます。

会議のプレゼンテーション デッキに含めるには、次の資産を完了する必要があります。

-   前の運営委員会のアクション アイテムのフォローアップ

-   運営委員会 PowerPoint、プロジェクトの高レベルステータス レポート

    -   月次/四半期別/年別の種類のレビューから、プロジェクトを見上げてみてください。

    -   プロジェクトの主要マイルストーンを特定する:

        -   マイルストーンの重要性と影響について話し合います (これらはリスク レジスタに取り込む必要があります)。

        -   準備に基づいて、特にビジネス上の意思決定者と、今後の移行に向けて移動/立ち入りを決定します。

    -   プロジェクトの実行日と影響の大きなアクティビティを呼び出します。

    -   委員会にとって重要と特定された OKRs に関する報告。使用、ユーザー の感情、品質、信頼性に関する指標を必ず含める必要があります。

-   運営委員会 PowerPoint、プロジェクトの週次報告書

    -   強調表示と低ライト、重要な問題、アクション アイテム、次の手順が含まれます。

    -   毎日の問題に重点を置き、全体的なプロジェクトの状態に関するより技術的な詳細を含む。

-   問題ログと [リスク 登録](1-envision-define-my-success-cloud-voice.md#define-okrs-ksis-and-risks)

    -   委員会が追跡している問題の状態を共有します。

        -   すべてのバグに関する更新プログラムを提供する準備をしてください。

    -   ビジネス成果の達成に影響を与える可能性がある新しい項目を特定します。

        -   プロジェクトへの影響を特定します。

        -   影響を受ける個人またはビジネス ユニットを特定します。

        -   次の手順と、委員会のメンバーに必要な支援を特定します。

-   品質の概要: 「Teams 用にネットワークを [準備する」のガイダンスを使用します](prepare-network.md)。

    -   これらの品質レポートの結果を詳細に説明し、特に示されている問題を修復するための推奨事項を含める準備をしてください。

    -   次の領域に焦点を当て、毎週品質レポートを調整します。

        -   組織の全体的な品質の傾向

        -   問題が発生している既知の問題サイトまたはサイトの状態。サイトが見出しの方向を示し、理想的には修復作業による肯定的な結果を報告する

        -   プロジェクトの進捗状況に基づく、最近有効にしたサイトの品質状態

次に、運営委員会会議の準備のために電話会議をスケジュールします。 この呼び出しには、コア プロジェクト チームの重要なチーム メンバーのみを含める必要があります。これには、委員会会議中に対処するアイテムを含むネットワーク、デスクトップ、操作、その他のチームからのリソースが含まれます。 主要なプロジェクト チーム メンバーは、運営委員会の呼び出し中にカバーする予定の資料に気を取らされる必要があります。

この準備の呼び出しは、運営委員会が会議を行う前に同期するための最適な方法です。 また、プロジェクト チーム間で開いている項目、意見の相違、不満を確実に明かす必要があります。 これは、オープンなコミュニケーションを促進し、会議中にだれもが驚かされるのを防ぐためです。

その週のレポートを主要なチームおよびパートナー関係者と確認し、懸念事項や質問に対処します。 チーム内でディスカッションに基づいてコンテンツを調整、追加、または削除したり、運営委員会から追加の説明を求める意見に同意したりします。 各セクションを発表するチーム メンバーを割り当て、フローと継続性を確保します。

### <a name="hold-the-meeting"></a>会議を開催する 

会議自体は、コア チームが行った準備作業の自然な継続である必要があります。 これらの会議の目的は、プロジェクトのビジョンを実現するために、コミュニケーションとコラボレーションの雰囲気を促進する点に注意してください。 会議が成功するには、次の提案を使用します。

1.  会議を対面でスケジュールします。 すべてのユーザーが座るのに十分な大きなテーブルがある会議室を特定します。 会議室にビデオ機能が搭載されている必要があります。 会議室に全員を入れるのが理想的ですが、必ずしも可能なとは限りないので、会議出席依頼には会議室とオンライン会議の情報が少なくとも音声で含まれていますが、理想的には音声、ビデオ、共有が含まれる必要があります。 リモートのすべての参加者が会議に参加するときにビデオを使うのを奨励します。

2.  議題を常に確認してください。しかし、対処が必要な気が散る可能性は間違いありません。 可能であれば (重要度に応じて)、これらの項目を "parking lot" セクションに追加し、後で対処します。 トピックを延期するには重要すぎる場合は、引き続き議論するために合意を得ますが、今後の会議に他の議題項目を移動する必要があります。 委員会の直ちに注意が必要な差し迫った項目に対処するために、時間 (5 から 10 分) を確保することに同意します。

3.  オープン サポート ケースについて話し合う準備をしてください。 この会議では、できるだけ高いレベル (チケットの数、チケットの種類、主要な開いているアイテム) のディスカッションを維持してください。 すべての問題に対処する準備をする必要があります。問題が軽微な場合でも、ビジネスへの影響とプロジェクトへの影響を理解する必要があります。

4.  メモ取りを委任する。 発表中や熱い議論の最中にメモを取るのは難しすぎます。 委員会で行われた決定の詳細が記載されている必要があります。 フォローアップ メールの一部としてこれらの決定とメモを送信する準備をします。

5.  ユーザーの作業時間とユーザーのフィードバックに関する肯定的なニュース (利用可能な場合) を常に宣伝してください。 ストーリーを伝え、販売します。 フィードバックが完全に否定的な場合は、フィードバックについて話し合い、負のユーザー エクスペリエンスの問題に対処するために何が行われるかを説明する準備をしてください。

    a.  自分のやっている作業を良くするために、個々のチーム メンバーにクドを与える機会を得る。 特に指導チームの前で、優れたハードワークがチーム メンバーによって達成されている場合に認識することが重要です。

6.  会議の終わりを常に保存して、次の手順に同意し、会議中に同意したアクション アイテムの所有者を特定します。

7.  作成したプレゼンテーションを含むフォローアップ メールを出席者に送信し、作成されたノートを追加します。 出席しなかったユーザーは必ず含める必要があります。

## <a name="next-steps"></a>次の手順

運営委員会の概要を確認した後は、次に行動する必要があります。プロジェクト チームは、運営委員会をプロジェクトのタイムラインに実装し、定期的な運営委員会の会議の更新を準備する作業を開始する必要があります。 理想的には、プロジェクトを開始する前に委員会を形成し、すべてのフェーズを通じてガイダンスを提供しますが、その場合は、プロジェクト チームを成功に導く委員会を形成します。

1.  運営委員会のメンバーを特定します。 次の質問をしてください。

    -   ビジネスとリーダーから適切な表現がありますか?

    -   特定されたメンバーは、他の単位で自分のビジネス ユニットや同僚に影響を与える可能性がありますか?

2.  運営委員会の各メンバーに送信するコミュニケーションを作成します。

    -   役割テーブルを含める。

    -   委員会の重要性を販売する。

3.  役割と責任について話し合い、委員会の条件を定義し、OKRs を検証または定義する最初の会議を設定します。 議題項目には、次の項目を含め、次の項目を含め、次

    -   会議の開催日について話し合います。

    -   この取り組みによって、重要なビジネス成果に向かっているのを検証します。

    -   戦略が OKRs と一緒に配置されているのを確認します。

4.  すべての会話で品質向上を開始する:

    -   品質チャンピオンを持つことの重要性を明確にする。

    -   Quality Champion の役割を引き受えるユーザーまたはチームを特定します。

運営委員会の成功は運によっては起こりません。 最適な目的で適切なメンバーを選択すると、低品質の結果が得られる可能性があります。 運営委員会の成功を推進するには、次の情報を確認する必要があります。

-   明確に定義された測定可能なビジネス成果を持つ、委員会が主張するプロジェクトの合意されたビジョンと範囲。

-   目的に合った会議を開催する、整理された議題。

-   オープンなコミュニケーションの雰囲気。

    -   オープンなコミュニケーションとコラボレーションにより、委員会が権限を与える。 すべての声が聞こえ、意見が尊重される環境を作ります。 独自のパースペクティブが最も大きな影響を与える可能性があります。

    -   オープン環境は、リーダーがメンバーの共同作業とコミュニケーションを促進する役割を行う場合に有効にできます。 いくつかのガイドラインを次に示します。

        -   すべての委員会メンバーは、組織に導入される変更を強く主張する必要があります。

        -   委員会の規模は、正確な画像を提供する必要がある人や、ビジネス ユニット全体で決定を下したり影響を与える人に限定します。 ブロックが表示される場合、組織の階層と各メンバーの直接の影響力を理解することは非常に重要です。

        -   プロジェクト マネージャーとチーム リーダーを委員会のディスカッションや検討に参加します。 特定の領域で必要になる可能性がある詳細なディスカッションを明確にするために、追加のコア チーム メンバーを招待する準備が整います。

ソリューションのライフサイクルを通じて、委員会の役割は変わります。 構想とオンボーディングがドライブ値フェーズに移行する場合、委員会にはいくつかの調整が必要になる場合があります。

-   会議の開催日を調整します。 組織の動的な動きを確認し、会議の開催頻度を低くする必要があるかどうかを判断します。 組織の変更が始まると、上レベルの通信の頻度が低下する可能性があります。 必要に応じて、会議を月次または四半期ごとに会議の開催日に変更する方法を調査する

-   会議の代表者を調整します。 ソリューションがドライブ値のフェーズに移動すると、コア プロジェクト チームが新しいプロジェクトに再割り当てされる可能性があります。 サポートと運用がソリューションの状態のリードになると報告しているチームを更新する必要があります。

-   運用用の OKRs に対する定義とレポート。 主要な結果と成功のインジケーターは、ソリューションのライフサイクルを通じて変化します。 望ましいビジネス成果は、引き続き追跡および報告する必要があります。 引き続き、ビジネスの成果に向けて推進する中で、音声ソリューションの品質と信頼性について報告します。 ユーザー の感情、ユーザーの導入、サポート指標など、追加のメトリックと主要な結果が、成功のトップ インジケーターになります。

-   大規模な運営委員会の傘を使用して、複数のソリューションのドライブ値のフェーズをカバーします。 運営委員会を効率的に使用する機会を探します。 委員会を使用して、より大規模なソリューション領域全体にガイダンスを提供します。
