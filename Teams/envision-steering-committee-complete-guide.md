# <a name="steering-committee"></a>運営委員会

## <a name="steering-committee-overview"></a>運営委員会の概要

運営委員会は、定義されたビジネス成果に向けてプロジェクトまたはプログラムを導くために集められた主要な利害関係者とプロジェクト リーダーの管理グループです。 運営委員会は、プロジェクトの配信 *方法* に直接責任を負うのではなく、プロジェクトがビジネスに提供するものを担当します。

すべてのプロジェクトには、合意されたビジョンと憲章が必要です。 ビジョンを明確に定義する必要があるプロジェクトの望ましい成果を提供するには、それを監視し、維持する必要があります。 これは運営委員会の責任となります。意思決定を推進し、助言し、戦略的な監督を提供し、プロジェクトのイニシアチブの組織の支持者として機能し、必要に応じてブロッカーを削除します。

組織は、運営委員会の形成に大きな思いを込める必要があります。 委員会は、組織全体の変革を推進するために定義したビジネス目標をプロジェクトが確実に達成し、定期的に会議を開いてプロジェクトの現在のパルスについて話し合い、移動中に発生した障害を取り除く手助けをする必要があります。

委員会は、いくつかの重要な目的を含めるために、その憲章を定義する必要があります。

-   プロジェクト チームとエグゼクティブ スポンサーまたはエグゼクティブ リーダーシップの間で、強固な連携を保ちます。

-   エグゼクティブ スポンサーまたはエグゼクティブ リーダーシップにプロジェクトの状態に関する分析情報を提供します。

-   エグゼクティブ スポンサーまたはエグゼクティブ リーダーシップ チームがプロジェクトに対する指示と入力を提供し、プロジェクト計画、目標キーの結果 (OKR)、およびその他のプロジェクト アクティビティを調整することで、ビジネス目標の包括的な内容に合わせて調整できるようにします。

運営委員会は、プロジェクトの有効期間を通じて定期的に会議を開催し、組織のリーダーシップとプロジェクト チームの間の連携を確保します。 この重要な会議は、プロジェクトの方向性がリーダーシップの完全なサポートを持ち、リーダーシップから提供されたフィードバックをプロジェクトに組み込んで成功を促進することを保証します。 委員会は、これらの会議を使用して、プロジェクトの状態に関する分析情報を取得し、次のことも行います。

-   ビジネス ケースに合ったビジネス成果に同意し、プロジェクトがこれらの成果の提供に向けて推進していることを確認します。

-   ビジネス ケースの正確性とコンプライアンスについて、プロジェクトを確認して承認します。

-   定義された結果に影響を与える可能性があるビジネス ケースに加えられた変更を確認し、確認します。

-   プロジェクト成果物の優先順位付けに関する戦略的な決定を行い、中間成果物を承認します。

-   委員会から追加の影響が必要なギャップ、リスク、および問題を特定、管理、軽減します。

-   エスカレーション、利害関係者の部署間の競合の優先順位付け、解決が必要な問題については、エグゼクティブ スポンサーまたはエグゼクティブ リーダーシップ チームからサポートを受けます。 

-   必要に応じて、エグゼクティブ リーダーシップ、変更アドバイザリ ボード、またはその他のビジネスおよび IT 関係者に正式なフィードバックと推奨事項を提供します。

## <a name="identify-members-of-the-steering-committee"></a>運営委員会のメンバーを特定する

委員会は、参加者が影響力を持っているのと同じくらい効果的です。 運営委員会は、ビジネスと技術の意思決定者の両方から、ソリューションに貢献する主要なプロジェクト チームのリーダーシップとパートナー チームの両方からの表現を維持する必要があります。

すべての運営委員会には、強力なリーダーシップと明確に定義された役割と責任が必要です。 委員会のパフォーマンスとメンバーの参加に対する現実的な期待を設定するには、割り当てられた役割と責任を文書化し、同意する必要があります。 組織全体に効果的な影響力を与え、多様な視点を確保するには、この委員会に対する応答性の高いエグゼクティブ スポンサーとビジネス リーダーシップが絶対に必要です。

> [!NOTE]
> 組織とプロジェクト チームの設計に基づいて、ロールと責任を調整します。 小規模な組織では、これらのロールが重複する可能性があります。

| ビジネスデシジョン 作成者 | プロジェクトを支持し、その方向が組織のビジョンと一致していることを確認する | | |

## <a name="schedule-steering-committee-meetings"></a>運営委員会の会議をスケジュールする 

すべての組織とすべてのプロジェクト チームは少し異なる方法で構成されていますが、運営委員会の会議が別のプロジェクトの立ち上げ会議になることはありません。 運営委員会の会議の目的を検討してください。これは、エグゼクティブ スポンサーまたはエグゼクティブ リーダーシップからのサポートと指示を集めることで、プロジェクトを軌道に乗わせ続ける機会です。 全員がこの会議に出席し、プロジェクト チーム全体とエグゼクティブ スポンサーまたはエグゼクティブ リーダーシップの間に直接メッセージが送信されるようにし、メッセージが中継されたときに誤ったコミュニケーションが行われないようにします。

委員会会議をスケジュールする場合は、プロジェクトと組織のケイデンスを考慮に入れます。 エグゼクティブ スポンサーの好みと組織のペースに基づいて、毎週または 1 週間おきにスケジュールを設定してみてください。

> [!TIP]
> 動きの速い環境の場合は、会議の間に時間を費やしすぎないようにしてください。これは、より頻繁に会議を行うことで回避できた問題につながる可能性があります。 ただし、ペースの遅い環境では、毎週会議を開催すると、会議の疲れにつながる可能性があり、運営委員会の有効性が低下する可能性があります。 経験と判断を使用して適切なケイデンスを判断し、スケジュールを変更できることを常に忘れないでください。 会議のためだけに会わないでください。

### <a name="prepare-for-the-meeting"></a>会議の準備

全員の時間を最大限に活用するには、会議の準備が必要です。 コア チームは、運営委員会会議の資料の準備を開始できます。 会議の準備に役立つPowerPointプレゼンテーションを作成できます。

次の資産を完了し、会議のプレゼンテーション デッキに含める必要があります。

-   以前の運営委員会のアクション項目に関するフォローアップ

-   運営委員会PowerPoint、プロジェクトの高レベルの状態レポート

    -   月次/四半期/年単位のレビューからプロジェクト全体を確認します。

    -   主要なプロジェクトマイルストーンを特定する:

        -   マイルストーンの重要性と影響について説明します (これらはリスク レジスタに取り込む必要があります)。

        -   準備に基づいて、特にビジネス意思決定者との間で、今後の移行に向けて go/no-go を決定します。

    -   プロジェクトの実行日と影響の大きいアクティビティを呼び出します。

    -   委員会にとって重要であると特定された OKR に関するレポート。使用状況、ユーザーセンチメント、品質、信頼性のメトリックを含める必要があります。

-   運営委員会のPowerPoint、プロジェクトの毎週の状態レポート

    -   強調表示と低光、主要な問題、アクション 項目、および次の手順が含まれます。

    -   日常的な問題に焦点を当て、プロジェクトの全体的な状態に焦点を当てた技術的な詳細が含まれています。

-   問題のログと [リスクの登録](1-envision-define-my-success-cloud-voice.md#define-okrs-ksis-and-risks)

    -   委員会が追跡している問題の状態を共有します。

        -   すべてのバグに関する更新プログラムを提供する準備をしてください。

    -   ビジネス成果の達成に影響を与える可能性のある新しい項目を特定します。

        -   プロジェクトへの影響を特定します。

        -   影響を受ける個人または部署を特定します。

        -   次の手順と、委員会のメンバーからどのような支援が必要になるかを特定します。

-   品質の概要: [ネットワークをTeamsに準備するに関するページのガイダンスを使用します](prepare-network.md)。

    -   これらの品質レポートの結果を詳細に説明し、注意された問題を修復するための推奨事項を含める準備をしてください。

    -   品質レポートを毎週調整して、次の領域に焦点を当てます。

        -   組織の全体的な品質の傾向

        -   問題が発生している既知の問題サイトまたはサイトの状態。サイトが向かう方向を示し、修復作業の肯定的な結果を報告するのが理想的です

        -   プロジェクトの進行状況に基づく最近有効になったサイトの品質状態

次に、運営委員会会議の準備のために通話をスケジュールします。 この呼び出しには、コア プロジェクト チームから重要なチーム メンバーのみを含める必要があります。これには、ネットワーク、デスクトップ、運用、または委員会の会議中に対処する項目を持つ他のチームからのリソースが含まれる場合があります。 主要なプロジェクト チーム メンバーは、運営委員会の呼び出し中に取り上げる予定の資料を見落としてはなりません。

この準備の呼び出しは、運営委員会が会議を開く前に同期するのに最適な方法です。 また、プロジェクト チーム間で開いているアイテム、意見の相違、不満が明るみになるようにする時間でもあります。 これは、オープンなコミュニケーションを促進し、会議中に誰かが不快に驚くのを防ぐのに役立ちます。

その週のレポートをコア チームとパートナーの利害関係者と共に確認し、懸念事項や質問に対処します。 チーム内で作業して、ディスカッションに基づいてコンテンツを調整、追加、削除するか、運営委員会に追加の説明を求めることに同意します。 各セクションを発表するチーム メンバーを割り当てて、フローと継続性を確保します。

### <a name="hold-the-meeting"></a>会議を開催する 

会議自体は、コア チームが行った準備作業の自然な継続である必要があります。 これらの会議の目標は、プロジェクトのビジョンを達成するためのコミュニケーションとコラボレーションの雰囲気を促進することです。 会議の成功を確実にするために、次の提案を使用します。

1.  顔を合わせるために会議をスケジュールします。 全員が座れるほど大きいテーブルを備えた会議室を特定します。 会議室にビデオ機能があることを確認します。 全員を会議室に入れるのが理想的ですが、常に可能であるとは限らないので、会議出席依頼に少なくとも音声を含む会議室とオンライン会議情報が含まれていることを確認してください。ただし、理想的には、オーディオ、ビデオ、共有です。 会議に参加するときに、すべてのリモート参加者にビデオを使用するよう勧めます。

2.  課題に取り組もうとしますが、対処する必要がある気晴らしが発生することは間違いありません。 可能であれば (重要度に応じて) これらの項目を "駐車場" セクションに追加し、後で対処します。 トピックを延期することが重要すぎる場合は、引き続き話し合うために合意を得ますが、他のいくつかの議題項目を将来の会議に移動する必要があることに注意してください。 委員会の直ちに注意を払う必要がある差し迫った項目に対処するために、しばらく時間 (5 分から 10 分) を確保することに同意します。

3.  開いているサポート ケースについて話し合う準備をしてください。 この会議では、チケットの数、チケットの種類、主要なオープン アイテムなど、できるだけ高いレベルでディスカッションを続けてみてください。 問題が軽微な場合でも、ビジネスへの影響とプロジェクトへの影響を理解し、すべての問題に対処する準備が必要です。

4.  メモを受け取るユーザーを委任します。 プレゼンテーション中や激しい議論の最中にメモを取るのは難しすぎます。 委員会で行われた決定の詳細が文書化されていることを確認します。 これらの決定事項とメモをフォローアップメールの一部として送信する準備をしてください。

5.  ユーザーの作業の取り組みとユーザーからのフィードバックに関する肯定的なニュースを常に宣伝してみてください (利用可能な場合)。 ストーリーを伝えて販売する。 フィードバックが完全に否定的な場合は、それを議論し、ユーザー エクスペリエンスの負の問題に対処するために何が行われているかを説明する準備をしてください。

    a.  個々のチーム メンバーが適切な作業を行うために、その目的に応じて適切な作業を行う機会を得てください。 特にリーダーシップ チームの前で、チーム メンバーが優れたハード ワークを達成していることを認識することが重要です。

6.  会議の最後を保存して、次の手順に同意し、会議中に合意されたアクション アイテムの所有者を特定します。

7.  作成したプレゼンテーションを含む出席者のフォローアップメールを送信し、撮影したメモを追加します。 出席しなかったユーザーは必ず含めます。

## <a name="next-steps"></a>次のステップ

運営委員会の概要を確認したら、今度は行動する時間です。プロジェクト チームは、プロジェクトのタイムラインに運営委員会を実装し、定期的な運営委員会会議の開催を準備する作業を開始する必要があります。 理想的には、プロジェクトが開始される前に委員会が形成され、すべてのフェーズを通じてガイダンスが提供されますが、それができない場合は、プロジェクト チームを成功に導く委員会を形成します。

1.  運営委員会のメンバーを特定する。 次の質問をしてください。

    -   ビジネスとリーダーシップからの適切な表現はありますか?

    -   特定したメンバーは、他のユニットのビジネス ユニットまたはピアに影響を与えることはできますか?

2.  運営委員会の各メンバーに送信するコミュニケーションを作成します。

    -   ロール テーブルを含めます。

    -   委員会の重要性を売る。

3.  役割と責任について話し合い、委員会の憲章を定義し、OKR を検証または定義するための最初の会議を設定します。 議題項目には、次の項目を含めることもできます。

    -   会議のケイデンスについて話し合います。

    -   憲章と OKR が意味のあるビジネス成果に向けて推進されていることを検証します。

    -   戦略が OKR と一致していることを確認します。

4.  すべての会話で品質の向上を開始します。

    -   品質チャンピオンを持つことの重要性を明確に示します。

    -   品質チャンピオンの役割を担うユーザーまたはチームを特定します。

運営委員会の成功は運では発生しません。 最適な意図で適切なメンバーを選択すると、結果が悪くなる可能性があります。 運営委員会の成功を支援するには、次のことを確認する必要があります。

-   委員会が主張する明確に定義された測定可能なビジネス成果を持つプロジェクトの合意されたビジョンと範囲。

-   目的に合った会議を推進する適切に整理された議題。

-   オープン コミュニケーションの雰囲気。

    -   オープンなコミュニケーションとコラボレーションは、委員会に力を与えます。 すべての声が聞こえ、意見が尊重される環境を作成します。 独自のパースペクティブが最も大きな影響を与える可能性があります。

    -   オープン環境は、リーダーがメンバーのコラボレーションとコミュニケーションを促進するために機能する場合に有効にすることができます。 いくつかのガイドラインを次に示します。

        -   すべての委員会メンバーは、組織に導入される変更に対する強力なアドボケイトである必要があります。

        -   委員会の規模は、正確な画像を提供するために絶対に必要な人と、部署間で意思決定を行ったり、影響を与えたりできる人に限定してください。 ブロックが出現する場合、組織の階層と各メンバーの直接的な影響力を理解することは非常に重要です。

        -   プロジェクト マネージャーとチーム リーダーと連携して、委員会のディスカッションと検討を行います。 追加のコア チーム メンバーを招待して、特定の分野で必要になる可能性がある詳細なディスカッションを明確にする準備をしてください。

ソリューションのライフサイクルを通して、委員会の役割は変わります。 ドライブ値フェーズへの移行を想定してオンボードする場合は、委員会に対していくつかの調整が必要になる場合があります。

-   会議のケイデンスを調整します。 組織の動的な状況を確認し、会議の頻度を低くする必要があるかどうかを判断します。 組織の変更が減少し始めるにつれて、上位レベルの通信の頻度は減少する可能性があります。 必要に応じて、会議を月次または四半期ごとの会議の周期に変更することを調査する

-   会議の代表者を調整します。 ソリューションがドライブ値フェーズに移行すると、コア プロジェクト チームが新しいプロジェクトに再割り当てされる可能性があります。 サポートとして報告しているチームを更新する必要があり、運用がソリューションの状態のリードになります。

-   運用 OKR に対する定義とレポート。 主要な結果と成功インジケーターは、ソリューションのライフサイクルで変化します。 目的のビジネス成果は、引き続き追跡および報告する必要があります。 引き続き、ビジネス成果に向けて推進するコンテキストにおける音声ソリューションの品質と信頼性について報告します。 ユーザーセンチメント、ユーザー導入、サポート メトリックなど、その他のメトリックと主要な結果が、成功の主要な指標になります。

-   より大きな運営委員会の傘を使用して、複数のソリューションのドライブ価値フェーズをカバーします。 運営委員会をより効率的に使用する機会を探します。 委員会を使用して、より広い範囲のソリューションに関するガイダンスを提供します。
