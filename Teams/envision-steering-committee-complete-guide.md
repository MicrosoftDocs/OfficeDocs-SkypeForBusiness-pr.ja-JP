# <a name="steering-committee"></a>運営委員会

## <a name="steering-committee-overview"></a>運営委員会の概要

運営委員会は、主要な利害関係者とプロジェクト リーダーの管理グループであり、定義されたビジネス成果に向けてプロジェクトまたはプログラムをガイドするために一緒に開催されています。 運営委員会は、プロジェクトの配信方法に直接責任を負うのではなく、プロジェクトがビジネスに提供する内容に直接責任を負います。

すべてのプロジェクトには、合意されたビジョンと計画が必要です。 ビジョンを明確に定義する必要があるプロジェクトの望ましい結果を実現するには、監視と維持が必要です。 これは運営委員会の責任になります。意思決定の推進、アドバイス、戦略的監督の提供、プロジェクトのイニシアティブに対する組織の支持者としての役割、および必要に応じてブロックの削除を行います。

組織は、運営委員会の編成に大きな検討を加える必要があります。 委員会は、プロジェクトが組織全体の変化を推進するために定義したビジネス目標を確実に達成し、定期的に会議を行ってプロジェクトの現在のパルスについて話し合い、体験中に発生した障害を取り除く必要があります。

委員会は、いくつかの重要な目標を含めるという取り組みについて定義する必要があります。

-   プロジェクト チームとエグゼクティブ スポンサーまたはエグゼクティブ リーダーシップの間で、強力な調整を行います。

-   プロジェクトの状態に関する洞察を、エグゼクティブ スポンサーまたはエグゼクティブ のリーダーシップに提供します。

-   プロジェクト計画、目標主要業績 (OKRs)、その他のプロジェクト 活動を調整することで、エグゼクティブ スポンサーまたはエグゼクティブ リーダーシップ チームがプロジェクトに対する方向性と入力を提供し、ビジネス目標の包括的な目標に合わせることができます。

運営委員会は、組織のリーダーシップとプロジェクト チームの間の調整を確保するために、プロジェクトの有効期間を通して定期的に会議を行います。 この重要な会議では、プロジェクトの方向性がリーダーシップの完全なサポートを受け、成功を後押しするために、リーダーから提供されたフィードバックをプロジェクトに組み込む必要があります。 委員会は、これらの会議を使用してプロジェクトの状態を把握し、次の情報を得る。

-   ビジネス ケースに合ったビジネス成果に同意し、プロジェクトがこれらの結果の配信に向かっている必要があります。

-   ビジネス ケースの正確性とコンプライアンスを確認し、プロジェクトを承認します。

-   定義された結果に影響を与える可能性があるビジネス ケースに加えた変更を確認し、確認します。

-   プロジェクト成果物の事前設定に関して戦略的な決定を行い、中間成果物を承認します。

-   委員会から追加の影響が必要なギャップ、リスク、問題を特定、管理、軽減します。

-   関係者のビジネス ユニット間の競合のエスカレーション、優先順位付け、解決が必要な問題について、エグゼクティブ スポンサーまたはエグゼクティブ リーダーシップ チームからサポートを集める。 

-   必要に応じ、エグゼクティブ リーダーシップ、変更勧告委員会、または他のビジネスおよび IT 関係者に正式なフィードバックと推奨事項を提供します。

## <a name="identify-members-of-the-steering-committee"></a>運営委員会のメンバーを特定する

委員会は、参加者が影響力を持つのと同じだけ有効です。 運営委員会は、ビジネスと技術の両方の意思決定者からの表現と、ソリューションに貢献する主要なプロジェクト チームのリーダーシップとパートナー チームの両方の表現を維持する必要があります。

すべての運営委員会には、強力なリーダーシップと明確に定義された役割と責任が必要です。 委員会のパフォーマンスとそのメンバーの参加に現実的な期待を設定するには、割り当てられた役割と責任を文書化し、合意する必要があります。 組織全体に効果的な影響力を与え、多様な視点を確保するには、この委員会に対する応答性の高いエグゼクティブ スポンサーとビジネス リーダーが必要です。

> [!NOTE]
> 組織とプロジェクト チームの設計に基づいて役割と責任を調整します。 小規模な組織では、これらのロールが重複する可能性があります。

## <a name="schedule-steering-committee-meetings"></a>運営委員会の会議をスケジュールする 

すべての組織とすべてのプロジェクト チームは、少し異なる方法で構成されていますが、運営委員会の会議は、単なる別のプロジェクト の立ち上がり会議になる必要があります。 運営委員会会議の目的を検討します。これは、エグゼクティブ スポンサーまたはエグゼクティブのリーダーシップからのサポートと指示を集め、プロジェクトを追跡する機会です。 全員がこの会議に出席し、プロジェクト チーム全体とエグゼクティブ スポンサーまたはエグゼクティブ のリーダーシップの間に直接メッセージが送信され、メッセージがリレーされる際に誤ったコミュニケーションが行えなかねばならなさそう。

委員会の会議をスケジュールする場合は、プロジェクトと組織の更新を考慮してください。 エグゼクティブ スポンサーの好みと組織のペースに基づいて、毎週または 1 週間おきにスケジュールを設定してみてください。

> [!TIP]
> 移動の速い環境の場合は、会議間で時間を過ぎすぎずに、会議の頻度が高い場合に回避できる問題が発生する可能性があります。 しかし、ペースの遅い環境では、毎週会議を開催すると、会議の疲れを引き出し、運営委員会の有効性が低下する可能性があります。 経験と判断を使用して適切な適時性を判断し、スケジュールを変更できるのを常に覚えておいてください。 会議のためだけの会議は行う必要があります。

### <a name="prepare-for-the-meeting"></a>会議の準備

全員の時間を利用するには、会議の準備をする必要があります。 コア チームは、運営委員会会議の資料の準備を開始できます。 会議の準備にPowerPointプレゼンテーションを作成できます。

次の資産を完了し、会議のプレゼンテーション デッキに含める必要があります。

-   以前の運営委員会のアクション項目のフォローアップ

-   運営委員会PowerPoint、プロジェクトの高レベルのステータス レポート

    -   月/四半期/毎年の種類のレビューからプロジェクトを確認します。

    -   プロジェクトの重要なマイルストーンを特定します。

        -   マイルストーンの重要性と影響について話し合います (これらはリスク レジスタに取り込む必要があります)。

        -   準備、特にビジネス上の意思決定者に基づいて、今後の移行に向けて移動/立ち入りなしを決定します。

    -   プロジェクトの実行日と影響の大きなアクティビティを呼び出します。

    -   委員会にとって重要と特定された OKRs に関するレポート。使用、ユーザーセンチメント、品質、信頼性のメトリックを必ず含める必要があります。

-   運営委員会PowerPoint、プロジェクトの週次ステータス レポート

    -   強調表示とローライト、主要な問題、アクション アイテム、次の手順が含まれます。

    -   日次の問題に重点を置き、全体的なプロジェクトの状態に関するより技術的な詳細を含む。

-   問題ログと [リスクレジスタ](1-envision-define-my-success-cloud-voice.md#define-okrs-ksis-and-risks)

    -   委員会が追跡している問題の状態を共有します。

        -   すべてのバグに関する更新プログラムを提供する準備をしてください。

    -   ビジネス成果を達成する成功に影響を与える可能性がある新しい項目を特定します。

        -   プロジェクトへの影響を特定します。

        -   影響を受ける個人または事業単位を特定します。

        -   次の手順と、委員会のメンバーに必要な支援を特定します。

-   品質の概要: 「ネットワークを準備する」のガイダンス[をTeams。](prepare-network.md)

    -   これらの品質レポートの結果を詳細に説明し、書き込まれる問題を修復するための推奨事項を含める準備をしてください。

    -   次の領域に重点を置き、品質レポートを毎週調整します。

        -   組織の全体的な品質の傾向

        -   問題が発生している既知の問題サイトまたはサイトの状態。サイトの見出しの方向を示し、理想的には修復作業から肯定的な結果を報告します。

        -   プロジェクトの進捗状況に基づいて、最近有効にされたサイトの品質状態

次に、運営委員会の会議に備える通話をスケジュールします。 この呼び出しには、コア プロジェクト チームの重要なチーム メンバーのみを含める必要があります。これには、ネットワーク、デスクトップ、操作、または委員会会議中に対処されるアイテムを持つ他のチームからのリソースが含まれます。 主要なプロジェクト チーム メンバーは、運営委員会の呼び出し中にカバーする予定の資料に決して気を付けずに入れるべきです。

この準備の呼び出しは、運営委員会が会議する前に同期するための最適な方法です。 また、プロジェクト チーム間で開いている項目、意見の相違、不満が明るみ出されるのを確認する必要があります。 これは、オープンなコミュニケーションを促進し、会議中にだれかが驚かされるのを防ぐことを意図しています。

その週のレポートをコア チームおよびパートナーの関係者と確認し、懸念事項や質問に対処します。 チーム内でディスカッションに基づいてコンテンツを調整、追加、または削除したり、運営委員会から追加の明確化を求める場合に同意したりします。 各セクションを発表するチーム メンバーを割り当て、フローと継続性を確保します。

### <a name="hold-the-meeting"></a>会議を開催する 

会議自体は、コア チームが行った準備作業の自然な継続である必要があります。 これらの会議の目的は、プロジェクトのビジョンを実現するためのコミュニケーションとコラボレーションの雰囲気を促進する点に注意してください。 会議の成功を確実に確保するには、次の提案を使用します。

1.  会議を対面するスケジュールを設定します。 すべてのユーザーが座るのに十分な大きなテーブルがある会議室を特定します。 会議室にビデオ機能が搭載されている必要があります。 全員を部屋に入れるのが理想的ですが、常に可能なとは限りないので、会議出席依頼には会議室とオンライン会議の情報と少なくとも音声が含まれますが、理想的には音声、ビデオ、共有が含まれる必要があります。 リモートの参加者全員が会議に参加するときにビデオを使うのを促します。

2.  議題に進もうとしますが、対処が必要な気が散る可能性は間違いありません。 可能であれば (重要度に応じて)、これらの項目を "駐車場" セクションに追加し、後で対処します。 トピックを延期するには重要すぎる場合は、引き続き話し合う同意を得ますが、他の議題項目を将来の会議に移動する必要があります。 委員会の直ちに注意を払う必要がある差し迫った項目に対処するために、ある程度の時間 (5 ~ 10 分) を確保することに同意します。

3.  オープン サポート ケースについて話し合う準備をしてください。 この会議では、できるだけ高いレベル (チケットの数、チケットの種類、主要な開いているアイテム) を確認してください。 すべての問題に対処する準備をする必要があります。問題が軽微な場合でも、ビジネスへの影響とプロジェクトへの影響を理解する必要があります。

4.  メモ取りを委任します。 発表中や、熱い議論の最中にノートを取るのは難しすぎます。 委員会で行われた決定に関して詳細が文書化されている必要があります。 フォローアップメールの一部としてこれらの決定とメモを送信する準備をしてください。

5.  常に、ユーザーの作業時間とユーザーからのフィードバックに関する肯定的なニュースを宣伝してください (可能な場合)。 ストーリーを伝え、販売します。 フィードバックが完全に否定的な場合は、そのフィードバックについて話し合い、ユーザー エクスペリエンスの否定的な問題に対処するために何が行われるかを説明する準備をしてください。

    a.  個々のチーム メンバーに kudos を与え、自分が行っている良い作業を行う機会を得る。 特にリーダー チームの前で、チーム メンバーが優れたハード ワークを達成している場合を認識することが重要です。

6.  次の手順に同意し、会議中に同意したアクション アイテムの所有者を特定するには、常に会議の終了を保存します。

7.  作成したプレゼンテーションを含むフォローアップ メールを出席者に送信し、作成したノートを追加します。 必ず、出席しなかったユーザーを含める必要があります。

## <a name="next-steps"></a>次の手順

運営委員会の概要を確認した後、今度は行動する必要があります。プロジェクト チームは、プロジェクトのタイムラインに運営委員会を実装し、定期的な運営委員会会議の開催日を準備する作業を開始する必要があります。 理想的には、プロジェクトが開始される前に委員会が形成され、すべてのフェーズでガイダンスが提供されますが、それができない場合は、プロジェクト チームを成功に導く委員会を形成します。

1.  運営委員会のメンバーを特定します。 次の質問を行います。

    -   ビジネスとリーダーシップの適切な表現はありますか。

    -   特定したメンバーは、他のユニットのビジネス ユニットやピアに影響を与える可能性がありますか。

2.  運営委員会の各メンバーに送信する通信を作成します。

    -   roles テーブルを含める。

    -   委員会の重要性を売る。

3.  役割と責任について話し合い、委員会の運営を定義し、OKRs を検証または定義する最初の会議を設定します。 議題アイテムには、次のものも含まれます。

    -   会議の開催日について話し合います。

    -   この取り組みによって、重要なビジネス成果に向けて、この取り組みと OK が推進されているのを検証します。

    -   戦略が OKRs と一直線に配置されているのを確認します。

4.  すべての会話で品質の向上を開始します。

    -   高品質のチャンピオンを持つことの重要性を明確にする。

    -   品質チャンピオンの役割を担うユーザーまたはチームを特定します。

運営委員会の成功は、運によっては起こりません。 最適な意図を持つ適切なメンバーを選択すると、結果が低下する可能性があります。 運営委員会の成功を推進するには、次の情報が必要です。

-   委員会が主張する明確に定義された測定可能なビジネス成果を持つ、プロジェクトの合意されたビジョンと範囲。

-   目的に合った会議を開催する、十分に整理された議題。

-   オープンなコミュニケーションの雰囲気。

    -   オープンなコミュニケーションとコラボレーションにより、委員会は強化されます。 すべての声が聞こえ、意見が尊重される環境を作成します。 独自のパースペクティブが最も大きな影響を与える可能性があります。

    -   オープンな環境は、リーダーがメンバーの共同作業とコミュニケーションを促進する役割を行う場合に有効にできます。 いくつかのガイドラインを次に示します。

        -   すべての委員会メンバーは、組織に導入される変更に対する強力な支持者である必要があります。

        -   委員会の規模は、正確な画像を提供するために絶対に必要な人や、複数の事業単位で意思決定を行う、または影響を与える人に限定してください。 ブロックが表示される場合は、組織の階層と各メンバーの直接の影響力を理解することが重要です。

        -   プロジェクト マネージャーとチーム リーダーを委員会のディスカッションと検討に参加します。 特定の分野で必要になる可能性がある詳細なディスカッションを明確にするために、追加のコア チーム メンバーを招待する準備が整います。

ソリューションのライフサイクルを通じて、委員会の役割が変わります。 ドライブ値フェーズへの移行を構想およびオンボードする場合、委員会にはいくつかの調整が必要になる場合があります。

-   会議の開催日を調整します。 組織の動的な動きを確認し、会議の開催頻度を低くする必要があるかどうかを判断します。 組織の変更が始まると、アップレベルの通信の頻度が低下する可能性があります。 必要に応じて、会議を月次または四半期の会議の開催日に変更する方法を調査する

-   会議の担当者を調整します。 ソリューションがドライブ値フェーズに移動すると、コア プロジェクト チームが新しいプロジェクトに再割り当てされる可能性があります。 サポートと運用がソリューションの状態のリードになると報告しているチームを更新する必要があります。

-   運用 OK に対する定義とレポート。 主要な結果と成功のインジケーターは、ソリューションのライフサイクルを通じて変化します。 望ましいビジネス成果は、引き続き追跡および報告する必要があります。 引き続き、ビジネス成果に向けて推進するコンテキストで、音声ソリューションの品質と信頼性について報告します。 ユーザーセンチメント、ユーザーの導入、サポート メトリックなど、追加のメトリックと主要な結果が、成功のトップ インジケーターになります。

-   複数のソリューションのドライブ値フェーズをカバーするには、より大きな運営委員会の傘を使用します。 運営委員会を効率的に使用する機会を探します。 委員会を使用して、より大規模なソリューション領域全体でガイダンスを提供します。
